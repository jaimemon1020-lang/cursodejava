## Qué vas a aprender

Después de esta lección serás capaz de arrancar una aplicación Spring Boot real con `@SpringBootApplication` y `SpringApplication.run`, entender qué automatiza `spring-boot-starter-parent` en el `pom.xml`, y ejecutar código al arrancar la aplicación con `CommandLineRunner`.

## Por qué necesitas aprenderlo

En la Etapa 10, cada demo (`SpringDemo`, `BeanScopesDemo`, `QualifierDemo`) arrancaba el contenedor MANUALMENTE con `new AnnotationConfigApplicationContext(AppConfig.class)`. Spring Boot es Spring Framework CON auto-configuración: en vez de que vos decidas explícitamente cada pieza, Boot detecta qué tenés en el `classpath` y configura sensatamente por vos — la base de CUALQUIER aplicación Spring moderna, incluyendo las que vas a construir con REST (Etapa 12) en adelante.

## Qué debes saber antes

`ApplicationContext`, `@Component`/`@Autowired` (Etapa 10 Lección 1) — Spring Boot no reemplaza esos conceptos, los envuelve en una capa de arranque más simple.

## Recupera lo aprendido

¿Qué gana `@Qualifier` sobre `@Primary` cuando ambos aplican al mismo punto de inyección?

<Reveal titulo="Respuesta">
@Qualifier siempre gana: es una instrucción local y específica para ESE punto de inyección exacto, mientras que @Primary es solo una preferencia global que aplica cuando no hay nada más específico indicado.
</Reveal>

## Problema

Cada demo de la Etapa 10 repetía el mismo arranque manual: `new AnnotationConfigApplicationContext(AppConfig.class)`, después `getBean(...)` para cada cosa que necesitabas. En una aplicación real, con decenas de dependencias (base de datos, seguridad, serialización JSON), configurar cada una a mano sería una tarea enorme — Spring Boot existe exactamente para eliminar ese trabajo repetitivo.

## Modelo mental

```
Spring Framework (Etapa 10): VOS armás el ApplicationContext
                              a mano, decidiendo cada pieza.

Spring Boot (esta etapa):    SpringApplication.run() detecta
                              qué hay en el classpath y
                              AUTO-CONFIGURA todo lo razonable
                              por vos, con valores por defecto
                              sensatos que podés sobrescribir.
```

## Explicación sencilla

Spring Boot no es un framework DISTINTO a Spring — es Spring Framework con una capa de arranque (`SpringApplication`) y auto-configuración inteligente encima. En vez de escribir `@Configuration` + `@ComponentScan` + `new AnnotationConfigApplicationContext(...)` a mano, una sola anotación (`@SpringBootApplication`) y una sola línea (`SpringApplication.run(...)`) hacen todo eso por vos, además de detectar automáticamente QUÉ MÁS necesitás configurar según las dependencias que tengas en el `pom.xml`.

## Explicación técnica

`spring-boot-starter-parent` (declarado como `<parent>` en `pom.xml`) es un POM padre que centraliza la gestión de VERSIONES de todo el ecosistema Spring — por eso `spring-boot-starter` ya no necesita declarar su propia `<version>`: la hereda del padre, garantizando que todas las piezas de Spring en el proyecto sean compatibles entre sí. También define plugins y configuraciones sensatas por defecto (como el `java.version` que reemplaza a `maven.compiler.release`).

`@SpringBootApplication` es en realidad TRES anotaciones combinadas en una: `@Configuration` (la clase puede definir beans con `@Bean`), `@ComponentScan` (escanea automáticamente el paquete de la clase anotada Y sus subpaquetes — sin necesitar `basePackages` explícito), y `@EnableAutoConfiguration` (activa la auto-configuración de Boot: detecta qué librerías están en el `classpath` y configura beans razonables automáticamente).

`SpringApplication.run(Application.class, args)` reemplaza `new AnnotationConfigApplicationContext(AppConfig.class)`: arranca el contenedor, ejecuta toda la auto-configuración, y deja la aplicación lista. `CommandLineRunner` es una interfaz funcional (`void run(String... args)`) que, si se registra como bean, Spring Boot ejecuta AUTOMÁTICAMENTE apenas el contexto termina de arrancar — el lugar natural para lógica de inicio, reemplazando el código que antes escribías manualmente después de crear el contexto.

## Cómo funciona

1. `spring-boot-starter-parent` en `pom.xml` gestiona versiones compatibles de todo Spring; `spring-boot-starter` trae Spring Core + Context como dependencia mínima.
2. `@SpringBootApplication` sobre `Application` (en `com.curso.catalogo`) escanea automáticamente todo ese paquete — encontrando `RepositorioProductos`, `ServicioVentas`, `Descuento`/`DescuentoFijo`/`DescuentoPorcentual`, y hasta `AppConfig` (que sigue siendo `@Configuration` con su propio `@Bean` de `ProcesadorPago`).
3. `SpringApplication.run(Application.class, args)` arranca todo el contexto de una sola línea.
4. El método `demo(RepositorioProductos repositorio, ServicioVentas servicio)`, anotado `@Bean` y devolviendo un `CommandLineRunner`, recibe sus parámetros INYECTADOS automáticamente (igual que un constructor `@Autowired`) — Spring resuelve `RepositorioProductos` y `ServicioVentas` sin que se los pidas explícitamente con `getBean`.
5. El lambda `args -> { ... }` (la implementación de `CommandLineRunner.run`) se ejecuta automáticamente apenas la aplicación termina de arrancar.

## Ejemplo mínimo

**Archivo:** `proyecto-maven/pom.xml` (fragmento relevante)

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.5.16</version>
    <relativePath/>
</parent>

<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter</artifactId>
    </dependency>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-test</artifactId>
        <scope>test</scope>
    </dependency>
</dependencies>

<build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
        </plugin>
    </plugins>
</build>
```

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/Application.java`

```java
package com.curso.catalogo;

import org.springframework.boot.CommandLineRunner;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.annotation.Bean;

@SpringBootApplication
public class Application {

    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }

    @Bean
    public CommandLineRunner demo(RepositorioProductos repositorio, ServicioVentas servicio) {
        return args -> {
            repositorio.agregar(new Product("Teclado mecanico", 25000.0, 15));
            servicio.venderProducto("Teclado mecanico", 5);
        };
    }
}
```

## Explicación paso a paso

- El banner ASCII y los logs de "Starting Application"/"Started Application" son característicos de Spring Boot — confirman que `SpringApplication.run` arrancó el contenedor completo, con auto-configuración incluida.
- "No active profile set, falling back to 1 default profile" es una pista de que Boot soporta PERFILES de configuración (útil más adelante, para distinguir entornos como desarrollo/producción) — sin que hayamos configurado ninguno todavía.
- "Vendido: 5 x Teclado mecanico" es exactamente el mismo resultado que en `SpringDemo` (Etapa 10) — el `CommandLineRunner` ejecutó la MISMA lógica de negocio, pero SIN que `main` escribiera ningún `getBean` explícito: Spring inyectó `RepositorioProductos` y `ServicioVentas` directamente como parámetros del método `@Bean`.
- Todo esto ocurrió en menos de un segundo ("Started Application in 0.589 seconds") — la auto-configuración de Boot no agrega una sobrecarga perceptible para una aplicación de este tamaño.

## Ejemplo aplicado al proyecto

`Application` reemplaza el rol de `AppConfig` + arranque manual de la Etapa 10 — pero `AppConfig` sigue funcionando SIN modificarse, porque `@SpringBootApplication` escanea el mismo paquete y encuentra su `@Configuration` y su `@Bean` de `ProcesadorPago` automáticamente. Nada de lo construido en la Etapa 10 se perdió; Spring Boot lo envuelve en una capa de arranque más simple.

## Ejercicio guiado

Cambiá el `CommandLineRunner` de `demo` para que, en vez de vender directamente, use `ServicioDescuentos` (Etapa 10 Lección 3) para calcular el precio con descuento ANTES de imprimir el resultado de la venta.

<Reveal titulo="Pista 1">
Agregá `ServicioDescuentos servicioDescuentos` como tercer parámetro del método `demo` — Spring lo va a inyectar automáticamente, igual que a los otros dos.
</Reveal>

<Reveal titulo="Solución">
```java
@Bean
public CommandLineRunner demo(RepositorioProductos repositorio, ServicioVentas servicio, ServicioDescuentos servicioDescuentos) {
    return args -> {
        repositorio.agregar(new Product("Teclado mecanico", 25000.0, 15));
        double conDescuento = servicioDescuentos.aplicarPorDefecto(25000.0);
        System.out.println("Precio con descuento: " + conDescuento);
        servicio.venderProducto("Teclado mecanico", 5);
    };
}
```
</Reveal>

## Mini reto de debugging

```java
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        // falta SpringApplication.run(...)
    }

    @Bean
    public CommandLineRunner demo(RepositorioProductos repositorio) {
        return args -> System.out.println("Productos: " + repositorio.listarTodos().size());
    }
}
```

¿Qué pasa al ejecutar `main` con este código?

<Reveal titulo="Diagnóstico">
El programa termina inmediatamente sin hacer NADA — nunca se imprime "Productos: ...". `@SpringBootApplication` solo DECLARA que la clase es el punto de entrada de una app Spring Boot; sin `SpringApplication.run(Application.class, args)`, nunca se arranca ningún contenedor, ningún bean se crea, y ningún `CommandLineRunner` se ejecuta. La anotación por sí sola no hace nada sin la llamada explícita.
</Reveal>

## Resumen

- `spring-boot-starter-parent` centraliza la gestión de versiones compatibles de todo el ecosistema Spring.
- `@SpringBootApplication` combina `@Configuration` + `@ComponentScan` + `@EnableAutoConfiguration` en una sola anotación.
- `SpringApplication.run` reemplaza la creación manual de `ApplicationContext`.
- `CommandLineRunner` registrado como bean se ejecuta automáticamente apenas la aplicación arranca, con sus dependencias inyectadas como parámetros.

## Modelo mental final

Pensá en Spring Framework puro (Etapa 10) como armar un auto pieza por pieza vos mismo, y en Spring Boot como comprarlo ya ensamblado de fábrica con configuración de serie sensata — seguís pudiendo abrir el capot y ajustar cualquier pieza (como `AppConfig` con su `@Bean` manual), pero no necesitás ensamblar todo desde cero cada vez.

## Conexión

Todo lo construido en la Etapa 10 (`RepositorioProductos`, `ServicioVentas`, `Descuento` con `@Qualifier`/`@Primary`, `AppConfig` con `@Bean`) sigue funcionando SIN cambios — Spring Boot es una capa de arranque y auto-configuración sobre el MISMO Spring Framework, no un reemplazo.

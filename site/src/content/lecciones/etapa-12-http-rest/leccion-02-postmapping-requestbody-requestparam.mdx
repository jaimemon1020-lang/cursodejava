## Qué vas a aprender

Después de esta lección serás capaz de crear endpoints `POST` que reciben datos en el CUERPO de la petición HTTP (`@RequestBody`) o como parámetros de la URL (`@RequestParam`), completando el ciclo de leer Y escribir datos a través de HTTP, no solo consultarlos.

## Por qué necesitas aprenderlo

`GET /api/productos` (Etapa 12 Lección 1) solo LEE datos — perfecto para consultas, pero nunca debería usarse para modificar el sistema, siguiendo la semántica estándar de HTTP. Crear un producto nuevo o vender unidades son ACCIONES que cambian el estado del sistema, y necesitan el verbo HTTP correcto (`POST`) junto con una forma de recibir los datos necesarios.

## Qué debes saber antes

`@RestController`, `@GetMapping`, `@PathVariable` (Etapa 12 Lección 1) — `@PostMapping` sigue el mismo patrón de mapeo de rutas.

## Recupera lo aprendido

¿Por qué Jackson puede convertir un `Product` a JSON sin que escribas ningún código de serialización manual?

<Reveal titulo="Respuesta">
Porque Jackson usa los GETTERS del objeto (getNombre, getPrecio, getStock, getFechaIngreso) via reflection para construir automáticamente las claves y valores del JSON, sin necesitar anotaciones adicionales para el caso simple.
</Reveal>

## Problema

Necesitás DOS acciones nuevas: crear un producto nuevo (recibiendo nombre, precio y stock desde afuera) y vender unidades de un producto existente (recibiendo cuántas). Ninguna de las dos es una simple CONSULTA — ambas MODIFICAN el estado del sistema, así que `GET` sería semánticamente incorrecto para cualquiera de las dos.

## Modelo mental

```
GET:  "dame información" — no debería modificar nada.
POST: "hacé algo con estos datos" — crea o modifica estado.

@RequestBody:   datos en el CUERPO de la petición (JSON completo).
@RequestParam:  datos en la URL, después de "?" (clave=valor).
```

## Explicación sencilla

`@PostMapping` marca un método para responder a peticiones `POST` — el verbo HTTP pensado para crear o enviar datos que van a cambiar algo en el servidor. `@RequestBody` le dice a Spring "convertí el CUERPO de la petición (normalmente JSON) en un objeto Java" — el proceso INVERSO al que ya viste con `@GetMapping` (que convierte un objeto Java A JSON). `@RequestParam` lee un valor específico de la QUERY STRING de la URL (la parte después de `?`), útil para datos simples que no ameritan un cuerpo JSON completo.

## Explicación técnica

`@RequestBody NuevoProductoRequest request` le dice a Spring: "deserializá el JSON del cuerpo de la petición en un objeto `NuevoProductoRequest`". Para que esto funcione, Jackson necesita saber cómo CONSTRUIR ese objeto a partir del JSON — un `record` (Etapa 4 Lección 5) funciona perfecto acá porque Jackson (desde versiones recientes) reconoce automáticamente su constructor canónico, mapeando cada campo del JSON a cada componente del record por nombre.

`NuevoProductoRequest` es intencionalmente una clase SEPARADA de `Product`, no el mismo `Product` reutilizado — esto es una primera aproximación al concepto de **DTO** (Data Transfer Object), que la Etapa 13 va a formalizar: el objeto que VIAJA por la red no tiene por qué ser idéntico al objeto de tu dominio interno. Por ejemplo, `NuevoProductoRequest` no incluye `fechaIngreso` (que `Product` calcula internamente con `LocalDate.now()`), porque quien crea un producto nunca debería poder elegir esa fecha manualmente.

`@RequestParam int cantidad` extrae el valor de la query string (`?cantidad=10`) y lo convierte automáticamente al tipo declarado (`int`) — si el valor no fuera un número válido, Spring respondería con un error `400 Bad Request` automáticamente, sin código adicional tuyo.

## Cómo funciona

1. `POST /api/productos` con cuerpo `{"nombre": "...", "precio": ..., "stock": ...}` se deserializa en un `NuevoProductoRequest` gracias a `@RequestBody`.
2. `crear(...)` construye un `Product` REAL a partir de esos datos (`new Product(request.nombre(), request.precio(), request.stock())`), lo agrega al repositorio, y lo devuelve — la respuesta SÍ es un `Product` completo (con `fechaIngreso` ya calculada), aunque la petición solo mandó tres campos.
3. `POST /api/productos/{nombre}/vender?cantidad=N` combina `@PathVariable` (para el nombre) con `@RequestParam` (para la cantidad, en la query string).
4. `vender(...)` reutiliza `ServicioVentas.venderProducto` (Etapa 10) — TODA la validación de stock (`StockInsuficienteException`, Etapa 4 Lección 6) sigue aplicando exactamente igual, ahora disparada desde una petición HTTP en vez de un `main` de consola.

## Ejemplo mínimo

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/NuevoProductoRequest.java`

```java
package com.curso.catalogo;

public record NuevoProductoRequest(String nombre, double precio, int stock) {
}
```

**Fragmento agregado a `proyecto-maven/src/main/java/com/curso/catalogo/ProductoController.java`:**

```java
private final ServicioVentas servicioVentas;

public ProductoController(RepositorioProductos repositorio, ServicioVentas servicioVentas) {
    this.repositorio = repositorio;
    this.servicioVentas = servicioVentas;
}

@PostMapping
public Product crear(@RequestBody NuevoProductoRequest request) {
    Product producto = new Product(request.nombre(), request.precio(), request.stock());
    repositorio.agregar(producto);
    return producto;
}

@PostMapping("/{nombre}/vender")
public Product vender(@PathVariable String nombre, @RequestParam int cantidad) {
    servicioVentas.venderProducto(nombre, cantidad);
    return repositorio.buscar(p -> p.getNombre().equals(nombre)).orElseThrow();
}
```

</Reveal>

## Explicación paso a paso

- La respuesta del `POST /api/productos` es un `Product` COMPLETO con `fechaIngreso` incluida ("2026-07-19"), aunque el cuerpo de la petición nunca mandó ese campo — `Product` la calculó internamente con `LocalDate.now()` (Etapa 4 Lección 7), exactamente igual que si se hubiera creado desde código Java.
- `GET /api/productos` ahora devuelve DOS productos: el "Teclado mecanico" del `CommandLineRunner` (Etapa 11) y el "Mouse inalambrico" recién creado por HTTP — ambos viven en el MISMO `RepositorioProductos` singleton (Etapa 10 Lección 2).
- El stock final es `30` (40 - 10) — `ServicioVentas.venderProducto` (Etapa 10) se ejecutó exactamente igual que en cualquier demo de consola, solo que esta vez la orden llegó desde una petición HTTP.
- El nombre con espacio ("Mouse inalambrico") necesitó `%20` tanto en la ruta (`@PathVariable`) como implícitamente al armar la URL — la codificación de URL aplica a cualquier parte de la ruta, no solo a la query string.

## Ejemplo aplicado al proyecto

Con `POST /api/productos` y `POST /api/productos/{nombre}/vender`, el catálogo completo (crear, consultar, vender) ahora es accesible desde CUALQUIER cliente HTTP — un navegador con JavaScript, una app móvil, u otro servicio — sin que ninguna de esas partes necesite conocer Java ni tener acceso directo al proceso de la aplicación.

## Ejercicio guiado

Agregá un endpoint `POST /api/productos/{nombre}/precio` que reciba un nuevo precio vía `@RequestParam double nuevoPrecio`, y use `Product.setPrecio` (Etapa 2) para actualizarlo.

<Reveal titulo="Pista 1">
El patrón es idéntico a `vender`: buscar el producto por nombre con `repositorio.buscar`, y llamar al método correspondiente sobre el resultado.
</Reveal>

<Reveal titulo="Solución">
```java
@PostMapping("/{nombre}/precio")
public Product actualizarPrecio(@PathVariable String nombre, @RequestParam double nuevoPrecio) {
    Product producto = repositorio.buscar(p -> p.getNombre().equals(nombre))
            .orElseThrow(() -> new IllegalArgumentException("Producto no encontrado: " + nombre));
    producto.setPrecio(nuevoPrecio);
    return producto;
}
```
</Reveal>

## Mini reto de debugging

```java
@PostMapping("/{nombre}/vender")
public Product vender(@PathVariable String nombre, @RequestParam int cantidad) {
    servicioVentas.venderProducto(nombre, cantidad);
    return repositorio.buscar(p -> p.getNombre().equals(nombre)).orElseThrow();
}
```

Si `cantidad` supera el stock disponible, `ServicioVentas.venderProducto` lanza `StockInsuficienteException` (Etapa 4 Lección 6, unchecked). ¿Qué código de estado HTTP esperás que reciba el cliente, sin ningún manejo adicional de excepciones en el controlador?

<Reveal titulo="Diagnóstico">
`500 Internal Server Error` — el código de estado por defecto de Spring Boot cuando una excepción NO manejada llega hasta el controlador. Esto es técnicamente correcto (algo falló), pero semánticamente pobre: un cliente HTTP debería recibir `400 Bad Request` (la petición en sí era inválida: pidió más de lo disponible), no un error genérico de servidor. La Etapa 14 (Validación y manejo de errores) va a resolver esto formalmente con manejadores de excepciones específicos.
</Reveal>

## Resumen

- `@PostMapping` responde a peticiones que crean o modifican estado — nunca uses `GET` para eso.
- `@RequestBody` deserializa el cuerpo JSON de la petición en un objeto Java, típicamente un `record`.
- `@RequestParam` lee valores individuales de la query string, con soporte para valores por defecto.
- Separar el objeto de ENTRADA (`NuevoProductoRequest`) del objeto de DOMINIO (`Product`) es la primera aproximación al patrón DTO.

## Modelo mental final

Pensá en `@RequestBody` como recibir un formulario completo por correo (varios campos juntos, en un sobre), y en `@RequestParam` como una pregunta rápida gritada desde la puerta (un solo dato, sin ceremonia) — cada uno tiene su lugar según cuánta información necesitás pasar.

## Conexión

## Qué vas a aprender

Después de esta lección serás capaz de crear una jerarquía de clases con `extends`, reutilizar el constructor y comportamiento de una clase base con `super`, y modelar una relación "tiene un" con composición — y decidir cuál de las dos usar en cada caso.

## Por qué necesitas aprenderlo

Un `Cliente` y un futuro `Empleado` comparten datos (nombre, email) pero no son lo mismo. Y un `Order` necesita un cliente y un producto, sin ser "una especie de" ninguno de los dos — necesitás las dos herramientas, cada una para su relación correcta.

## Qué debes saber antes

Clases, objetos, constructores, `this`, y encapsulamiento con `private` (lecciones anteriores de esta etapa).

## Problema

Necesitás modelar un `Cliente`: tiene nombre y email, como cualquier persona registrada en el sistema, pero además una dirección de envío. Copiar `nombre` y `email` de nuevo en una clase `Cliente` separada duplicaría exactamente lo que ya escribiste. Y por otro lado, un `Order` necesita "contener" un cliente y un producto — pero un pedido no ES un cliente, ni ES un producto.

## Modelo mental

```
HERENCIA ("es un")              COMPOSICIÓN ("tiene un")
                                
   Persona                        Order
      ▲                             │
      │ extends                     │ tiene una referencia a
   Cliente                        Cliente,  Product
                                
Cliente ES UNA Persona           Order TIENE UN Cliente
(hereda sus atributos/métodos)   (los usa, no los "es")
```

## Explicación sencilla

Herencia es cuando una clase ES un caso particular de otra: un `Cliente` es una `Persona` con algo extra. Composición es cuando una clase TIENE otra como parte de sus datos: un `Order` tiene un `Cliente` y un `Product`, pero no es ninguno de los dos.

## Explicación técnica

**Herencia (`extends`):** una clase hija reutiliza los atributos y métodos de una clase padre. `protected` (en vez de `private`) permite que la clase hija acceda directamente a esos atributos heredados. `super(...)` en el constructor de la hija invoca el constructor del padre — es obligatoriamente la primera línea si se usa. `super.metodo()` invoca la versión del padre de un método que la hija sobrescribió (**override**, marcado con `@Override`).

Sobrescribir un método (`@Override`) es la base del **polimorfismo**: cuando llamás `cliente.describir()`, Java ejecuta la versión de `Cliente` (más específica), no la de `Persona` — aunque la variable esté declarada como `Persona`. Vas a profundizar esto (con clases abstractas e interfaces) en la próxima lección.

**Composición:** una clase simplemente declara un atributo cuyo tipo es otra clase. No hay `extends`, no hay jerarquía — solo una referencia normal, como cualquier otro atributo.

## Cómo funciona

1. `Cliente extends Persona` hereda `nombre`, `email`, `getNombre()`.
2. `super(nombre, email)` en el constructor de `Cliente` ejecuta el constructor de `Persona`, evitando repetir esa asignación.
3. `describir()` en `Cliente` usa `@Override` para agregar comportamiento nuevo, llamando primero a `super.describir()` para no perder lo que ya hacía `Persona`.
4. `Order` no hereda de nada — simplemente guarda una referencia a un `Cliente` y a un `Product` como cualquier atributo, y los usa en `calcularTotal()`.

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-02/Persona.java`

```java
public class Persona {

    protected String nombre;
    protected String email;

    Persona(String nombre, String email) {
        this.nombre = nombre;
        this.email = email;
    }

    public String getNombre() {
        return this.nombre;
    }

    public void describir() {
        System.out.println("Persona: " + this.nombre + " (" + this.email + ")");
    }
}
```

**Archivo:** `proyecto/etapa-02/Cliente.java`

```java
public class Cliente extends Persona {

    private String direccionEnvio;

    Cliente(String nombre, String email, String direccionEnvio) {
        super(nombre, email);
        this.direccionEnvio = direccionEnvio;
    }

    public String getDireccionEnvio() {
        return this.direccionEnvio;
    }

    @Override
    public void describir() {
        super.describir();
        System.out.println("Direccion de envio: " + this.direccionEnvio);
    }
}
```

**Archivo:** `proyecto/etapa-02/Order.java`

```java
public class Order {

    private Cliente cliente;
    private Product producto;
    private int cantidad;

    Order(Cliente cliente, Product producto, int cantidad) {
        this.cliente = cliente;
        this.producto = producto;
        this.cantidad = cantidad;
    }

    public double calcularTotal() {
        return this.producto.getPrecio() * this.cantidad;
    }

    public void describir() {
        System.out.println("Pedido de: " + this.cliente.getNombre());
        System.out.println("Producto: " + this.producto.getNombre() + " x" + this.cantidad);
        System.out.println("Total: " + this.calcularTotal());
    }
}
```

## Predice

`Cliente.describir()` llama primero a `super.describir()`. ¿Va a imprimir SOLO los datos de dirección, o los datos de `Persona` también?

<Reveal titulo="Resultado real (compilado y ejecutado)">
```

---
Persona: Juan Perez (juan@ejemplo.com)
Direccion de envio: Av. Siempre Viva 123
---
Pedido de: Juan Perez
Producto: Teclado mecanico x2
Total: 50000.0
```
</Reveal>

## Explicación paso a paso

- `new Persona(...)` imprime solo su propia línea — no tiene noción de `Cliente`.
- `new Cliente(...).describir()` imprime PRIMERO la línea de `Persona` (por el `super.describir()`) y DESPUÉS la línea propia de `Cliente` — ambas, porque `super` no reemplaza, complementa.
- `Order` nunca hereda de `Cliente` ni de `Product` — simplemente los usa a través de sus getters (`getNombre()`, `getPrecio()`), como cualquier objeto colaborador.

## Ejemplo aplicado al proyecto

Esta es la primera vez que `Order` aparece en el proyecto: por composición, contiene un `Cliente` y (por ahora) un solo `Product`. Cuando lleguemos a Collections (Etapa 4), vas a reemplazar "un producto" por una lista de `OrderItem`, sin cambiar la idea de fondo: `Order` siempre va a TENER esos datos, nunca SER uno de ellos.

## Resumen

- Herencia (`extends`) modela "es un"; reutiliza atributos y métodos de una clase padre.
- `super(...)` invoca el constructor padre; debe ser la primera línea del constructor hijo.
- `super.metodo()` invoca la versión del padre de un método sobrescrito (`@Override`).
- Composición modela "tiene un": una clase simplemente contiene una referencia a otra, sin jerarquía.
- Sobrescribir un método es la base del polimorfismo — profundizado en la próxima lección.

## Modelo mental final

Cuando pienses en herencia vs. composición, preguntate: "¿esto ES un caso particular de aquello, o simplemente LO TIENE como parte de sus datos?" La respuesta correcta casi siempre es evidente si la formulás así de literal.

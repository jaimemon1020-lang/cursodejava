## Qué vas a aprender

Después de esta lección serás capaz de centralizar la decisión de "qué implementación crear" en una clase Factory, separándola del código que USA esa implementación, y de reconocer formalmente que `Descuento` (con `DescuentoFijo`/`DescuentoPorcentual`) y `MetodoEnvio` (con `EnvioEstandar`/`EnvioExpress`) son el patrón Strategy, aplicado desde la Etapa 2 sin nombrarlo.

## Por qué necesitas aprenderlo

En `SolidDemo` (Etapa 7 Lección 2) creaste manualmente `new DescuentoFijo(2000.0)` y `new DescuentoPorcentual(0.10)` directamente en el código. Pero decidir CUÁL descuento aplicar según una regla de negocio (por ejemplo, según el monto total de la compra) es lógica que no debería mezclarse con el código que simplemente USA el descuento ya decidido.

## Qué debes saber antes

`Descuento`, `DescuentoFijo`, `DescuentoPorcentual` (Etapa 2 Lección 4) y Dependency Injection (Etapa 8 Lección 1).

## Recupera lo aprendido

¿Por qué `ServicioVentas` recibe un `RepositorioEnMemoria<Product>` por constructor en vez de crear uno con `new` internamente?

<Reveal titulo="Respuesta">
Para que quien construye ServicioVentas decida qué repositorio (datos reales o de prueba) usar, sin que ServicioVentas tenga que modificarse para cada caso — eso es Dependency Injection.
</Reveal>

## Problema

Necesitás decidir automáticamente qué tipo de descuento aplicar según el monto total de una compra: compras chicas sin descuento, compras medianas con 10%, compras grandes con 15%. Si esa lógica de decisión (`if (monto >= 100000) ... else if (monto >= 50000) ...`) se repite en CADA lugar del código que necesita un descuento, cualquier cambio en las reglas de negocio obliga a tocar múltiples lugares.

## Modelo mental

```
Strategy:  una FAMILIA de algoritmos intercambiables detrás de
           una interfaz común (Descuento ya es esto desde Etapa 2).

Factory:   una clase que centraliza la decisión de QUÉ implementación
           de esa familia crear, según alguna regla de negocio.
```

## Explicación sencilla

**Strategy** es tener varias formas de hacer lo MISMO (calcular un descuento) detrás de una interfaz común, de forma que el código que las usa no necesite saber cuál está usando en cada momento — eso es exactamente `Descuento`. **Factory** es una clase separada cuyo único trabajo es decidir CUÁL de esas estrategias crear, según alguna condición — separando "decidir qué usar" de "usarlo".

## Explicación técnica

El patrón **Strategy** consta de: una interfaz común (`Descuento`), múltiples implementaciones intercambiables (`DescuentoFijo`, `DescuentoPorcentual`), y código cliente que opera sobre la interfaz sin conocer la implementación concreta (`aplicarDescuento(Product, Descuento)` de la Etapa 7). Ya identificaste esto formalmente en la Etapa 7 Lección 2 como una aplicación de OCP, LSP y DIP — Strategy es el NOMBRE del patrón que resulta de aplicar esos tres principios juntos a un problema de "algoritmos intercambiables".

El patrón **Factory** resuelve un problema DISTINTO: ¿quién decide CUÁL estrategia usar? Sin Factory, esa decisión (`if`/`else` sobre alguna condición de negocio) tendría que repetirse en cada lugar que necesite crear un `Descuento`. Con Factory, esa lógica de decisión vive en UN SOLO lugar (`DescuentoFactory.crearParaMontoTotal`), y el resto del código simplemente le pide al Factory "dame el descuento correcto para este monto", sin conocer las reglas de decisión.

Una Factory típicamente expone un método ESTÁTICO (no necesita estado propio, es una decisión pura basada en el input) que devuelve el TIPO de la interfaz (`Descuento`), nunca el tipo concreto — así el código que llama al Factory sigue programando contra la abstracción, no contra `DescuentoFijo` o `DescuentoPorcentual` directamente.

## Cómo funciona

1. `DescuentoFactory.crearParaMontoTotal(double montoTotal)` centraliza la regla de negocio: montos ≥ 100.000 obtienen 15% de descuento, montos ≥ 50.000 obtienen 10%, el resto no tiene descuento (representado como `DescuentoFijo(0.0)`, que no resta nada).
2. El método devuelve `Descuento` (la interfaz), nunca `DescuentoPorcentual` ni `DescuentoFijo` directamente — el código que llama a la Factory sigue sin conocer la implementación concreta.
3. `FactoryStrategyDemo` recorre distintos montos, pidiéndole a la Factory el descuento correcto para cada uno, y aplicándolo SIN saber (ni necesitar saber) qué implementación concreta le devolvió.
4. Si mañana cambia el umbral de 100.000 a 120.000, ese cambio ocurre en UN SOLO lugar: `DescuentoFactory` — ningún código cliente necesita modificarse.

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-08/DescuentoFactory.java`

```java
public class DescuentoFactory {

    public static Descuento crearParaMontoTotal(double montoTotal) {
        if (montoTotal >= 100000.0) {
            return new DescuentoPorcentual(0.15);
        } else if (montoTotal >= 50000.0) {
            return new DescuentoPorcentual(0.10);
        } else {
            return new DescuentoFijo(0.0);
        }
    }
}
```

**Archivo:** `proyecto/etapa-08/FactoryStrategyDemo.java`

```java
public class FactoryStrategyDemo {
    public static void main(String[] args) {
        double[] montos = {30000.0, 60000.0, 150000.0};

        for (double monto : montos) {
            Descuento descuento = DescuentoFactory.crearParaMontoTotal(monto);
            double total = descuento.aplicar(monto);
            System.out.println("Monto " + monto + " -> " + descuento.getClass().getSimpleName() + " -> total: " + total);
        }
    }
}
```

## Predice

30000 no llega al umbral mínimo (50000), así que no tiene descuento. 60000 supera 50000 pero no 100000 (10%). 150000 supera 100000 (15%). ¿Qué `Descuento` concreto va a crear la Factory para cada monto?

<Reveal titulo="Resultado real (compilado y ejecutado)">
```
Monto 30000.0 -> DescuentoFijo -> total: 30000.0
Monto 60000.0 -> DescuentoPorcentual -> total: 54000.0
Monto 150000.0 -> DescuentoPorcentual -> total: 127500.0
```
</Reveal>

## Explicación paso a paso

- 30000 no alcanza los 50000 mínimos: la Factory devuelve `DescuentoFijo(0.0)` — resta cero, el total queda igual (30000.0).
- 60000 supera 50000 pero no 100000: la Factory devuelve `DescuentoPorcentual(0.10)` — `60000 - 6000 = 54000`.
- 150000 supera 100000: la Factory devuelve `DescuentoPorcentual(0.15)` — `150000 - 22500 = 127500`.
- El bucle en `FactoryStrategyDemo` NUNCA menciona `DescuentoFijo` ni `DescuentoPorcentual` por nombre — solo interactúa con `Descuento` (la interfaz) y `DescuentoFactory` (quien decide).

## Ejemplo aplicado al proyecto

Con Factory + Strategy combinados, agregar una CUARTA regla de negocio (por ejemplo, "montos sobre 500.000 tienen envío gratis Y 20% de descuento") solo requiere modificar `DescuentoFactory` — ningún código que ya use `Descuento` a través de la interfaz necesita cambiar.

## Ejercicio guiado

Escribí una `MetodoEnvioFactory` que devuelva `EnvioExpress` si el monto de la compra supera $200.000 (envío rápido para compras grandes), o `EnvioEstandar` en cualquier otro caso.

<Reveal titulo="Pista 1">
El patrón es idéntico a `DescuentoFactory`: un método estático que devuelve el TIPO de la interfaz/clase abstracta (`MetodoEnvio`), decidiendo la implementación concreta con un `if`.
</Reveal>

<Reveal titulo="Solución">
```java
public class MetodoEnvioFactory {
    public static MetodoEnvio crearParaMonto(double montoCompra) {
        return montoCompra > 200000.0 ? new EnvioExpress() : new EnvioEstandar();
    }
}
```
</Reveal>

## Mini reto de debugging

```java
public class DescuentoFactory {
    public static DescuentoPorcentual crearParaMontoTotal(double montoTotal) {
        if (montoTotal >= 100000.0) {
            return new DescuentoPorcentual(0.15);
        }
        return new DescuentoPorcentual(0.0);
    }
}
```

¿Qué problema tiene esta Factory, comparada con la versión que devuelve `Descuento`?

<Reveal titulo="Diagnóstico">
El tipo de retorno es `DescuentoPorcentual` (la implementación concreta), no `Descuento` (la interfaz) — esto rompe el propósito del patrón: cualquier código que llame a esta Factory queda ATADO a `DescuentoPorcentual` específicamente, y si mañana la Factory necesitara devolver un `DescuentoFijo` en algún caso, el código no compilaría. La Factory debe devolver siempre el tipo de la ABSTRACCIÓN, nunca de la implementación concreta.
</Reveal>

## Resumen

- Strategy: una interfaz común con múltiples implementaciones intercambiables — `Descuento`/`MetodoEnvio` ya lo eran desde la Etapa 2.
- Factory: centraliza la LÓGICA DE DECISIÓN de qué implementación crear, separándola del código que la usa.
- Una Factory debe devolver el tipo de la ABSTRACCIÓN (interfaz o clase abstracta), nunca la implementación concreta.
- Combinados, agregar una regla de negocio nueva solo toca la Factory — el código cliente permanece intacto.

## Modelo mental final

Pensá en Strategy como un cajón con distintas herramientas que sirven para el mismo propósito (destornilladores de distinto tamaño), y en Factory como el asistente que elige automáticamente cuál herramienta darte según el tornillo que tenés enfrente — vos nunca tenés que decidir manualmente, ni saber cómo el asistente decide.

## Conexión

Strategy formaliza lo que `Descuento`/`MetodoEnvio` (Etapa 2) y SOLID (Etapa 7) ya establecieron; Factory (nuevo en esta lección) agrega la pieza que faltaba: centralizar CUÁNDO usar cada estrategia, en vez de decidirlo manualmente en cada lugar del código, como hacía `SolidDemo` (Etapa 7).

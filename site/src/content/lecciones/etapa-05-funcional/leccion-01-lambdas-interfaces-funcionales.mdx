## Qué vas a aprender

Después de esta lección serás capaz de leer y escribir expresiones lambda con confianza, reconocer las cuatro interfaces funcionales más usadas de Java (`Predicate`, `Function`, `Consumer`, `Supplier`), y crear tu propia interfaz funcional cuando ninguna de las estándar encaje.

## Por qué necesitas aprenderlo

Ya usaste una lambda sin nombrarla como tal: `repositorio.buscar(p -> p.getNombre().equals("Teclado mecanico"))` en la Etapa 4. Esa lambda encaja en `Predicate<T>`, una interfaz funcional del propio Java. Antes de usar Streams (la próxima lección, que se apoya EN estas interfaces), necesitás entender qué es realmente una lambda y por qué el compilador acepta esa sintaxis tan compacta.

## Qué debes saber antes

Generics (Etapa 4 Lección 3) y el uso de `Predicate<T>` en `RepositorioEnMemoria.buscar` (Etapa 4 Lección 7).

## Problema

Ya usaste `p -> p.getNombre().equals("Teclado mecanico")` como argumento de `buscar`. ¿Qué ES exactamente esa lambda? ¿Cómo sabe Java qué tipo tiene, si no escribiste ningún tipo explícito? Y si mañana necesitás una operación que no encaja en `Predicate`, `Function`, `Consumer` ni `Supplier` — como calcular un descuento sobre un `Product` — ¿cómo la representás?

## Modelo mental

```
interfaz funcional: una interfaz con EXACTAMENTE UN método abstracto.
lambda:             una implementación de ese único método, escrita
                     de forma compacta, sin nombre de clase ni new.

(parametros) -> cuerpo
```

## Explicación sencilla

Una interfaz funcional es una interfaz que tiene un solo método sin implementar. Una lambda es una forma corta de escribir "una instancia de esa interfaz, implementando ese único método" — sin tener que escribir una clase entera con `new NombreClase() { ... }`.

`p -> p.getNombre().equals("Teclado mecanico")` es equivalente a escribir una clase que implementa `Predicate<Product>` con un método `test(Product p) { return p.getNombre().equals("Teclado mecanico"); }` — pero en una sola línea.

## Explicación técnica

`@FunctionalInterface` es una anotación opcional (pero recomendada) que le pide al compilador que verifique que la interfaz tenga exactamente un método abstracto — si agregás un segundo, el compilador rechaza la interfaz con ese anotación.

Java trae interfaces funcionales listas para usar en `java.util.function`, cada una con un propósito distinto según su firma:

- **`Predicate<T>`**: `boolean test(T t)` — evalúa una condición sobre `T`. Ya la usaste en `buscar`.
- **`Function<T, R>`**: `R apply(T t)` — transforma un `T` en un `R` de otro tipo.
- **`Consumer<T>`**: `void accept(T t)` — hace algo con `T`, sin devolver nada.
- **`Supplier<T>`**: `T get()` — no recibe nada, produce un `T`.

Cuando ninguna de estas encaja (por ejemplo, necesitás una operación con una firma específica de tu dominio), podés declarar tu propia interfaz funcional con `@FunctionalInterface` y un único método abstracto, con el nombre y la firma que tenga sentido para tu problema.

## Cómo funciona

1. El compilador infiere el tipo de la lambda por el CONTEXTO donde se usa — si la asignás a una variable `Predicate<Product>`, el compilador sabe que la lambda debe implementar `test(Product) -> boolean`.
2. `CalculadoraDescuento` es una interfaz funcional propia con un único método `calcular(Product) -> double`, porque ninguna interfaz estándar representa "aplicar un descuento a un producto" con ese nombre específico.
3. `diezPorCiento.calcular(teclado)` invoca la lambda como si fuera una llamada de método normal.

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-05/CalculadoraDescuento.java`

```java
@FunctionalInterface
public interface CalculadoraDescuento {
    double calcular(Product producto);
}
```

**Archivo:** `proyecto/etapa-05/LambdasDemo.java`

```java
import java.util.function.Consumer;
import java.util.function.Function;
import java.util.function.Predicate;
import java.util.function.Supplier;

public class LambdasDemo {
    public static void main(String[] args) {
        Product teclado = new Product("Teclado mecanico", 25000.0, 15);

        Predicate<Product> tieneStockBajo = p -> p.getStock() < 20;
        System.out.println("Stock bajo: " + tieneStockBajo.test(teclado));

        Function<Product, String> resumen = p -> p.getNombre() + " ($" + p.getPrecio() + ")";
        System.out.println(resumen.apply(teclado));

        Consumer<Product> imprimir = p -> System.out.println("Producto: " + p.getNombre());
        imprimir.accept(teclado);

        Supplier<Product> productoPorDefecto = () -> new Product("Producto generico", 0.0, 0);
        Product generico = productoPorDefecto.get();
        System.out.println("Generico: " + generico.getNombre());

        CalculadoraDescuento diezPorCiento = p -> p.getPrecio() * 0.9;
        System.out.println("Precio con descuento: " + diezPorCiento.calcular(teclado));
    }
}
```

## Predice

`tieneStockBajo` chequea `stock < 20`, y `teclado` tiene stock 15. ¿Qué imprime la primera línea?

<Reveal titulo="Resultado real (compilado y ejecutado)">
```
Stock bajo: true
Teclado mecanico ($25000.0)
Producto: Teclado mecanico
Generico: Producto generico
Precio con descuento: 22500.0
```
</Reveal>

## Explicación paso a paso

- `tieneStockBajo.test(teclado)` da `true` porque 15 < 20 — la lambda se ejecutó como el método `test` de `Predicate<Product>`.
- `resumen.apply(teclado)` devuelve un `String`, no un `boolean` ni `void` — porque `Function<T, R>` transforma, no evalúa ni consume.
- `imprimir.accept(teclado)` no devuelve nada — solo IMPRIME, que es justamente lo que `Consumer<T>` representa.
- `productoPorDefecto.get()` no recibió NINGÚN argumento — `Supplier<T>` es la única interfaz funcional de las cuatro cuyo método no toma parámetros.
- `diezPorCiento.calcular(teclado)` usa una interfaz que vos mismo escribiste, no una de `java.util.function` — y el compilador la trata exactamente igual que a las estándar.

## Mini reto de debugging

```java
Function<Product, String> resumen = p -> p.getNombre() + " ($" + p.getPrecio() + ")";
boolean resultado = resumen.apply(teclado);
```

¿Por qué esta línea no compila?

<Reveal titulo="Diagnóstico">
`resumen.apply(teclado)` devuelve un `String` (porque `Function<Product, String>` así lo define), y se está intentando asignar ese `String` a una variable `boolean` — un error de tipos, no de lambdas. El tipo de retorno de una `Function<T, R>` es `R`, en este caso `String`, no `boolean`.
</Reveal>

## Resumen

- Una interfaz funcional tiene exactamente un método abstracto; una lambda es una implementación compacta de ese método.
- `Predicate<T>` evalúa, `Function<T, R>` transforma, `Consumer<T>` actúa sin devolver nada, `Supplier<T>` produce sin recibir nada.
- Cuando ninguna interfaz estándar encaja, se puede declarar una interfaz funcional propia con `@FunctionalInterface`.
- El compilador infiere el tipo de una lambda por el contexto (la variable o parámetro al que se asigna), no por la lambda en sí misma.

## Modelo mental final

Pensá en una interfaz funcional como un molde con un solo hueco de la forma exacta que necesitás llenar — la lambda es lo que rellena ese hueco, sin que tengas que fabricar un molde nuevo (una clase) cada vez.

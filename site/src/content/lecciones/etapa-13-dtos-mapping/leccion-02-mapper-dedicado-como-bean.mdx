## Qué vas a aprender

Después de esta lección serás capaz de centralizar TODA la lógica de mapeo entre `Product` y sus DTOs (`NuevoProductoRequest`, `ProductoResponse`) en una clase `ProductoMapper` dedicada, gestionada por Spring como cualquier otro bean (Etapa 10), y de explicar cuándo esta separación vale más que el método estático de la lección anterior.

## Por qué necesitas aprenderlo

`ProductoResponse.desde(Product)` (Etapa 13 Lección 1) funcionó bien para un mapeo SIMPLE, sin dependencias externas. Pero un mapeo real puede necesitar colaborar con otros servicios (por ejemplo, calcular un precio con impuestos usando un servicio externo) — un método estático NO puede recibir dependencias inyectadas; un bean de Spring sí.

## Qué debes saber antes

`ProductoResponse.desde` (Etapa 13 Lección 1), Dependency Injection por constructor (Etapa 8/10) — el `Mapper` de esta lección es, en esencia, otro bean más inyectado en el controlador.

## Recupera lo aprendido

¿Por qué `ProductoResponse` NO incluye `fechaIngreso`, aunque `Product` sí la tenga?

<Reveal titulo="Respuesta">
Porque el DTO decide explícitamente qué exponer hacia afuera — fechaIngreso es un detalle interno del dominio que no se consideró necesario para los clientes de la API.
</Reveal>

## Problema

`ProductoResponse.desde(Product)` es un método ESTÁTICO — no puede recibir ninguna dependencia inyectada por Spring. Si mañana necesitaras que el mapeo consultara OTRO servicio (por ejemplo, un servicio de conversión de moneda antes de mostrar el precio), un método estático no tiene forma de recibir ese servicio via `@Autowired`. Necesitás que el MAPEO EN SÍ sea un bean gestionado por Spring.

## Modelo mental

```
Mapeo estático (Etapa 13 L1):  método fijo en el propio DTO,
                                sin posibilidad de dependencias.

Mapper dedicado (esta lección): una clase separada, bean de
                                 Spring, que PUEDE recibir
                                 dependencias como cualquier otro
                                 @Component/@Service.
```

## Explicación sencilla

En vez de que `ProductoResponse` sepa construirse A SÍ MISMO a partir de un `Product` (mapeo estático), una clase SEPARADA (`ProductoMapper`) se encarga de TODA la traducción entre `Product` y sus DTOs — en ambas direcciones (`toResponse` y `toEntity`). Esta clase es un bean común de Spring (`@Component`), así que se inyecta por constructor exactamente igual que `RepositorioProductos` o `ServicioVentas`.

## Explicación técnica

`ProductoResponse` vuelve a ser un `record` PURO — solo datos, sin ningún método de mapeo. `ProductoMapper` (anotada `@Component`) concentra `toResponse(Product) -> ProductoResponse` y `toEntity(NuevoProductoRequest) -> Product`, los DOS sentidos del mapeo, en un solo lugar. `ProductoController` recibe `ProductoMapper` por constructor (el CUARTO parámetro, junto a `RepositorioProductos`, `ServicioVentas`) — Spring lo resuelve automáticamente, igual que cualquier otra dependencia (Etapa 10 Lección 1).

La ventaja concreta: si `ProductoMapper` necesitara MAÑANA una dependencia adicional (por ejemplo, un `ServicioDeConversionDeMoneda`), bastaría con agregarla a SU PROPIO constructor — Spring la inyectaría automáticamente, sin que `ProductoController` ni ningún otro código cliente se enteren del cambio. Un método estático en el DTO nunca podría hacer eso: los métodos estáticos no participan del contenedor de Spring, no pueden recibir `@Autowired`.

`mapper::toResponse` (method reference, Etapa 5 Lección 2) sigue funcionando exactamente igual dentro de `.map()` en el `Stream` de `listar()` — la única diferencia es que ahora la referencia apunta a un método de INSTANCIA (`mapper.toResponse`) en vez de uno estático (`ProductoResponse.desde`).

## Cómo funciona

1. `ProductoResponse` vuelve a ser un `record(String nombre, double precio, int stock)` sin ningún método adicional — solo la FORMA de los datos.
2. `ProductoMapper` (`@Component`) implementa `toResponse` y `toEntity`, con la MISMA lógica de copia de campos que antes vivía en `ProductoResponse.desde` y en `crear()`.
3. `ProductoController` recibe `ProductoMapper mapper` por constructor — cuarto bean inyectado, junto a `RepositorioProductos`, `ServicioVentas`.
4. Cada método del controlador reemplaza `ProductoResponse.desde(producto)` por `mapper.toResponse(producto)`, y el `new Product(...)` manual de `crear()` por `mapper.toEntity(request)`.

## Ejemplo mínimo

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/ProductoResponse.java` (simplificado)

```java
package com.curso.catalogo;

public record ProductoResponse(String nombre, double precio, int stock) {
}
```

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/ProductoMapper.java`

```java
package com.curso.catalogo;

import org.springframework.stereotype.Component;

@Component
public class ProductoMapper {

    public ProductoResponse toResponse(Product producto) {
        return new ProductoResponse(producto.getNombre(), producto.getPrecio(), producto.getStock());
    }

    public Product toEntity(NuevoProductoRequest request) {
        return new Product(request.nombre(), request.precio(), request.stock());
    }
}
```

**Fragmento actualizado de `proyecto-maven/src/main/java/com/curso/catalogo/ProductoController.java`:**

```java
private final ProductoMapper mapper;

public ProductoController(RepositorioProductos repositorio, ServicioVentas servicioVentas, ProductoMapper mapper) {
    this.repositorio = repositorio;
    this.servicioVentas = servicioVentas;
    this.mapper = mapper;
}

@GetMapping
public List<ProductoResponse> listar() {
    return repositorio.listarTodos().stream()
            .map(mapper::toResponse)
            .collect(Collectors.toList());
}

@PostMapping
public ProductoResponse crear(@RequestBody NuevoProductoRequest request) {
    Product producto = mapper.toEntity(request);
    repositorio.agregar(producto);
    return mapper.toResponse(producto);
}
```

## Predice

`ProductoMapper` es un bean inyectado por constructor, igual que `RepositorioProductos` y `ServicioVentas`. Al crear "Auriculares" (precio 15000, stock 20) y venderle 3 unidades, ¿qué stock esperás en la respuesta de la venta?

<Reveal titulo="Resultado real (ejecutado con Spring Boot + curl)">
```
$ curl -s http://localhost:8080/api/productos
[{"nombre":"Teclado mecanico","precio":25000.0,"stock":10}]

$ curl -s -X POST http://localhost:8080/api/productos \
    -H "Content-Type: application/json" \
    -d '{"nombre":"Auriculares","precio":15000.0,"stock":20}'
{"nombre":"Auriculares","precio":15000.0,"stock":20}

$ curl -s -X POST "http://localhost:8080/api/productos/Auriculares/vender?cantidad=3"
{"nombre":"Auriculares","precio":15000.0,"stock":17}
```
</Reveal>

## Explicación paso a paso

- El JSON de `GET /api/productos` sigue sin mostrar `fechaIngreso` — el comportamiento del DTO no cambió, solo QUIÉN hace el mapeo.
- "Auriculares" se creó correctamente con `mapper.toEntity(request)` construyendo el `Product`, y `mapper.toResponse(producto)` devolviendo el DTO — dos llamadas a un bean inyectado, en vez de un método estático y un `new` manual.
- Stock final `17` (20 - 3) confirma que `ServicioVentas.venderProducto` (Etapa 10) sigue funcionando exactamente igual — el `Mapper` no tocó NINGUNA lógica de negocio, solo la traducción de datos.
- Todo el comportamiento observable es IDÉNTICO al de la Lección 1 — el refactor cambió la ESTRUCTURA interna (dónde vive el mapeo), no el RESULTADO, exactamente el criterio de un refactor válido (Etapa 7 Lección 1).

## Ejemplo aplicado al proyecto

`ProductoMapper` centraliza AMBAS direcciones del mapeo (`Product` → DTO, DTO → `Product`) en un solo lugar con nombre propio — si mañana necesitás lógica de mapeo más compleja (formatear un precio, calcular un campo derivado), tenés un único punto de extensión, en vez de dispersar esa lógica entre distintos DTOs y el propio controlador.

## Ejercicio guiado

Agregá un método `toEntityConValidacion(NuevoProductoRequest request)` a `ProductoMapper` que lance `IllegalArgumentException` si `request.precio()` es negativo, ANTES de construir el `Product` (que también valida esto en su constructor, Etapa 2, pero acá se práctica validar en la capa de mapeo).

<Reveal titulo="Pista 1">
Un simple `if (request.precio() < 0) { throw new IllegalArgumentException(...); }` antes del `return new Product(...)` alcanza para este ejercicio.
</Reveal>

<Reveal titulo="Solución">
```java
public Product toEntityConValidacion(NuevoProductoRequest request) {
    if (request.precio() < 0) {
        throw new IllegalArgumentException("El precio no puede ser negativo");
    }
    return new Product(request.nombre(), request.precio(), request.stock());
}
```
</Reveal>

## Mini reto de debugging

```java
@RestController
@RequestMapping("/api/productos")
public class ProductoController {

    private final ProductoMapper mapper;

    public ProductoController() {
        this.mapper = new ProductoMapper();
    }
}
```

¿Qué principio de las Etapas 8 y 10 viola este código, aunque compile y funcione?

<Reveal titulo="Diagnóstico">
Viola Dependency Injection (Etapa 8 Lección 1, Etapa 10 Lección 1): `ProductoMapper` se crea con `new` DENTRO del controlador, en vez de recibirse por constructor con `@Autowired`/inyección automática. Aunque `ProductoMapper` no tenga dependencias propias hoy, crear instancias manualmente rompe el patrón — y si mañana `ProductoMapper` necesitara una dependencia inyectada, este código dejaría de compilar o de funcionar correctamente, porque `new ProductoMapper()` no pasa por el contenedor de Spring.
</Reveal>

## Resumen

- Un Mapper dedicado (`@Component`) centraliza el mapeo bidireccional entre dominio y DTOs.
- A diferencia de un método estático, un Mapper PUEDE recibir dependencias inyectadas por Spring.
- Consolidar el mapeo en UN solo lugar evita lógica duplicada entre distintos DTOs y el controlador.
- El refactor de método estático a Mapper dedicado preserva el comportamiento exacto — la prueba con `curl` lo confirma.

## Modelo mental final

Pensá en el método estático como un traductor que solo sabe traducir UN documento fijo, sin acceso a un diccionario externo; y en el Mapper dedicado como un traductor profesional con acceso a TODAS las herramientas (diccionarios, otros expertos) que pueda necesitar — mismo trabajo básico, pero uno puede crecer con el proyecto y el otro no.

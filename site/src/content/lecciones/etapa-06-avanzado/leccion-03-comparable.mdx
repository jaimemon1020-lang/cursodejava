## Qué vas a aprender

Después de esta lección serás capaz de implementar `Comparable<T>` en una clase propia para darle un "orden natural" por defecto, usable con `Collections.sort()` y `TreeSet` sin pasar ningún `Comparator` explícito — y de decidir cuándo esto tiene más sentido que el `Comparator` externo que ya usaste con Streams.

## Por qué necesitas aprenderlo

En la Etapa 5 Lección 3 usaste `Comparator.comparingDouble(Product::getPrecio)` para ordenar por precio — un criterio EXTERNO, definido en el momento de ordenar. Pero si "por precio" es casi siempre el orden que tiene sentido para tu catálogo, tener que repetir ese `Comparator` cada vez es redundante. `Comparable` resuelve eso: le da a la clase un orden natural incorporado.

## Qué debes saber antes

`Comparator`, `sorted`, `Comparator.comparingDouble` (Etapa 5 Lección 3) — esta lección introduce la contraparte de eso: un orden definido DENTRO de la clase.

## Recupera lo aprendido

¿Por qué hay que sobrescribir `hashCode()` cada vez que se sobrescribe `equals()`?

<Reveal titulo="Respuesta">
Porque el contrato de Java exige que si a.equals(b) es true, a.hashCode() debe ser igual a b.hashCode() — si no, HashSet y HashMap fallan en detectar duplicados de forma consistente, porque usan hashCode() para ubicar el bucket y equals() para confirmar.
</Reveal>

## Problema

`Collections.sort(productos)` (sin pasar ningún criterio) no compila si `Product` no le dice a Java CÓMO ordenarse — a diferencia de `sorted(Comparator...)` en Streams, donde el criterio se pasa explícitamente cada vez. Necesitás una forma de decir "así es como se comparan dos productos, por defecto", una sola vez, dentro de la clase.

## Modelo mental

```
Comparable<T>:  "yo (esta clase) SÉ compararme con otro T" — un método,
                 compareTo(T otro), definido DENTRO de la clase.
Comparator<T>:  "yo defino un criterio EXTERNO para comparar dos T" —
                 se pasa como argumento, puede haber varios distintos.
```

## Explicación sencilla

`Comparable<T>` es una interfaz con un solo método: `compareTo(T otro)`, que devuelve un número negativo si `this` va ANTES que `otro`, cero si son "iguales" en orden, o positivo si `this` va DESPUÉS. Cuando una clase implementa `Comparable`, herramientas como `Collections.sort()` y `TreeSet` saben ordenarla SIN que le pases ningún criterio adicional — usan ese `compareTo` automáticamente.

## Explicación técnica

`Comparable<Product>` se implementa con `class Product implements Comparable<Product>` y un único método `compareTo(Product otro)`. La convención (igual que con `Comparator`) es usar `Double.compare`, `Integer.compare`, etc. para tipos primitivos, en vez de restar directamente (`this.precio - otro.precio`), que puede dar resultados incorrectos con `double` por problemas de redondeo o overflow con `int`.

Una clase puede tener SOLO UN orden natural (un solo `compareTo`), pero puede combinarse con TANTOS `Comparator` externos como necesites para casos alternativos — como ya viste en la Etapa 5 con `Comparator.comparingDouble(...).thenComparing(...)`. La regla de diseño: el orden natural (`Comparable`) debería ser el que tiene sentido en LA MAYORÍA de los casos; los `Comparator` externos cubren las excepciones puntuales.

`TreeSet<T>` requiere que `T` implemente `Comparable` (o que le pases un `Comparator` al construirlo) — internamente usa ese orden para mantener sus elementos siempre ordenados, y también para decidir si dos elementos son "el mismo" a los fines del `Set` (¡ojo! `TreeSet` usa `compareTo() == 0` para eso, NO `equals()` — una diferencia sutil frente a `HashSet`, que sí usa `equals()`/`hashCode()`).

## Cómo funciona

1. `Product implements Comparable<Product>` declara que la clase tiene un orden natural.
2. `compareTo(Product otro)` devuelve `Double.compare(this.precio, otro.precio)` — negativo si `this` es más barato, positivo si es más caro, cero si tienen el mismo precio.
3. `Collections.sort(productos)` ordena la lista completa usando ese `compareTo`, sin necesitar ningún argumento adicional.
4. `new TreeSet<>(productos)` construye un `Set` que mantiene el orden natural automáticamente — `.first()` da el elemento "menor" (más barato), `.last()` da el "mayor" (más caro).
5. `productos.sort(Comparator.comparing(Product::getNombre))` sigue funcionando igual que en la Etapa 5: un `Comparator` externo IGNORA el orden natural para ese llamado puntual, sin modificar el `compareTo` de la clase.

## Ejemplo mínimo

**Fragmento agregado a `proyecto/etapa-02/Product.java`:**

```java
public class Product implements Comparable<Product> {
    // ... campos, constructor, equals, hashCode, toString ...

    @Override
    public int compareTo(Product otro) {
        return Double.compare(this.precio, otro.precio);
    }
}
```

**Archivo:** `proyecto/etapa-06/ComparableDemo.java`

```java
import java.util.ArrayList;
import java.util.Collections;
import java.util.Comparator;
import java.util.List;
import java.util.TreeSet;

public class ComparableDemo {
    public static void main(String[] args) {
        List<Product> productos = new ArrayList<>();
        productos.add(new Product("Teclado mecanico", 25000.0, 15));
        productos.add(new Product("Mouse inalambrico", 8000.0, 40));
        productos.add(new Product("Monitor 27 pulgadas", 180000.0, 3));

        Collections.sort(productos);
        System.out.println("Orden natural (Comparable, por precio ascendente):");
        for (Product p : productos) {
            System.out.println("  " + p.getNombre() + ": " + p.getPrecio());
        }

        TreeSet<Product> ordenadoPorNaturaleza = new TreeSet<>(productos);
        System.out.println("Mas barato (TreeSet.first): " + ordenadoPorNaturaleza.first().getNombre());
        System.out.println("Mas caro (TreeSet.last): " + ordenadoPorNaturaleza.last().getNombre());

        productos.sort(Comparator.comparing(Product::getNombre));
        System.out.println("Orden alternativo (Comparator externo, por nombre):");
        for (Product p : productos) {
            System.out.println("  " + p.getNombre());
        }
    }
}
```

## Predice

El orden natural de `Product` es por precio ascendente. Los precios son 25000, 8000 y 180000. ¿Qué producto queda primero después de `Collections.sort(productos)`?

## Explicación paso a paso

- `Collections.sort(productos)` usó `compareTo` SIN que se le pasara ningún criterio — Mouse (8000) quedó primero, Monitor (180000) último, exactamente el orden natural definido en `Product`.
- `TreeSet.first()` y `.last()` confirman lo mismo desde otra estructura: el `TreeSet` mantiene el orden natural automáticamente en todo momento, no solo al construirse.
- El último bloque usa `Comparator.comparing(Product::getNombre)` — un criterio EXTERNO que ordena alfabéticamente, completamente distinto al orden natural por precio, sin que `Product.compareTo` haya cambiado en absoluto.
- Los dos órdenes (natural por precio, externo por nombre) conviven sin conflicto: uno vive DENTRO de la clase, el otro se pasa como argumento cuando se necesita.

## Ejemplo aplicado al proyecto

Con `Comparable`, cualquier código que solo necesite "el orden por defecto" del catálogo (`Collections.sort(productos)`, `new TreeSet<>(productos)`) obtiene el orden por precio sin que cada lugar del código tenga que repetir `Comparator.comparingDouble(Product::getPrecio)` — esa decisión vive en un solo lugar: la clase misma.

## Mini reto de debugging

```java
TreeSet<Product> catalogo = new TreeSet<>();
catalogo.add(new Product("Teclado mecanico", 25000.0, 15));
catalogo.add(new Product("Teclado gamer", 25000.0, 8));
System.out.println(catalogo.size());
```

¿El `TreeSet` termina con 1 o 2 elementos, sabiendo que ambos productos tienen el MISMO precio (25000.0) pero nombres y stock distintos?

<Reveal titulo="Diagnóstico">
Termina con `1` elemento. `TreeSet` NO usa `equals()`/`hashCode()` para decidir duplicados — usa `compareTo() == 0`. Como `Product.compareTo` solo compara `precio`, y ambos tienen 25000.0, `TreeSet` los considera "el mismo" a pesar de tener nombres y stock distintos — una diferencia importante frente a `HashSet`, que sí los trataría como diferentes porque `equals()` compara nombre, precio Y stock.
</Reveal>

## Resumen

- `Comparable<T>` define el orden natural DENTRO de la clase con un único método `compareTo`.
- `Collections.sort()` (sin argumentos) y `TreeSet` requieren que la clase implemente `Comparable`, o que se les pase un `Comparator` externo.
- `Comparator` (Etapa 5) sigue siendo la herramienta correcta para órdenes ALTERNATIVOS, sin tocar la clase.
- `TreeSet` detecta "duplicados" con `compareTo() == 0`, NO con `equals()`/`hashCode()` — una diferencia importante frente a `HashSet`.

## Modelo mental final

Pensá en `Comparable` como el orden "de fábrica" de una clase — el que trae por defecto sin que nadie tenga que pedirlo — y en `Comparator` como una regla de ordenamiento que armás vos mismo para una ocasión específica, sin modificar la fábrica.

## Conexión

`Comparator` (Etapa 5 Lección 3) y `Comparable` (esta lección) resuelven el mismo problema (ordenar) desde dos ángulos complementarios: uno vive fuera de la clase para casos puntuales, el otro vive dentro para el caso por defecto. `equals`/`hashCode` (Lección 2) y `compareTo` (esta lección) también son complementarios pero DISTINTOS: uno define igualdad de CONTENIDO, el otro define ORDEN — y como viste en el mini reto, `TreeSet` usa uno y `HashSet` usa el otro.

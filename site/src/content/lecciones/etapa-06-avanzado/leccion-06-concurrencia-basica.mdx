## Qué vas a aprender

Después de esta lección serás capaz de crear y ejecutar un hilo (`Thread`) a partir de una tarea (`Runnable`), esperar su finalización con `join()`, y explicar por qué incrementar una variable compartida desde varios hilos sin protección puede dar un resultado INCORRECTO — y cómo `AtomicInteger` lo resuelve.

## Por qué necesitas aprenderlo

Todo el código del curso hasta ahora ejecuta en un único flujo secuencial: una línea termina antes de que empiece la siguiente. Los programas reales (sobre todo con Spring, más adelante en el curso) manejan MÚLTIPLES peticiones simultáneas — necesitás un modelo mental de qué puede salir mal cuando dos flujos de ejecución tocan el mismo dato al mismo tiempo.

## Qué debes saber antes

Stack y Heap (Etapa 6 Lección 1) — cada hilo tiene su PROPIO Stack, pero todos comparten el MISMO Heap, que es justamente la raíz del problema que esta lección explica.

## Recupera lo aprendido

¿Por qué `Files.delete()` lanza una excepción si el archivo ya no existe, en vez de fallar en silencio?

<Reveal titulo="Respuesta">
Es una decisión de diseño de NIO: fallar explícitamente ayuda a detectar bugs temprano (por ejemplo, borrar el mismo archivo dos veces por error), en vez de que el programa siga como si nada hubiera pasado cuando en realidad algo inesperado ocurrió.
</Reveal>

## Problema

Necesitás ejecutar una tarea SIN bloquear el flujo principal del programa (por ejemplo, procesar algo en paralelo), y necesitás sumar un contador desde MÚLTIPLES tareas simultáneas sin perder incrementos. Hacerlo de forma ingenua con una variable `int` compartida puede dar un resultado menor al esperado, de forma impredecible.

## Modelo mental

```
Thread:    un flujo de ejecución independiente, con su PROPIO Stack.
Runnable:  la tarea que ese hilo va a ejecutar (una interfaz funcional:
           void run()).
join():    el hilo que llama a join() ESPERA a que el otro termine
           antes de seguir.
```

## Explicación sencilla

Un `Thread` es un flujo de ejecución adicional que corre "en paralelo" al principal. `Runnable` es la tarea que le das a ese hilo — literalmente una interfaz funcional con un método `run()`, así que se puede escribir como una lambda, igual que `Predicate` o `Consumer` en la Etapa 5. `hilo.start()` arranca el hilo; `hilo.join()` hace que el hilo que llama espere a que el otro termine antes de continuar — sin `join()`, el programa principal podría terminar ANTES de que el hilo secundario haya hecho su trabajo.

## Explicación técnica

Cada `Thread` tiene su PROPIO Stack (Etapa 6 Lección 1) — sus propias variables locales son privadas a ese hilo. Pero TODOS los hilos de un mismo programa comparten el MISMO Heap — cualquier objeto puede ser accedido por múltiples hilos simultáneamente. Ahí está el problema: si dos hilos leen y modifican la MISMA variable en el Heap al mismo tiempo, sin coordinación, pueden "pisarse" — un hilo lee el valor, otro lo modifica, el primero escribe de vuelta un valor ya desactualizado, y se pierde un incremento.

`int contador++` NO es una operación atómica (de un solo paso indivisible) — en realidad son tres pasos: leer el valor actual, sumarle uno, y guardar el resultado. Si dos hilos ejecutan esos tres pasos entrelazados, es posible que ambos lean el MISMO valor inicial antes de que ninguno haya guardado su resultado, y el incremento de uno de los dos se "pierda" — esto se llama una **condición de carrera** (race condition).

`AtomicInteger` (de `java.util.concurrent.atomic`) resuelve esto: `incrementAndGet()` garantiza que leer, sumar y guardar ocurra como una sola operación indivisible, sin que otro hilo pueda intercalarse en el medio — sin necesitar que vos escribas sincronización manual con `synchronized`.

## Cómo funciona

1. `Runnable tarea = () -> { ... }` define el trabajo a ejecutar como una lambda — `Runnable` es una interfaz funcional (`void run()`), igual patrón que `Predicate`/`Consumer` de la Etapa 5.
2. `new Thread(tarea, "hilo-inventario")` crea el hilo con un nombre identificable (útil para debugging con múltiples hilos).
3. `hilo.start()` arranca el hilo de forma asíncrona — el `main` NO espera automáticamente a que termine.
4. `hilo.join()` bloquea el hilo `main` hasta que `hilo` termine completamente — por eso el orden de salida es 100% predecible en el primer ejemplo.
5. En `ContadorConcurrenteDemo`, dos hilos ejecutan la MISMA lambda `incrementar`, cada uno sumando 1000 veces a un `AtomicInteger` compartido.
6. `hilo1.join(); hilo2.join();` asegura que AMBOS hilos terminaron antes de leer `contador.get()` — sin esas dos líneas, el programa podría imprimir el contador ANTES de que los hilos terminaran de incrementar.

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-06/ThreadsDemo.java`

```java
public class ThreadsDemo {
    public static void main(String[] args) throws InterruptedException {
        Runnable tarea = () -> {
            System.out.println("Ejecutando en: " + Thread.currentThread().getName());
            for (int i = 1; i <= 3; i++) {
                System.out.println("Paso " + i);
            }
        };

        Thread hilo = new Thread(tarea, "hilo-inventario");
        System.out.println("Antes de start, hilo principal: " + Thread.currentThread().getName());
        hilo.start();
        hilo.join();
        System.out.println("Despues de join: el hilo termino, el main continua");
    }
}
```

**Archivo:** `proyecto/etapa-06/ContadorConcurrenteDemo.java`

```java
import java.util.concurrent.atomic.AtomicInteger;

public class ContadorConcurrenteDemo {
    public static void main(String[] args) throws InterruptedException {
        AtomicInteger contador = new AtomicInteger(0);
        Runnable incrementar = () -> {
            for (int i = 0; i < 1000; i++) {
                contador.incrementAndGet();
            }
        };

        Thread hilo1 = new Thread(incrementar);
        Thread hilo2 = new Thread(incrementar);
        hilo1.start();
        hilo2.start();
        hilo1.join();
        hilo2.join();

        System.out.println("Contador final (2 hilos x 1000 incrementos): " + contador.get());
    }
}
```

## Predice

Dos hilos incrementan el mismo `AtomicInteger` 1000 veces cada uno, y `main` espera a ambos con `join()` antes de imprimir. ¿Qué valor final esperás, sabiendo que `AtomicInteger` garantiza que ningún incremento se pierde?

<Reveal titulo="Resultado real (compilado y ejecutado)">
```
Antes de start, hilo principal: main
Ejecutando en: hilo-inventario
Paso 1
Paso 2
Paso 3
Despues de join: el hilo termino, el main continua
```

```

```

## Explicación paso a paso

- "Antes de start" imprime desde `main`, ANTES de que el hilo secundario haya hecho nada — el orden es garantizado porque esa línea se ejecuta antes de `hilo.start()`.
- "Ejecutando en: hilo-inventario" y los tres "Paso" confirman que el `Runnable` corrió en un hilo CON NOMBRE PROPIO, distinto de `main`.
- "Despues de join" aparece SIEMPRE al final, nunca antes — `hilo.join()` garantiza que `main` esperó a que el hilo secundario terminara por completo.
- El contador final da EXACTAMENTE `2000` (1000 + 1000), sin ningún incremento perdido — gracias a que `AtomicInteger.incrementAndGet()` es atómico, y a que `join()` en ambos hilos asegura leer el valor DESPUÉS de que terminaron de incrementar.
- Si `contador` fuera un `int` normal con `contador++` (no atómico) en vez de `AtomicInteger`, el resultado podría dar MENOS de 2000 de forma impredecible — una condición de carrera clásica.

## Mini reto de debugging

```java
Thread hilo = new Thread(() -> System.out.println("Trabajo en segundo plano"));
hilo.start();
System.out.println("Main termino");
```

Sin `join()`, ¿está garantizado que "Trabajo en segundo plano" se imprima ANTES que "Main termino"?

<Reveal titulo="Diagnóstico">
No, NO está garantizado. Sin `join()`, `main` sigue ejecutando inmediatamente después de `start()`, sin esperar al hilo secundario — el orden entre "Trabajo en segundo plano" y "Main termino" depende de cómo el sistema operativo decida repartir tiempo de CPU entre ambos hilos, y puede variar entre ejecuciones. ## Resumen

- Cada `Thread` tiene su propio Stack, pero todos comparten el mismo Heap — ahí nace la necesidad de coordinación.
- `Runnable` es la tarea (interfaz funcional `void run()`); `Thread` es el hilo que la ejecuta; `join()` espera su finalización.
- `contador++` no es atómico: leer+sumar+guardar pueden entrelazarse entre hilos, perdiendo incrementos (condición de carrera).
- `AtomicInteger` garantiza operaciones atómicas sobre un valor compartido, sin necesitar `synchronized` manual para casos simples.

## Modelo mental final

Pensá en dos cajeros (hilos) cobrando del mismo cajón de dinero (una variable compartida en el Heap): si ambos miran el total, hacen la cuenta por separado, y después escriben el resultado sin coordinarse, uno de los dos puede pisar el trabajo del otro — `AtomicInteger` es como si cada movimiento de dinero fuera una operación instantánea e indivisible, sin que nadie pueda interrumpirla a la mitad.

## Qué vas a aprender

Después de esta lección tendrás el mapa concreto de cómo se extrajo la interfaz `RepositorioProductos` del código que ya existía, y cómo se creó el adaptador `RepositorioProductosEnMemoria` — el MISMO patrón que aplicarías para migrar a PostgreSQL, MongoDB o cualquier otra tecnología de persistencia.

## Por qué necesitas aprenderlo

Ver el concepto (Etapa 20 L1) no es suficiente — necesitás ver el CODE DIFF, la diferencia línea por línea entre "antes" y "después", para internalizar que el refactoring hexagonal NO es magia: es extraer una interfaz, mover la implementación, y no tocar nada más.

## Qué debes saber antes

`RepositorioProductos` original (Etapa 8 L1), el concepto de puerto y adaptador (Etapa 20 L1).

## Recupera lo aprendido

¿Cuál es la diferencia entre un puerto de entrada (driving) y un puerto de salida (driven)?

<Reveal titulo="Respuesta">
Puerto de entrada: define cómo el mundo exterior USA el dominio (ej. interfaz `ServicioVentas`). Puerto de salida: define qué necesita el dominio DEL mundo exterior (ej. interfaz `RepositorioProductos`). La dirección de la dependencia es la misma en ambos casos: hacia el dominio.
</Reveal>

## Problema

`RepositorioProductos` era una CLASE concreta con `@Repository` que implementaba almacenamiento en `ArrayList`. `ServicioVentas` y `ProductoController` la usaban directamente. Si quisieras cambiar a PostgreSQL, tendrías que REESCRIBIR `RepositorioProductos` — la misma clase que contiene tu dominio y tus controladores. Una clase, dos responsabilidades.

## Modelo mental

```
ANTES:
RepositorioProductos.java  ← @Repository + ArrayList (TODO junto)

AHORA:
RepositorioProductos.java  ← INTERFAZ pura (QUÉ)
        ▲
        │  implements
        │
RepositorioProductosEnMemoria.java  ← @Repository + ArrayList (CÓMO)
```

## Explicación sencilla

Extraer una interfaz de una clase existente es la operación de refactoring más común en arquitectura hexagonal: identificás qué MÉTODOS definen el contrato (el QUÉ), los ponés en una interfaz, y la implementación concreta (el CÓMO) se vuelve un adaptador que implementa esa interfaz. El resto del código (ServicioVentas, ProductoController, tests) sigue funcionando sin cambios porque ya usaban `RepositorioProductos` como tipo.

## Cómo funciona

1. **ANTES**: `RepositorioProductos` era una clase `@Repository` con `ArrayList<Product>` y métodos `agregar`, `listarTodos`, `buscar`.
2. **PASO 1**: Convertir la clase en interfaz — solo las firmas de los métodos, sin implementación, sin `@Repository`, sin `ArrayList`.
3. **PASO 2**: Crear `RepositorioProductosEnMemoria` en `infraestructura/persistencia/` — la implementación concreta con `@Repository` y `ArrayList`.
4. **RESULTADO**: `ServicioVentas` y `ProductoController` siguen inyectando `RepositorioProductos` (ahora interfaz). Spring, vía DI, inyecta automáticamente `RepositorioProductosEnMemoria` (el único bean que implementa esa interfaz).
5. **VERIFICACIÓN**: `./mvnw test` — 11 tests, 0 fallos. Mismo comportamiento, arquitectura mejorada.

## Ejemplo mínimo — el diff completo

**ANTES — `RepositorioProductos.java` (clase concreta):**
```java
package com.curso.catalogo;

import java.util.ArrayList;
import java.util.List;
import java.util.Optional;
import java.util.function.Predicate;

import org.springframework.stereotype.Repository;

@Repository
public class RepositorioProductos {

    private final List<Product> productos = new ArrayList<>();

    public void agregar(Product producto) {
        this.productos.add(producto);
    }

    public List<Product> listarTodos() {
        return this.productos;
    }

    public Optional<Product> buscar(Predicate<Product> criterio) {
        return this.productos.stream().filter(criterio).findFirst();
    }
}
```

**DESPUÉS — `RepositorioProductos.java` (interfaz = puerto):**
```java
package com.curso.catalogo;

import java.util.List;
import java.util.Optional;
import java.util.function.Predicate;

public interface RepositorioProductos {

    void agregar(Product producto);

    List<Product> listarTodos();

    Optional<Product> buscar(Predicate<Product> criterio);
}
```

Cambios:
- `public class` → `public interface`
- Sin `@Repository` (las interfaces no son beans)
- Sin `ArrayList<Product> productos` (la interfaz no tiene estado)
- Sin implementación en los métodos (solo firmas, llevan `;`)
- Sin imports de `ArrayList` ni `@Repository`

**NUEVO — `infraestructura/persistencia/RepositorioProductosEnMemoria.java` (adaptador):**
```java
package com.curso.catalogo.infraestructura.persistencia;

import java.util.ArrayList;
import java.util.List;
import java.util.Optional;
import java.util.function.Predicate;

import org.springframework.stereotype.Repository;

import com.curso.catalogo.Product;
import com.curso.catalogo.RepositorioProductos;

@Repository
public class RepositorioProductosEnMemoria implements RepositorioProductos {

    private final List<Product> productos = new ArrayList<>();

    @Override
    public void agregar(Product producto) {
        this.productos.add(producto);
    }

    @Override
    public List<Product> listarTodos() {
        return this.productos;
    }

    @Override
    public Optional<Product> buscar(Predicate<Product> criterio) {
        return this.productos.stream().filter(criterio).findFirst();
    }
}
```

Cambios respecto al original:
- Package: `com.curso.catalogo.infraestructura.persistencia`
- `implements RepositorioProductos` (la interfaz)
- `@Override` en cada método
- Imports explícitos de `Product` y `RepositorioProductos` (están en otro paquete)

## Explicación paso a paso

1. La interfaz `RepositorioProductos` es Java PURO — cero dependencias de Spring, JPA, o cualquier framework. Esto es lo que hace que sea un PUERTO genuino.
2. `RepositorioProductosEnMemoria` está en `infraestructura/persistencia/` — un paquete SEPARADO que deja claro que esto es infraestructura, no dominio.
3. `@Repository` en la implementación (no en la interfaz) — Spring solo necesita anotaciones en beans concretos.
4. `@Override` en cada método — el compilador verifica que realmente estás implementando la interfaz, protegiéndote de errores de tipeo.
5. `ServicioVentas` NO SE MODIFICÓ — sigue recibiendo `RepositorioProductos` por constructor. Spring resuelve cuál implementación inyectar.
6. Tests NO SE MODIFICARON — `ServicioVentasTest` usa `@Mock RepositorioProductos repositorio`, que sigue funcionando igual con la interfaz.

## Ejemplo aplicado al proyecto

Este mismo patrón se aplica a CUALQUIER dependencia externa:
- **Base de datos**: `RepositorioProductos` (interfaz) → `RepositorioProductosPostgres` (nuevo adaptador)
- **Notificaciones**: `ServicioNotificaciones` (interfaz) → `ServicioNotificacionesEmail`, `ServicioNotificacionesSMS` (adaptadores)
- **Pagos**: `ProcesadorPago` (interfaz, Etapa 8) → `AdaptadorPagoExterno` (ya existente como adaptador)
- **Cache**: `CacheProductos` (interfaz) → `CacheProductosRedis`, `CacheProductosEnMemoria` (adaptadores)

## Mini reto de diseño

¿Qué cambiaría si `ServicioVentasTest` (Etapa 17 L2) usara `RepositorioProductosEnMemoria` real en vez de `@Mock`? ¿Seguiría siendo un test unitario?

<Reveal titulo="Respuesta">
Dejaría de ser un test UNITARIO (pasaría a ser de integración). `RepositorioProductosEnMemoria` es una clase REAL con estado (`ArrayList`) — si el test falla, no sabés si el bug está en `ServicioVentas` o en `RepositorioProductosEnMemoria`. El mock AISLA la lógica de `ServicioVentas` de cualquier bug en la implementación del repositorio — por eso los tests unitarios mockean los puertos de salida.
</Reveal>

## Transferencia

El patrón PUERTO (interfaz en dominio) + ADAPTADOR (implementación en infraestructura) es el MISMO que usan frameworks como Spring Data JPA: vos definís `interface ClienteRepository extends JpaRepository<Cliente, Long>` (tu PUERTO, en tu dominio), y Spring genera el ADAPTADOR automáticamente en runtime. La diferencia es que con Spring Data, el adaptador es generado por el framework — pero el principio arquitectónico es idéntico.

## Resumen

- Extraer una interfaz de una clase existente es el refactoring hexagonal mínimo y más poderoso.
- La interfaz (`RepositorioProductos`) es el PUERTO — define QUÉ, sin dependencias externas.
- La implementación (`RepositorioProductosEnMemoria`) es el ADAPTADOR — define CÓMO, con dependencias de infraestructura.
- El resto del código (`ServicioVentas`, `ProductoController`, tests) no se modificó — sigue funcionando igual.
- `./mvnw test`: 11 tests, 0 fallos — mismo comportamiento, mejor arquitectura.

## Modelo mental final

Extraer una interfaz es como separar el CONTROL REMOTO del televisor: el control (interfaz) define QUÉ botones existen (encender, volumen, canal), sin saber CÓMO funciona el televisor internamente. El televisor (adaptador) implementa esos botones con circuitos, pantalla, y parlantes. Si cambiás el televisor por otro modelo, el control remoto sigue siendo el mismo.

## Conexión

Este refactoring es la culminación de Dependency Injection (Etapa 8 L1: `ServicioVentas` recibe `RepositorioProductos` por constructor), interfaces (Etapa 2 L4: definir contratos), y SOLID (Etapa 7 L2: Dependency Inversion — depender de abstracciones, no de concreciones).

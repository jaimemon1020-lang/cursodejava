## Qué vas a aprender

Después de esta lección serás capaz de declarar un `enum` para representar un conjunto fijo de valores posibles (como el estado de un pedido), y usarlo en un `switch` moderno sabiendo que el compilador te protege de olvidarte un caso.

## Por qué necesitas aprenderlo

Un pedido pasa por estados concretos: pendiente, confirmado, cancelado — nunca "en el aire". Si modelás eso con un `String estado = "pendiente"`, nada impide escribir `"Pendiente"`, `"PENDIENTE "` con un espacio, o directamente `"volando"` por error de tipeo. Un enum elimina esa clase entera de bugs.

## Qué debes saber antes

Switch moderno (Etapa 1) y las clases `Order`/`OrderItem` (Etapa 3-4).

## Problema

`Order` necesita un estado: pendiente, confirmado o cancelado. Representarlo como `String` o `int` deja la puerta abierta a valores inválidos ("cnofirmado", con un typo, o el número `7` sin ningún significado) que el compilador jamás va a detectar.

## Modelo mental

```
enum EstadoPedido {
    PENDIENTE,
    CONFIRMADO,
    CANCELADO
}

Solo estos tres valores existen. Ningún otro compila.
```

## Explicación sencilla

Un `enum` declara una lista cerrada de valores con nombre. `EstadoPedido.PENDIENTE` es un valor real y único — no un texto que alguien podría escribir mal.

## Explicación técnica

Un `enum` es, en el fondo, una clase especial cuyas únicas instancias posibles son las que vos declarás explícitamente (`PENDIENTE`, `CONFIRMADO`, `CANCELADO`). El compilador garantiza que una variable de tipo `EstadoPedido` solo puede contener uno de esos tres valores — nunca `null` por accidente de tipeo, nunca un cuarto valor inventado.

Comparar enums se hace con `==` de forma segura (a diferencia de `String`, donde `==` compara referencias y `.equals()` es lo correcto) — cada constante de enum es un único objeto en toda la aplicación.

En un `switch` moderno sobre un enum, el compilador puede advertirte si te falta cubrir algún valor posible (exhaustividad) — una ventaja real sobre comparar `String`s con `if`/`else`.

## Cómo funciona

1. `EstadoPedido` declara sus tres únicos valores posibles.
2. `Order` guarda un `EstadoPedido estado`, inicializado en `PENDIENTE` en el constructor.
3. `confirmar()` y `cancelar()` validan la transición ANTES de cambiar el estado — protegiendo reglas como "un pedido confirmado no puede cancelarse".
4. El `switch` moderno sobre `pedido.getEstado()` cubre los tres casos posibles, sin necesitar un `default` (porque ya cubriste TODOS los valores que `EstadoPedido` puede tomar).

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-04/EstadoPedido.java`

```java
public enum EstadoPedido {
    PENDIENTE,
    CONFIRMADO,
    CANCELADO
}
```

**Fragmento relevante de `proyecto/etapa-04/Order.java`:**

```java
private EstadoPedido estado;

Order(Cliente cliente) {
    this.cliente = cliente;
    this.items = new ArrayList<>();
    this.estado = EstadoPedido.PENDIENTE;
}

public void confirmar() {
    if (this.estado != EstadoPedido.PENDIENTE) {
        throw new IllegalStateException("Solo un pedido pendiente puede confirmarse");
    }
    this.estado = EstadoPedido.CONFIRMADO;
}

public void cancelar() {
    if (this.estado == EstadoPedido.CONFIRMADO) {
        throw new IllegalStateException("Un pedido confirmado no puede cancelarse");
    }
    this.estado = EstadoPedido.CANCELADO;
}
```

**Archivo:** `proyecto/etapa-04/EnumDemo.java`

```java
public class EnumDemo {
    public static void main(String[] args) {
        Cliente cliente = new Cliente("Juan Perez", "juan@ejemplo.com", "Av. Siempre Viva 123");
        Order pedido = new Order(cliente);

        Product teclado = new Product("Teclado mecanico", 25000.0, 15);
        pedido.agregarItem(new OrderItem(teclado, 1));

        System.out.println("Estado inicial: " + pedido.getEstado());

        pedido.confirmar();
        System.out.println("Estado despues de confirmar: " + pedido.getEstado());

        String mensaje = switch (pedido.getEstado()) {
            case PENDIENTE -> "Esperando confirmacion";
            case CONFIRMADO -> "Listo para preparar envio";
            case CANCELADO -> "El pedido fue cancelado";
        };
        System.out.println("Mensaje: " + mensaje);
    }
}
```

## Predice

Un pedido recién creado, antes de llamar `confirmar()`. ¿Qué imprime `pedido.getEstado()`?

<Reveal titulo="Resultado real (compilado y ejecutado)">
```
Estado inicial: PENDIENTE
Estado despues de confirmar: CONFIRMADO
Mensaje: Listo para preparar envio
```
</Reveal>

## Explicación paso a paso

- El constructor de `Order` fija `estado = EstadoPedido.PENDIENTE` — nunca queda en un estado indefinido.
- `confirmar()` verifica `this.estado != EstadoPedido.PENDIENTE` — como recién se creó, la condición es falsa, y el cambio a `CONFIRMADO` procede.
- El `switch` moderno no necesita `EstadoPedido.CONFIRMADO` en cada `case` — dentro de un switch sobre un enum, alcanza con el nombre de la constante (`CONFIRMADO`, no `EstadoPedido.CONFIRMADO`).

## Mini reto de debugging

```java
EstadoPedido estado = EstadoPedido.PENDIENTE;
if (estado == "PENDIENTE") {
    System.out.println("Es pendiente");
}
```

¿Por qué este código ni siquiera compila?

<Reveal titulo="Diagnóstico">
`EstadoPedido.PENDIENTE` es un valor de tipo `EstadoPedido`, no un `String` — comparar un `EstadoPedido` contra el texto `"PENDIENTE"` con `==` es comparar tipos incompatibles, y el compilador lo rechaza directamente. La comparación correcta es `estado == EstadoPedido.PENDIENTE`.
</Reveal>

## Resumen

- `enum` declara un conjunto cerrado y con nombre de valores posibles.
- El compilador impide crear o asignar un valor que no esté declarado.
- Comparar enums con `==` es seguro, a diferencia de comparar `String` con `==`.
- Un switch moderno sobre un enum puede detectar en compilación si te falta cubrir algún valor.
- Las transiciones válidas (qué estado puede pasar a cuál) siguen siendo responsabilidad de tu propio código, no del enum.

## Modelo mental final

Cuando pienses en un enum, imaginá un semáforo: rojo, amarillo, verde — nada más existe. No hay "rojo oscuro" ni "verde con typo"; el conjunto de posibilidades está cerrado por diseño.

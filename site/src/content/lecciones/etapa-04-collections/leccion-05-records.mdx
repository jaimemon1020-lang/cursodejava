## Qué vas a aprender

Después de esta lección serás capaz de modelar un Value Object inmutable (como un monto de dinero) con `record`, entendiendo qué te genera automáticamente el compilador y por qué eso importa para comparar valores correctamente.

## Por qué necesitas aprenderlo

La Etapa 3 mencionó conceptualmente los Value Objects: un objeto definido por su VALOR, no por una identidad — dos montos de "$100" deberían ser intercambiables. Escribir eso a mano con una clase común requiere constructor, getters, `equals()`, `hashCode()` y `toString()` — mucho código repetitivo para una idea simple.

## Qué debes saber antes

Encapsulamiento e inmutabilidad (Etapa 2) y el concepto de Value Object (Etapa 3).

## Recupera lo aprendido

¿Por qué un enum es más seguro que un String para representar un conjunto fijo de valores?

<Reveal titulo="Respuesta">

</Reveal>

## Problema

Necesitás representar un monto de dinero como un Value Object real: inmutable, comparable por valor (no por identidad), y sin la ceremonia de escribir manualmente constructor + getters + `equals` + `hashCode` + `toString` para algo tan simple como "un número con una regla de validación".

## Modelo mental

```
record Dinero(double monto) {
    // el compilador genera automáticamente:
    //   - constructor Dinero(double monto)
    //   - accessor monto()  (no getMonto())
    //   - equals() por VALOR
    //   - hashCode() consistente con equals()
    //   - toString() legible
}
```

## Explicación sencilla

Un `record` es una forma compacta de decir "esta clase es solamente los datos que declaro, inmutables, y quiero que Java me genere todo el código repetitivo de siempre". Vos solo escribís el nombre y los campos; el resto lo genera el compilador.

## Explicación técnica

`record Dinero(double monto) { ... }` declara un tipo inmutable con un único campo `monto`. El compilador genera automáticamente:

- Un **constructor canónico** `Dinero(double monto)`.
- Un **accessor** por campo, con el mismo nombre del campo (`monto()`, no `getMonto()` — distinto a la convención de getters que usaste en clases comunes).
- `equals()` que compara TODOS los campos por valor — dos `Dinero` con el mismo `monto` son `equals`, aunque sean objetos distintos en memoria.
- `hashCode()` consistente con ese `equals()`.
- `toString()` legible, con el formato `Dinero[monto=25000.0]`.

Todos los campos de un record son implícitamente `private` y `final` — no hay forma de reasignarlos después de creado, ni siquiera desde dentro de la propia clase (salvo en el constructor). Un **constructor compacto** (`public Dinero { ... }`, sin paréntesis ni reasignación explícita) te permite validar antes de que el objeto se termine de construir — igual que ya validabas en el constructor de `Product`.

## Cómo funciona

1. `new Dinero(25000.0)` pasa por el constructor compacto, que valida `monto < 0` antes de aceptar el valor.
2. `precioTeclado.sumar(precioMouse)` no modifica ningún objeto existente — crea y devuelve un `Dinero` NUEVO, porque los records son inmutables por diseño.
3. `total.monto()` accede al valor con el accessor generado, sin escribir vos mismo ningún getter.
4. `precioTeclado.equals(otroPrecioTeclado)` compara por VALOR (da `true` si el monto es igual), mientras que `==` compara identidad de objeto (da `false`, porque son dos instancias distintas).

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-04/Dinero.java`

```java
public record Dinero(double monto) {

    public Dinero {
        if (monto < 0) {
            throw new IllegalArgumentException("El monto no puede ser negativo");
        }
    }

    public Dinero sumar(Dinero otro) {
        return new Dinero(this.monto + otro.monto);
    }

    public Dinero aplicarDescuento(double porcentaje) {
        return new Dinero(this.monto - (this.monto * porcentaje));
    }
}
```

**Archivo:** `proyecto/etapa-04/RecordsDemo.java`

```java
public class RecordsDemo {
    public static void main(String[] args) {
        Dinero precioTeclado = new Dinero(25000.0);
        Dinero precioMouse = new Dinero(8000.0);

        Dinero total = precioTeclado.sumar(precioMouse);
        System.out.println("Total: " + total);
        System.out.println("Monto accesible: " + total.monto());

        Dinero totalConDescuento = total.aplicarDescuento(0.10);
        System.out.println("Total con descuento: " + totalConDescuento);

        Dinero otroPrecioTeclado = new Dinero(25000.0);
        System.out.println("precioTeclado equals otroPrecioTeclado: " + precioTeclado.equals(otroPrecioTeclado));
        System.out.println("precioTeclado == otroPrecioTeclado: " + (precioTeclado == otroPrecioTeclado));
    }
}
```

## Predice

`precioTeclado` y `otroPrecioTeclado` son dos instancias DISTINTAS, ambas con `monto = 25000.0`. ¿`equals()` va a dar `true` o `false`? ¿Y `==`?

<Reveal titulo="Resultado real (compilado y ejecutado)">
```
Total: Dinero[monto=33000.0]
Monto accesible: 33000.0
Total con descuento: Dinero[monto=29700.0]
precioTeclado equals otroPrecioTeclado: true
precioTeclado == otroPrecioTeclado: false
```
</Reveal>

## Explicación paso a paso

- `System.out.println(total)` usa el `toString()` generado automáticamente: `Dinero[monto=33000.0]` — nunca escribiste ese método.
- `equals()` da `true`: compara el VALOR de `monto` en ambos objetos, y `25000.0 == 25000.0`.
- `==` da `false`: son dos objetos distintos en memoria, aunque tengan el mismo contenido — exactamente la diferencia entre identidad y valor que motiva usar `equals()` para records (y objetos en general).
- Vas a profundizar el contrato `equals()`/`hashCode()` de forma completa (incluso para clases que NO son records) en la Etapa 6.

## Ejercicio guiado

Creá un record `Coordenada(double latitud, double longitud)` con un constructor compacto que valide que la latitud esté entre -90 y 90.

<Reveal titulo="Pista 1">
Un record puede tener más de un campo — se listan separados por coma, igual que parámetros de un método.
</Reveal>

<Reveal titulo="Pista 2">
El constructor compacto es `public Coordenada { ... }`, sin paréntesis, igual que viste en `Dinero`.
</Reveal>

<Reveal titulo="Solución">
```java
public record Coordenada(double latitud, double longitud) {
    public Coordenada {
        if (latitud < -90 || latitud > 90) {
            throw new IllegalArgumentException("Latitud invalida");
        }
    }
}
```
</Reveal>

## Mini reto de debugging

```java
public record Dinero(double monto) {
    public Dinero {
        monto = Math.abs(monto);
    }
}
```

Este constructor compacto compila. ¿Qué hace exactamente esa línea, sabiendo que el campo `monto` es final?

<Reveal titulo="Diagnóstico">
Dentro del constructor compacto (antes de que termine implícitamente asignando los campos), reasignar el parámetro `monto` SÍ está permitido — es una excepción especial del constructor compacto para poder "normalizar" el valor antes de que se guarde definitivamente. Una vez que el objeto termina de construirse, ese `monto` ya es inmutable como siempre.
</Reveal>

## Resumen

- `record` genera automáticamente constructor, accessors, `equals`, `hashCode` y `toString`.
- Los campos de un record son implícitamente `private final` — inmutables tras la construcción.
- Un constructor compacto permite validar (o normalizar) antes de terminar de construir el objeto.
- `equals()` en un record compara por valor; `==` sigue comparando identidad de objeto.

## Modelo mental final

Cuando pienses en un record, imaginá un billete: dos billetes de $100 son "el mismo valor" aunque sean físicamente distintos — no importa CUÁL billete tenés, importa CUÁNTO vale.

## Qué vas a aprender

Después de esta lección entenderás cómo Testcontainers usa Docker para levantar una base de datos REAL (PostgreSQL, MySQL, etc.) exclusivamente durante la ejecución de los tests, destruyéndola al terminar — permitiendo testear `ClienteRepository` contra PostgreSQL real en vez de H2, sin instalar ni configurar nada manualmente.

## Por qué necesitas aprenderlo

H2 en memoria (Etapa 15-17) es una base de datos Java con modo compatibilidad — PERO no es PostgreSQL. Diferencias REALES de dialecto SQL, tipos de datos, funciones, y comportamientos pueden esconder bugs que solo aparecen en producción con la base de datos verdadera. Testcontainers elimina ese riesgo levantando la base de datos REAL en un contenedor Docker efímero.

## Qué debes saber antes

`JpaRepository`, `@DataJpaTest`, H2 (Etapa 15). Docker instalado y corriendo en la máquina (requisito del entorno, no del código).

## Recupera lo aprendido

¿Cuál es la diferencia clave entre un test UNITARIO con Mockito y un test de INTEGRACIÓN con `@DataJpaTest` + H2?

<Reveal titulo="Respuesta">
El test unitario (Mockito) aísla UNA clase de sus dependencias con mocks — rápido, pero no prueba interacciones reales. El test de integración (`@DataJpaTest` + H2) levanta un contexto de Spring recortado y prueba el repositorio contra una base de datos REAL — más lento, pero valida que las consultas y las entidades realmente funcionan juntas.
</Reveal>

## Problema

`ClienteRepositoryTest` (con `@DataJpaTest` + H2) prueba que `findByEmail` funciona — pero contra H2. Si tu aplicación en producción usa PostgreSQL, y tu `@Query` contiene una función SQL que H2 no soporta (como `string_agg` o `jsonb_exists`), el test pasa con H2 pero falla en producción. Necesitás testear contra la MISMA base de datos que usás en producción, sin instalar PostgreSQL manualmente en cada máquina de desarrollo.

## Modelo mental

```
H2 en tests:          "Probemos en un simulador de avión."
                       Barato, fácil — pero no vuela igual.

Testcontainers:       "Probemos en un avión REAL, pero solo
                       en la pista de pruebas."
                       Es la base de datos de producción
                       real, viviendo lo que dura el test.
```

## Explicación sencilla

Testcontainers es una biblioteca Java que habla con Docker para CREAR contenedores de bases de datos PostgreSQL (o MySQL, Redis, Kafka...) AUTOMÁTICAMENTE cuando tus tests arrancan, y DESTRUIRLOS cuando terminan. No necesitás instalar nada — solo tener Docker corriendo. Testcontainers descarga la imagen, levanta el contenedor, ejecuta Flyway (tus migraciones) sobre esa base real, y el test corre contra PostgreSQL genuino.

## Explicación técnica

`testcontainers` provee `PostgreSQLContainer` (un `GenericContainer` especializado) que, al ser declarado como `static` en una clase de test, se comparte entre todos los tests de esa clase — evitando crear y destruir un contenedor por cada método. `@DynamicPropertySource` (o `@ServiceConnection` en Spring Boot 3.1+) inyecta dinámicamente `spring.datasource.url`, `username` y `password` apuntando al contenedor efímero, SOBREESCRIBIENDO `application.properties` SOLO para ese test.

El contenedor se crea UNA vez (gracias a `static`), Flyway aplica las migraciones automáticamente (porque `spring.flyway.enabled=true`), y los tests de `ClienteRepository` corren contra PostgreSQL real — detectando cualquier incompatibilidad de dialecto, tipo de dato, o sintaxis SQL que H2 hubiera ocultado.

## Cómo funciona

1. `@Testcontainers` + `@SpringBootTest` activan la integración: antes de que Spring arranque el contexto, Testcontainers verifica que Docker está corriendo y descarga (o usa cache local) la imagen de PostgreSQL.
2. `static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16")` declara el contenedor — `static` asegura que solo se cree UNO para toda la clase de test.
3. `@DynamicPropertySource` (método static) registra las propiedades de conexión (`spring.datasource.url=jdbc:postgresql://localhost:...` con el puerto aleatorio que Docker asignó) en el contexto de Spring ANTES de que se cree el `DataSource`.
4. Spring Boot arranca con `DataSource` apuntando al PostgreSQL del contenedor — Flyway detecta la base y aplica `V1__crear_clientes.sql`, creando la tabla `clientes` real.
5. `clienteRepository.save(...)` y `clienteRepository.findByEmail(...)` se ejecutan contra PostgreSQL genuino.
6. Al terminar la suite de tests, Testcontainers destruye el contenedor automáticamente (configurable con `.withReuse(true)` para desarrollo local).

## Ejemplo mínimo

**Archivo:** `proyecto-maven/pom.xml` (dependencia a agregar)

```xml
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>testcontainers</artifactId>
    <version>1.20.4</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>postgresql</artifactId>
    <version>1.20.4</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <scope>test</scope>
</dependency>
```

**Archivo:** `proyecto-maven/src/test/java/com/curso/catalogo/ClienteRepositoryTestcontainersTest.java`

```java
package com.curso.catalogo;

import java.util.Optional;

import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.context.DynamicPropertyRegistry;
import org.springframework.test.context.DynamicPropertySource;
import org.testcontainers.containers.PostgreSQLContainer;
import org.testcontainers.junit.jupiter.Container;
import org.testcontainers.junit.jupiter.Testcontainers;

import static org.junit.jupiter.api.Assertions.assertEquals;
import static org.junit.jupiter.api.Assertions.assertTrue;

@Testcontainers
@SpringBootTest
class ClienteRepositoryTestcontainersTest {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16");

    @DynamicPropertySource
    static void configurarPropiedades(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired
    private ClienteRepository clienteRepository;

    @Test
    @DisplayName("Guardar y recuperar un cliente en PostgreSQL real")
    void guardarYRecuperarClienteEnPostgresReal() {
        Cliente cliente = new Cliente("Maria Lopez", "maria@test.com", "Calle 123, Buenos Aires");
        Cliente guardado = clienteRepository.save(cliente);

        Optional<Cliente> encontrado = clienteRepository.findByEmail("maria@test.com");

        assertTrue(encontrado.isPresent());
        assertEquals("Maria Lopez", encontrado.get().getNombre());
    }
}
```

## Predice

Este test levanta un PostgreSQL REAL en Docker, Flyway aplica las migraciones, y corre `clienteRepository.findByEmail(...)` contra PostgreSQL. ¿Qué pasa si `@Query("SELECT c FROM Cliente c WHERE c.direccionEnvio LIKE %:ciudad%")` contiene una función que PostgreSQL no entiende?

<Reveal titulo="Resultado">
El test FALLA con un error de PostgreSQL — detectando el bug ANTES de llegar a producción, algo que H2 no habría detectado porque H2 habría interpretado la consulta de forma distinta (o simplemente la habría ignorado sin error).
</Reveal>

## Explicación paso a paso

- `static PostgreSQLContainer<?> postgres` inicializa el contenedor UNA vez, antes de cualquier método de test — el puerto es aleatorio, asignado por Docker.
- `@DynamicPropertySource static void configurarPropiedades(...)` es llamado por Spring ANTES de crear beans — inyecta `spring.datasource.url=jdbc:postgresql://localhost:54321/test` (el puerto real asignado) en el `Environment`.
- `@SpringBootTest` levanta el contexto COMPLETO de la aplicación, incluyendo Flyway (que aplica V1__crear_clientes.sql sobre PostgreSQL) y `ClienteRepository`.
- `@Autowired ClienteRepository clienteRepository` obtiene el repositorio real, conectado al PostgreSQL del contenedor, con las migraciones ya aplicadas.
- `clienteRepository.save(cliente)` ejecuta `INSERT INTO clientes ...` en PostgreSQL real — no en H2.
- Al terminar la clase de test, Testcontainers detiene y elimina el contenedor automáticamente (a menos que se configure `withReuse(true)`).

## Ejemplo aplicado al proyecto

En un proyecto real con PostgreSQL en producción, reemplazarías `@DataJpaTest` (que usa H2) por `@SpringBootTest` + Testcontainers para los tests de integración del repositorio — detectando diferencias de dialecto SQL, tipos de datos (como `JSONB` vs `TEXT`), y comportamientos específicos de PostgreSQL que H2 emula pero no replica exactamente.

## Ejercicio trabajado

Configurar `postgres.withReuse(true)` para desarrollo local — el contenedor NO se destruye entre ejecuciones de tests, acelerando iteraciones de TDD:

```java
static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16")
        .withReuse(true);
```

Más `testcontainers.reuse.enable=true` en `~/.testcontainers.properties` — sin esto, `withReuse(true)` es ignorado (es una protección para CI).

## Ejercicio guiado

Investigá cómo usar `@ServiceConnection` (Spring Boot 3.1+) en vez de `@DynamicPropertySource` — simplifica la configuración a una sola línea, sin el método static manual.

<Reveal titulo="Solución">
```java
@Testcontainers
@SpringBootTest
class ClienteRepositoryTestcontainersTest {

    @Container
    @ServiceConnection
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16");

    @Autowired
    private ClienteRepository clienteRepository;

    // tests...
}
```
`@ServiceConnection` detecta automáticamente que es un contenedor de PostgreSQL y registra `spring.datasource.url`, `username`, `password` — sin necesidad del método `@DynamicPropertySource` manual.
</Reveal>

## Mini reto de debugging

```java
@Testcontainers
@SpringBootTest
class ClienteRepositoryTest {
    @Container
    PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16");
    // sin static, sin @DynamicPropertySource
}
```

El test empieza, pero Spring Boot se conecta a H2 (desde `application.properties`) en vez de a PostgreSQL. ¿Por qué?

<Reveal titulo="Diagnóstico">
Dos errores: (1) `postgres` NO es `static` — cada método de test crearía y destruiría un contenedor nuevo. (2) Falta `@DynamicPropertySource` (o `@ServiceConnection`) — sin eso, Spring Boot lee `application.properties` y usa H2, ignorando completamente el contenedor de PostgreSQL que Testcontainers levantó.
</Reveal>

## Resumen

- Testcontainers usa Docker para levantar bases de datos reales (PostgreSQL, MySQL) exclusivamente durante los tests, destruyéndolas al terminar.
- `static PostgreSQLContainer` + `@DynamicPropertySource` inyecta la conexión al contenedor en el contexto de Spring, reemplazando `application.properties` solo para ese test.
- Los tests de integración contra la base de datos REAL detectan diferencias de dialecto SQL que H2 oculta — bugs que solo aparecerían en producción.
- Requiere Docker instalado y corriendo — sin Docker, los tests con Testcontainers no pueden ejecutarse.

## Modelo mental final

Pensá en H2 como un simulador de vuelo: excelente para practicar, barato, instantáneo — pero no vuela igual que un avión real. Testcontainers es el avión real, en una pista de prueba: tarda más en despegar, pero lo que probás ahí es EXACTAMENTE lo que va a volar en producción.

## Conexión

Así como `@Mock` aísla `ServicioVentas` de `RepositorioProductos` (Etapa 17 L2), Testcontainers aísla tus tests de "la base de datos instalada en esta máquina" — el contenedor ES la base de datos, y es idéntico sin importar en qué máquina corra (tu notebook, la de un compañero, el CI).

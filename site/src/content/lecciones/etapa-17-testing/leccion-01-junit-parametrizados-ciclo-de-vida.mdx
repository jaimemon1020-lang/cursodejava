## Qué vas a aprender

Después de esta lección serás capaz de compartir configuración común entre tests con `@BeforeEach`, eliminar duplicación probando MÚLTIPLES casos con un solo método usando `@ParameterizedTest`, y hacer que los resultados de test sean más legibles con `@DisplayName`.

## Por qué necesitas aprenderlo

`ProductTest` (Etapa 9 Lección 2) repetía `new Product("Teclado mecanico", 25000.0, 15)` en CADA método de test — y probar "vender 1 unidad dejar stock en 14", "vender 5 dejar en 10", "vender 15 dejar en 0" como métodos SEPARADOS sería triplicar (o multiplicar N veces) el mismo código con solo los números cambiando. JUnit tiene herramientas específicas para ambos problemas.

## Qué debes saber antes

`@Test`, `assertEquals`, `assertThrows` (Etapa 9 Lección 2) — esta lección extiende ese mismo archivo de test.

## Problema

Cada método de `ProductTest` empieza creando el MISMO `Product` de prueba — repetido en cada test. Y probar que `venderUnidades` funciona correctamente con distintas cantidades (1, 5, 15 unidades) requeriría escribir un método `@Test` CASI IDÉNTICO por cada caso, cambiando solo los números — puro código repetitivo que además complica agregar un caso nuevo (hay que copiar y pegar un método entero).

## Modelo mental

```
@BeforeEach:        se ejecuta ANTES de cada @Test — configuración
                     compartida, sin repetirla en cada método.

@ParameterizedTest:  UN método de test, ejecutado VARIAS veces,
                     una por cada conjunto de datos de entrada.

@DisplayName:        un nombre legible para humanos, en vez del
                     nombre técnico del método Java.
```

## Explicación sencilla

`@BeforeEach` marca un método que JUnit ejecuta ANTES de CADA `@Test` de la clase — el lugar perfecto para preparar datos comunes, evitando repetir la misma configuración en cada método. `@ParameterizedTest` convierte un ÚNICO método de test en una PLANTILLA que JUnit ejecuta múltiples veces, una por cada fila de datos que le proporciones — cada ejecución es un test INDEPENDIENTE, con su propio resultado de pasa/falla.

## Explicación técnica

`@BeforeEach` sobre `configurar()` hace que JUnit cree una instancia NUEVA de `ProductTest` y ejecute `configurar()` antes de CADA `@Test` — el campo `teclado` queda listo, fresco, para cada test individual, sin que un test pueda "contaminar" el estado que otro test usa (cada `@Test` obtiene su propia instancia limpia de la clase).

`@ParameterizedTest` reemplaza `@Test` cuando el método recibe PARÁMETROS. `@CsvSource({"1, 14", "5, 10", "15, 0"})` proporciona los datos como texto separado por comas — cada línea se convierte en una ejecución del test, con los valores convertidos automáticamente al tipo del parámetro (`int cantidad, int stockEsperado`). `@ValueSource(doubles = {-1.0, -100.0, -0.01})` es más simple: una lista de valores del MISMO tipo, cada uno ejecuta el test una vez.

`name = "vender {0} unidades deja stock en {1}"` en `@ParameterizedTest` personaliza el nombre que aparece en los resultados — `{0}` y `{1}` se reemplazan por los parámetros de esa ejecución específica, haciendo mucho más legible CUÁL caso falló si algo sale mal.

## Cómo funciona

1. `@BeforeEach void configurar()` crea un `Product` nuevo antes de cada test — `venderUnidadesReduceElStock` y `venderMasDeLoDisponibleLanzaExcepcion` ya no necesitan crear su propio `Product`.
2. `venderDistintasCantidadesReduceElStockCorrectamente(int cantidad, int stockEsperado)` es UN método, pero JUnit lo ejecuta TRES veces (una por línea de `@CsvSource`), con `cantidad`/`stockEsperado` distintos en cada ejecución.
3. `precioNegativoLanzaExcepcionAlCrear(double precioInvalido)` prueba TRES valores negativos distintos con un solo método — confirmando que `Product` rechaza CUALQUIER precio negativo, no solo un caso puntual.
4. `@DisplayName` en cada método da un nombre legible en el reporte de test, sin depender de que el nombre del método Java (`venderDistintasCantidadesReduceElStockCorrectamente`) sea autoexplicativo por sí solo.

## Ejemplo mínimo

**Archivo:** `proyecto-maven/src/test/java/com/curso/catalogo/ProductTest.java`

```java
package com.curso.catalogo;

import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.CsvSource;
import org.junit.jupiter.params.provider.ValueSource;

import static org.junit.jupiter.api.Assertions.assertEquals;
import static org.junit.jupiter.api.Assertions.assertThrows;

class ProductTest {

    private Product teclado;

    @BeforeEach
    void configurar() {
        teclado = new Product("Teclado mecanico", 25000.0, 15);
    }

    @Test
    @DisplayName("Vender unidades reduce el stock")
    void venderUnidadesReduceElStock() {
        teclado.venderUnidades(5);
        assertEquals(10, teclado.getStock());
    }

    @Test
    @DisplayName("Vender mas de lo disponible lanza StockInsuficienteException")
    void venderMasDeLoDisponibleLanzaExcepcion() {
        assertThrows(StockInsuficienteException.class, () -> teclado.venderUnidades(100));
    }

    @ParameterizedTest(name = "vender {0} unidades deja stock en {1}")
    @CsvSource({
            "1, 14",
            "5, 10",
            "15, 0"
    })
    @DisplayName("Vender distintas cantidades reduce el stock correctamente")
    void venderDistintasCantidadesReduceElStockCorrectamente(int cantidad, int stockEsperado) {
        teclado.venderUnidades(cantidad);
        assertEquals(stockEsperado, teclado.getStock());
    }

    @ParameterizedTest
    @ValueSource(doubles = {-1.0, -100.0, -0.01})
    @DisplayName("Precio negativo lanza excepcion al crear el producto")
    void precioNegativoLanzaExcepcionAlCrear(double precioInvalido) {
        assertThrows(IllegalArgumentException.class, () -> new Product("Producto invalido", precioInvalido, 10));
    }
}
```

## Predice

`venderDistintasCantidadesReduceElStockCorrectamente` tiene 3 filas en `@CsvSource`, y `precioNegativoLanzaExcepcionAlCrear` tiene 3 valores en `@ValueSource` — sumados a los 2 tests originales (`@Test`). ¿Cuántos tests EN TOTAL esperás que reporte Maven?

<Reveal titulo="Resultado real (ejecutado con Maven)">
```
$ ./mvnw test -Dtest=ProductTest
...
[INFO] Tests run: 8, Failures: 0, Errors: 0, Skipped: 0
[INFO] BUILD SUCCESS
```
</Reveal>

## Explicación paso a paso

- "Tests run: 8" = 2 (`@Test` normales) + 3 (`@CsvSource`, una ejecución por fila) + 3 (`@ValueSource`, una ejecución por valor) — CADA fila/valor cuenta como un test INDEPENDIENTE, no como parte de un solo test grande.
- "Failures: 0" confirma que las 3 cantidades distintas (1, 5, 15) TODAS dejaron el stock exacto esperado, y los 3 precios negativos TODOS lanzaron `IllegalArgumentException` correctamente.
- Un solo método de test (`venderDistintasCantidadesReduceElStockCorrectamente`) generó TRES resultados en el reporte — sin que hayas escrito tres métodos separados.
- `@BeforeEach` se ejecutó 8 VECES en total (una por cada test, incluyendo cada ejecución parametrizada) — cada test parte de un `Product` fresco con stock 15, sin que un test afecte al siguiente.

## Ejemplo aplicado al proyecto

Con tests parametrizados, agregar un caso nuevo de "vender X unidades, esperar stock Y" es una LÍNEA nueva en `@CsvSource`, no un método nuevo completo — el mismo espíritu de Clean Code (Etapa 7) aplicado a los tests: menos repetición, más fácil de extender.

<Reveal titulo="Solución">
```java

@DisplayName("Vender mas del stock disponible siempre lanza excepcion")

    assertThrows(StockInsuficienteException.class, () -> teclado.venderUnidades(cantidadExcesiva));
}
```

## Mini reto de debugging

```java
class ProductTest {
    private Product teclado = new Product("Teclado mecanico", 25000.0, 15);

    @Test
    void primerTest() {
        teclado.venderUnidades(5);
        assertEquals(10, teclado.getStock());
    }

    @Test
    void segundoTest() {
        assertEquals(15, teclado.getStock());
    }
}
```

Sin `@BeforeEach` (el campo se inicializa directamente en la declaración), ¿es seguro asumir que `segundoTest` siempre va a ver stock 15?

<Reveal titulo="Diagnóstico">
Depende de un detalle importante de JUnit 5: por defecto, JUnit crea una instancia NUEVA de la clase de test para CADA método `@Test` — así que, aunque `teclado` se inicialice directamente en la declaración del campo (sin `@BeforeEach`), cada test SÍ obtiene su propia instancia fresca, con stock 15. En este caso específico el resultado sería el mismo, pero usar `@BeforeEach` explícitamente comunica la INTENCIÓN con claridad, y es necesario cuando la configuración requiere lógica más allá de una simple inicialización de campo (como leer datos de un archivo, o configurar un mock).
</Reveal>

## Resumen

- `@BeforeEach` comparte configuración común, ejecutándose antes de cada test con una instancia fresca de la clase.
- `@ParameterizedTest` con `@CsvSource`/`@ValueSource` ejecuta el MISMO método varias veces con datos distintos, cada ejecución contando como un test independiente.
- `@DisplayName` mejora la legibilidad del reporte de test, sin depender del nombre técnico del método.
- Menos repetición en los tests significa menos trabajo para agregar casos nuevos, y menos riesgo de errores de copy-paste.

## Modelo mental final

Pensá en `@BeforeEach` como poner la mesa antes de cada comensal (cada test), siempre igual, sin que un comensal tenga que preocuparse por lo que dejó el anterior — y en `@ParameterizedTest` como una receta única que cocinás varias veces con ingredientes ligeramente distintos cada vez, en vez de escribir una receta nueva por cada variación.

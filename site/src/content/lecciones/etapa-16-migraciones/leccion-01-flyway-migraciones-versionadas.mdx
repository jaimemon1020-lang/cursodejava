## Qué vas a aprender

Después de esta lección serás capaz de versionar el esquema de tu base de datos con Flyway, reemplazando `spring.jpa.hibernate.ddl-auto=update` (cómodo para desarrollo, riesgoso en producción) por migraciones SQL explícitas y ordenadas.

## Por qué necesitas aprenderlo

`spring.jpa.hibernate.ddl-auto=update` (Etapa 15) dejó que Hibernate GENERARA automáticamente la tabla `clientes` a partir de las anotaciones de `Cliente`. Cómodo para desarrollo — pero en un equipo real, con una base de datos de PRODUCCIÓN con datos reales, dejar que un framework decida automáticamente cómo modificar el esquema es peligroso: no hay control de versiones, no hay forma de saber QUÉ cambió entre un deploy y el siguiente, ni de revertir un cambio problemático.

## Qué debes saber antes

`@Entity`, `JpaRepository`, `spring.jpa.hibernate.ddl-auto` (Etapa 15 Lección 2) — Flyway reemplaza esa auto-generación.

## Recupera lo aprendido

¿Cuándo conviene usar `@Query` en vez de un método de consulta derivado (`findByXxx`) en un `JpaRepository`?

<Reveal titulo="Respuesta">
Cuando la consulta no se puede expresar claramente con la convención de nombres — campos que no existen literalmente en la entidad, lógica compleja, o JOINs — como buscarPorCiudad, que necesitaba buscar dentro del campo direccionEnvio.
</Reveal>

## Problema

`ddl-auto=update` funciona bien mientras estás APRENDIENDO o prototipando — pero en un equipo real, necesitás que el esquema de la base de datos esté BAJO CONTROL DE VERSIONES, igual que el código: cada cambio debe quedar registrado, aplicado en orden, y de forma REPRODUCIBLE en cualquier ambiente (tu máquina, la de un compañero, el servidor de producción). Sin eso, "¿qué versión del esquema tiene esta base de datos?" es una pregunta sin respuesta confiable.

## Modelo mental

```
ddl-auto=update:  Hibernate DECIDE automáticamente cómo modificar
                   el esquema, comparando las entidades con la
                   base actual — cómodo, pero sin historial ni control.

Flyway:            VOS escribís migraciones SQL numeradas
                   (V1__algo.sql, V2__otra_cosa.sql), y Flyway
                   las aplica EN ORDEN, una sola vez cada una,
                   dejando un registro de qué se aplicó y cuándo.
```

## Explicación sencilla

Flyway es una herramienta de migración de base de datos: en vez de dejar que el ORM adivine cómo debería verse el esquema, VOS escribís archivos SQL numerados (`V1__crear_clientes.sql`, `V2__agregar_columna.sql`, etc.) que describen CADA cambio, en el orden en que deben aplicarse. Flyway lleva un registro (una tabla especial, `flyway_schema_history`) de qué migraciones ya se ejecutaron, así que cada vez que la aplicación arranca, solo aplica las migraciones NUEVAS que todavía no corrieron.

## Explicación técnica

Los archivos de migración viven en `src/main/resources/db/migration/`, con un nombre que sigue la convención `V<versión>__<descripción>.sql` (doble guión bajo entre versión y descripción). Flyway los ordena por número de versión y los ejecuta EN ORDEN, UNA SOLA VEZ cada uno — una vez aplicada, una migración nunca se vuelve a ejecutar (y modificarla después de aplicada en cualquier ambiente es un error grave: Flyway detecta el cambio de contenido con un checksum y falla).

Con Flyway gestionando el esquema, `spring.jpa.hibernate.ddl-auto` cambia de `update` a `validate`: Hibernate ya NO crea ni modifica tablas — solo VERIFICA que las entidades (`@Entity`) coincidan con lo que Flyway ya creó, fallando temprano si hay una discrepancia (por ejemplo, si agregaste un campo a `Cliente` pero te olvidaste de escribir la migración correspondiente).

Al arrancar, Spring Boot detecta `flyway-core` en el classpath y ejecuta Flyway AUTOMÁTICAMENTE, ANTES de que Hibernate/JPA se inicialice — así el esquema ya existe cuando JPA hace su verificación de `validate`.

## Cómo funciona

1. `V1__crear_clientes.sql` en `src/main/resources/db/migration/` define el `CREATE TABLE clientes` explícitamente, en SQL puro (compatible con H2).
2. Al arrancar, Flyway crea automáticamente una tabla `flyway_schema_history` (si no existe), revisa qué migraciones ya se aplicaron, y ejecuta las que faltan — en este caso, `V1` completa.
3. `spring.jpa.hibernate.ddl-auto=validate` hace que Hibernate compare `Cliente` (la entidad) con la tabla `clientes` YA CREADA por Flyway — si coinciden, la aplicación arranca normalmente; si no, falla con un error claro.
4. Todo el resto de la aplicación (`ClienteRepository`, sus métodos derivados, `@Query`) sigue funcionando EXACTAMENTE igual que en la Etapa 15 — Flyway solo cambió QUIÉN crea el esquema, no cómo se usa después.

## Ejemplo mínimo

**Fragmento agregado a `proyecto-maven/pom.xml`:**

```xml
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>
```

**Archivo:** `proyecto-maven/src/main/resources/db/migration/V1__crear_clientes.sql`

```sql
CREATE TABLE clientes (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    nombre VARCHAR(255),
    email VARCHAR(255),
    direccion_envio VARCHAR(255)
);
```

**Fragmento actualizado de `proyecto-maven/src/main/resources/application.properties`:**

```properties
spring.jpa.hibernate.ddl-auto=validate
spring.flyway.enabled=true
```

## Explicación paso a paso

- "Migrating schema PUBLIC to version 1 - crear clientes" confirma que Flyway leyó `V1__crear_clientes.sql`, extrajo el número de versión (1) y la descripción ("crear clientes", con guiones bajos convertidos a espacios), y ejecutó el SQL.
- "Successfully applied 1 migration" — la tabla `clientes` ahora existe, creada por Flyway, NO por Hibernate.
- La aplicación arrancó SIN errores de `validate` — porque las columnas que `V1` creó (`id`, `nombre`, `email`, `direccion_envio`) coinciden EXACTAMENTE con lo que `Cliente` (Etapa 15 Lección 2) espera.
- "Cliente guardado con id: 1", "Total de clientes: 1" — TODO el comportamiento de `ClienteRepository` sigue idéntico a la Etapa 15, confirmando que Flyway solo cambió la CREACIÓN del esquema, no su uso posterior.

## Ejemplo aplicado al proyecto

Con Flyway, el esquema de `clientes` queda registrado como un ARCHIVO versionado en el repositorio (`V1__crear_clientes.sql`) — cualquier persona que clone el proyecto, o cualquier ambiente donde se despliegue, va a terminar con EXACTAMENTE la misma estructura de base de datos, de forma reproducible y auditable.

## Ejercicio guiado

Agregá `V2__agregar_telefono_clientes.sql` que agregue una columna `telefono VARCHAR(20)` a la tabla `clientes`, sin modificar `V1` (las migraciones aplicadas nunca se editan).

<Reveal titulo="Pista 1">
El nombre del archivo debe seguir la convención `V2__descripcion.sql` — el número debe ser MAYOR al de la última migración aplicada.
</Reveal>

<Reveal titulo="Solución">
```sql
ALTER TABLE clientes ADD COLUMN telefono VARCHAR(20);
```

Guardado como `V2__agregar_telefono_clientes.sql`. Al reiniciar la aplicación, Flyway detecta que `V1` ya se aplicó y ejecuta SOLO `V2` — sin que hayas tocado `Cliente.java` todavía (agregar el campo `telefono` a la entidad sería un paso adicional, separado de la migración).
</Reveal>

## Mini reto de debugging

```
V1__crear_clientes.sql   (ya aplicada en la base local)
V1__agregar_indice.sql   (nueva, agregada por error con el mismo numero)
```

¿Qué esperás que pase al arrancar la aplicación con estos dos archivos?

<Reveal titulo="Diagnóstico">
Flyway falla con un error de migración DUPLICADA — no puede haber dos migraciones con el MISMO número de versión (`V1`). Cada migración necesita un número ÚNICO Y CRECIENTE. El archivo nuevo debería renombrarse a `V2__agregar_indice.sql`.
</Reveal>

## Resumen

- Flyway versiona el esquema de base de datos con archivos SQL numerados (`V1__descripcion.sql`).
- Las migraciones se aplican en orden, una sola vez, y quedan registradas en `flyway_schema_history`.
- `ddl-auto=validate` (en vez de `update`) hace que Hibernate solo VERIFIQUE el esquema, sin modificarlo — Flyway es quien lo crea.
- Una migración ya aplicada NUNCA debe editarse — los cambios nuevos van en un archivo con número de versión mayor.

## Modelo mental final

Pensá en Flyway como el historial de commits de tu esquema de base de datos: cada migración es un "commit" que describe UN cambio específico, aplicado en orden, con un registro permanente de qué se hizo y cuándo — exactamente el mismo principio que ya usás con Git para el código, aplicado a la estructura de la base de datos.

## Conexión

`Cliente` (Etapa 15) y su tabla `clientes` ahora tienen un origen EXPLÍCITO y versionado (`V1__crear_clientes.sql`), en vez de haber sido generada implícitamente por `ddl-auto=update`. `ClienteRepository` y sus métodos de consulta (Etapa 15 Lección 3) siguen funcionando sin ningún cambio.

## Qué vas a aprender

Después de esta lección serás capaz de ejecutar SQL directo contra una base de datos real usando `JdbcTemplate`, entendiendo qué hace CADA paso (crear tabla, insertar, consultar, mapear filas a objetos) antes de que JPA automatice todo eso en las próximas lecciones.

## Por qué necesitas aprenderlo

Todo el proyecto, desde la Etapa 4, guardó datos en memoria (`RepositorioProductos`) — se pierden cada vez que la aplicación se reinicia. Un sistema real necesita persistencia DURADERA en una base de datos. Antes de usar JPA/Hibernate (que automatiza el mapeo objeto-relacional), necesitás ver QUÉ es lo que se está automatizando — de lo contrario, JPA se siente como magia en vez de una herramienta que entendés.

## Qué debes saber antes

Todo lo visto hasta la Etapa 14 — esta lección introduce una pieza de infraestructura nueva (una base de datos real) sin modificar la lógica de negocio existente.

## Recupera lo aprendido

¿Por qué `StockInsuficienteException` se mapea a `409 Conflict` en vez de `400 Bad Request`?

<Reveal titulo="Respuesta">

## Problema

Necesitás guardar datos en una base de datos REAL, no en una `List` que vive solo mientras el proceso corre. Antes de dejar que un framework mapee objetos Java a filas de tabla automáticamente, necesitás ver el trabajo MANUAL que eso reemplaza: escribir el SQL, ejecutar la consulta, y convertir cada fila del resultado en un objeto Java.

## Modelo mental

```
JdbcTemplate:  vos escribís el SQL exacto. Spring simplifica
               la conexión y el manejo de errores, pero el
               mapeo fila -> objeto lo hacés vos con un RowMapper.

(Próxima lección) JPA/Hibernate: el SQL se GENERA automáticamente
               a partir de anotaciones sobre tus clases.
```

## Explicación sencilla

`JdbcTemplate` es la forma que Spring da para trabajar con SQL sin la ceremonia manual de JDBC puro (`Connection`, `PreparedStatement`, `ResultSet`, cerrar todo en bloques `finally`). Vos seguís escribiendo el SQL exacto — `CREATE TABLE`, `INSERT`, `SELECT` — pero `JdbcTemplate` maneja la conexión, ejecuta la consulta, y te deja convertir cada fila del resultado en un objeto Java con una función que vos definís (`RowMapper`).

## Explicación técnica

`spring-boot-starter-data-jpa` trae, entre otras cosas, `spring-boot-starter-jdbc` (que incluye `JdbcTemplate`) — junto con Hibernate y Spring Data JPA que vas a usar en las próximas lecciones. `com.h2database:h2` es una base de datos EN MEMORIA escrita en Java — perfecta para desarrollo y aprendizaje, porque no requiere instalar ni configurar un servidor de base de datos externo.

`spring.datasource.url=jdbc:h2:mem:catalogo` en `application.properties` (Etapa 11 Lección 2) configura una base H2 en memoria llamada `catalogo`. Boot AUTO-CONFIGURA un `DataSource` (con pool de conexiones HikariCP) y un bean `JdbcTemplate` automáticamente, sin que escribas ninguna configuración manual — el mismo espíritu de auto-configuración que ya viste en la Etapa 11.

`jdbcTemplate.execute(sql)` ejecuta SQL que no devuelve resultados (como `CREATE TABLE`). `jdbcTemplate.update(sql, parametros...)` ejecuta `INSERT`/`UPDATE`/`DELETE`, con los `?` del SQL reemplazados de forma SEGURA por los parámetros (evitando inyección SQL — nunca concatenes valores directamente en el string SQL). `jdbcTemplate.query(sql, RowMapper)` ejecuta un `SELECT` y aplica el `RowMapper` (una lambda `(rs, rowNum) -> ...`) a CADA fila del `ResultSet`, devolviendo una `List` con el resultado de cada mapeo.

## Cómo funciona

1. `spring.datasource.url`, `driver-class-name`, `username`, `password` en `application.properties` describen CÓMO conectarse a la base H2.
2. `spring.jpa.hibernate.ddl-auto=update` y `spring.h2.console.enabled=true` se usan en las próximas lecciones (JPA); en esta, solo importa que la conexión a H2 exista.
3. `jdbcTemplate.execute("CREATE TABLE personas_demo (id INT PRIMARY KEY, nombre VARCHAR(100))")` crea la tabla manualmente — SQL puro, sin ninguna anotación de mapeo.
4. `jdbcTemplate.update("INSERT INTO personas_demo (id, nombre) VALUES (?, ?)", 1, "Ana")` inserta una fila, con los `?` reemplazados por los argumentos en orden.
5. `jdbcTemplate.query("SELECT nombre FROM personas_demo ORDER BY id", (rs, rowNum) -> rs.getString("nombre"))` ejecuta el `SELECT` y convierte cada fila en un `String` (el nombre), usando `ResultSet.getString(columna)` manualmente.

## Ejemplo mínimo

**Fragmento agregado a `proyecto-maven/pom.xml`:**

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>
<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <scope>runtime</scope>
</dependency>
```

**Fragmento agregado a `proyecto-maven/src/main/resources/application.properties`:**

```properties
spring.datasource.url=jdbc:h2:mem:catalogo;DB_CLOSE_DELAY=-1
spring.datasource.driver-class-name=org.h2.Driver
spring.datasource.username=sa
spring.datasource.password=
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
spring.h2.console.enabled=true
```

**Fragmento agregado a `proyecto-maven/src/main/java/com/curso/catalogo/Application.java`:**

```java
@Bean
public CommandLineRunner jdbcDemo(JdbcTemplate jdbcTemplate) {
    return args -> {
        jdbcTemplate.execute("CREATE TABLE personas_demo (id INT PRIMARY KEY, nombre VARCHAR(100))");
        jdbcTemplate.update("INSERT INTO personas_demo (id, nombre) VALUES (?, ?)", 1, "Ana");
        jdbcTemplate.update("INSERT INTO personas_demo (id, nombre) VALUES (?, ?)", 2, "Beto");

        List<String> nombres = jdbcTemplate.query(
                "SELECT nombre FROM personas_demo ORDER BY id",
                (rs, rowNum) -> rs.getString("nombre"));

        System.out.println("Nombres desde JdbcTemplate: " + nombres);
    };
}
```

## Explicación paso a paso

- Los logs de arranque muestran `HikariPool-1` conectándose a `jdbc:h2:mem:catalogo` — la base de datos REAL (aunque en memoria) está funcionando, no es una simulación.
- "Nombres desde JdbcTemplate: [Ana, Beto]" confirma el ciclo completo: crear tabla, insertar dos filas, consultar, y mapear cada fila a un `String` — TODO escrito explícitamente por vos, sin ninguna anotación de mapeo automático.
- El orden `[Ana, Beto]` coincide con `ORDER BY id` — Ana (id 1) antes que Beto (id 2), confirmando que el SQL se ejecutó tal como se escribió.
- Este mismo patrón (SQL manual + `RowMapper` manual) es exactamente lo que JPA/Hibernate (próxima lección) reemplaza con anotaciones — vas a poder comparar directamente el "antes" y el "después".

## Mini reto de debugging

```java
jdbcTemplate.update("INSERT INTO personas_demo (id, nombre) VALUES (" + id + ", '" + nombre + "')");
```

¿Qué problema de seguridad tiene esta línea, comparada con `jdbcTemplate.update("INSERT INTO personas_demo (id, nombre) VALUES (?, ?)", id, nombre)`?

<Reveal titulo="Diagnóstico">
Es vulnerable a **inyección SQL**: si `nombre` viniera de una entrada de usuario y contuviera algo como `'; DROP TABLE personas_demo; --`, esa cadena se ejecutaría COMO SQL, no como un simple valor de texto. Los parámetros con `?` (placeholders) le dicen a la base de datos "esto es un VALOR, nunca lo interpretes como código SQL" — concatenar strings manualmente en el SQL nunca es seguro cuando el valor puede venir de afuera.
</Reveal>

## Resumen

- `JdbcTemplate` simplifica la mecánica de JDBC (conexiones, errores) sin ocultar el SQL — vos seguís escribiéndolo.
- `RowMapper` convierte cada fila de un `ResultSet` en un objeto Java, explícitamente.
- H2 en memoria permite desarrollar contra una base de datos real sin instalar nada externo.
- Los parámetros `?` en el SQL previenen inyección SQL — nunca concatenar valores externos directamente.

## Modelo mental final

Pensá en JDBC puro como cocinar desde cero cada ingrediente; `JdbcTemplate` como tener los utensilios y el fuego ya preparados, pero la receta (el SQL) la seguís escribiendo vos; y JPA/Hibernate (próxima lección) como pedirle a un chef automático que cocine el plato completo a partir de una descripción de lo que querés — cada nivel automatiza más, a costa de menos control directo.

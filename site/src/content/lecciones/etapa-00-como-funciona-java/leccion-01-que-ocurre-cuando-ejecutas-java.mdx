## Qué vas a aprender

Después de esta lección vas a ser capaz de explicar qué ocurre entre escribir `HolaMundo.java` y ver el resultado en pantalla, y de ejecutar ese proceso manualmente con `javac` y `java`, sin depender de un botón "Run" de un IDE.

## Por qué necesitas aprenderlo

Todo lo que vas a construir en este curso —incluido el Sistema de Gestión de Pedidos e Inventario que vas a ir armando etapa por etapa— corre sobre la JVM. Más adelante vas a usar Spring Boot, vas a empaquetar aplicaciones con Maven, vas a leer stack traces y vas a diagnosticar errores de arranque. Nada de eso tiene sentido si no entiendes qué es la JVM, qué es el bytecode y por qué Java compila en un paso separado de la ejecución. Esta lección es la base sobre la que se apoya literalmente todo lo demás.

## Problema

Escribes un archivo de texto plano con código Java, por ejemplo `HolaMundo.java`. Ese archivo es texto: la computadora no puede "ejecutar texto". ¿Cómo se convierte ese texto en un programa que corre de verdad? ¿Y por qué en Java hace falta un paso explícito de compilación antes de poder correrlo, en lugar de simplemente "ejecutar el archivo" como pasa con un script?

## Modelo mental

```
HolaMundo.java  --[ javac ]-->  HolaMundo.class  --[ JVM ]-->  Sistema operativo / CPU
 (código fuente,                  (bytecode,                    (resultado en pantalla)
  texto legible)                  independiente de la           
                                   plataforma)
```

Dos pasos, dos herramientas distintas: `javac` compila (código fuente → bytecode), `java` ejecuta (bytecode → resultado, a través de la JVM).

## Explicación sencilla

Java funciona en dos etapas separadas. Primero, un programa llamado compilador (`javac`) lee tu código fuente y lo traduce a un formato intermedio llamado bytecode, guardado en un archivo `.class`. Ese bytecode todavía no es algo que tu procesador entienda directamente. Segundo, otro programa —la JVM (Java Virtual Machine)— lee ese bytecode y lo va ejecutando, traduciéndolo sobre la marcha a instrucciones que tu sistema operativo y tu procesador sí entienden.

## Explicación técnica

Java es, a la vez, un lenguaje y una plataforma. El JDK (Java Development Kit) es el conjunto de herramientas que se instala para desarrollar: incluye el compilador `javac`, el lanzador `java`, la consola interactiva `jshell` y la propia JVM. (Vas a ver referencias a un "JRE" —Java Runtime Environment— en documentación más vieja: era el subconjunto necesario solo para *ejecutar* Java, sin herramientas de desarrollo. Hoy ese concepto quedó absorbido dentro del JDK y no hace falta instalarlo por separado.)

`javac` no traduce tu código a instrucciones nativas del procesador. Lo traduce a **bytecode**: un conjunto de instrucciones intermedias, definidas por la especificación de la JVM, que son las mismas sin importar si vas a correr el programa en Linux, Windows o macOS, en x86 o en ARM. El bytecode queda guardado en un archivo `.class` por cada clase pública.

La JVM es la que realmente ejecuta ese bytecode. Lo interpreta instrucción por instrucción y, para las partes que se ejecutan muchas veces, aplica compilación JIT (*just-in-time*): las traduce a código máquina nativo en caliente, mientras el programa corre, para que sea más rápido. La implementación de referencia, libre y de código abierto, es OpenJDK.

Esto es exactamente lo que significa la frase "write once, run anywhere" (escribe una vez, ejecuta en cualquier lado): el bytecode que se genera en una máquina es el mismo bytecode que corre en un servidor Linux en la nube, porque en el medio no hay código nativo específico de una plataforma — hay una JVM en cada plataforma que sabe interpretar ese bytecode.

Un punto de contraste que puede servir si vienes de PHP: en PHP no hay un paso de compilación explícito — el intérprete lee el script directamente en cada request. En Java, el paso de `javac` ocurre una sola vez, por adelantado (*ahead-of-time*), y produce el bytecode que la JVM va a ejecutar todas las veces que haga falta, sin volver a leer ni traducir el código fuente.

## Cómo funciona (paso a paso interno)

1. Escribes el código fuente en un archivo de texto con extensión `.java`.
2. Ejecutas `javac NombreDelArchivo.java`. El compilador analiza la sintaxis, verifica tipos y, si todo es válido, genera un archivo `NombreDeLaClase.class` con el bytecode.
3. Ejecutas `java NombreDeLaClase` (sin la extensión `.class`). La JVM arranca, carga ese archivo `.class` en memoria y busca dentro de él un método `main` con la firma exacta `public static void main(String[] args)` — ese es el punto de entrada.
4. La JVM empieza a interpretar el bytecode desde `main`, instrucción por instrucción.
5. Cada instrucción que produce una salida (como imprimir texto) termina, en algún punto de la cadena, llamando a servicios del sistema operativo — es el sistema operativo el que efectivamente escribe los caracteres en tu terminal.
6. Cuando `main` termina, la JVM finaliza el proceso (salvo que haya otros hilos corriendo).

## Explicación paso a paso (del código)

- `public class HolaMundo` — declara una clase pública llamada `HolaMundo`. En Java, si una clase es `public`, el archivo que la contiene tiene que llamarse **exactamente igual** a la clase, con el sufijo `.java` (`HolaMundo.java`). Esto no es una convención de estilo: es una regla que el compilador exige.
- `public static void main(String[] args)` — este es el método que la JVM busca como punto de entrada. `public` para que la JVM pueda invocarlo desde afuera de la clase; `static` porque la JVM lo llama sin crear antes un objeto `HolaMundo`; `void` porque no devuelve ningún valor; `String[] args` es el arreglo donde llegan los argumentos que le pases al programa por línea de comandos (en este ejemplo no usamos ninguno).
- `System.out.println("...")` — `System.out` es el flujo de salida estándar (la terminal). `println` escribe el texto que se le pasa y agrega un salto de línea al final.

## Diagnóstico (lectura del error)

El mensaje de `javac` sigue siempre el mismo patrón: `archivo:línea: error: descripción`. En este caso te dice tres cosas a la vez: en qué archivo está el problema (`holamundo.java`), en qué línea (`1`) y cuál es la causa (`class HolaMundo is public, should be declared in a file named HolaMundo.java`). No hace falta adivinar nada — el compilador dice exactamente qué corregir y dónde. Conviene acostumbrarse a leer el mensaje completo antes de tocar el código: casi siempre la solución está en la última línea con `error:`.

Otro error frecuente al empezar: correr `java HolaMundo` **antes** de haber compilado (o después de borrar el `.class`). La JVM no encuentra el bytecode y falla al arrancar, con un mensaje del estilo `Error: no se ha encontrado o cargado la clase principal HolaMundo` (el texto exacto puede variar según el idioma configurado en tu sistema, pero el mensaje siempre identifica la clase que no pudo cargar). La causa siempre es la misma: falta el paso `javac` antes del paso `java`, o el `.class` no está en la carpeta desde la que estás corriendo `java`.

## Transferencia

Piensa esta situación: compilas `HolaMundo.java` en tu máquina con Linux y le mandas el archivo `HolaMundo.class` resultante a un compañero que tiene Windows. ¿Va a poder ejecutarlo? La respuesta es sí, siempre que tenga una JVM instalada (cualquier JVM compatible, no necesariamente la misma que se usó para compilar) — porque el `.class` contiene bytecode, no código máquina específico de Linux. Es exactamente el mecanismo de portabilidad que viste en la explicación técnica: no le mandaste un ejecutable de Linux, le mandaste instrucciones intermedias que la JVM de Windows sabe interpretar igual.

## Mini reto de diseño

¿Por qué crees que Java separa la compilación y la ejecución en dos pasos distintos (`javac` y después `java`), en lugar de leer y ejecutar el código fuente directamente cada vez, como hace un lenguaje interpretado? Piénsalo en términos de qué se gana al compilar una sola vez y ejecutar el mismo bytecode muchas veces.

## Resumen

- Java tiene dos pasos separados: compilación (`javac`) y ejecución (`java`).
- `javac` traduce código fuente (`.java`) a bytecode (`.class`); no ejecuta nada ni produce resultado en pantalla.
- La JVM es la que ejecuta el bytecode, interpretándolo y, para las partes calientes, compilándolo *just-in-time* a código nativo.
- El bytecode es portable: el mismo `.class` corre en cualquier sistema operativo que tenga una JVM compatible instalada.
- El JDK incluye `javac`, `java`, `jshell` y la propia JVM; el "JRE" es un concepto histórico ya absorbido por el JDK.
- Un archivo con una clase `public` tiene que llamarse exactamente igual que esa clase, incluidas mayúsculas.
- `jshell` sirve para probar código suelto de forma interactiva, sin compilar archivos completos.

## Modelo mental final

Cuando pienses en ejecutar Java, imagina una traducción en dos etapas: primero un traductor (`javac`) pasa el texto fuente a un idioma intermedio y universal (bytecode) que cualquier "intérprete" Java entiende, sin importar el país (sistema operativo) en el que esté. Después, ese intérprete (la JVM) lee el idioma intermedio y lo va diciendo en voz alta, en el idioma local de la máquina donde corre (instrucciones nativas del procesador). El código fuente se escribe una sola vez; el intérprete se encarga de adaptarse a cada máquina.

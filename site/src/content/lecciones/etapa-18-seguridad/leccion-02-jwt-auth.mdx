## Qué vas a aprender

Después de esta lección serás capaz de implementar autenticación con JWT (JSON Web Token): un endpoint de login que recibe credenciales y devuelve un token firmado, un filtro que extrae y valida ese token en cada petición posterior, y un `SecurityFilterChain` que usa JWT en vez de HTTP Basic.

## Por qué necesitas aprenderlo

HTTP Basic (Etapa 18 L1) envía `usuario:contraseña` en CADA petición — aunque vaya en Base64, cualquiera que intercepte el tráfico puede decodificarlo. Con JWT, las credenciales viajan UNA SOLA VEZ (en el login), y el resto de las peticiones usan un token firmado digitalmente que:
- No contiene la contraseña
- Tiene fecha de expiración (24 horas, configurable)
- Es verificable sin consultar la base de datos (contiene la firma del servidor)

## Qué debes saber antes

`SecurityFilterChain`, `UserDetailsService`, `PasswordEncoder` (Etapa 18 L1). Esta lección REEMPLAZA HTTP Basic, no lo complementa.

## Recupera lo aprendido

¿Por qué `SessionCreationPolicy.STATELESS` es importante para APIs REST, independientemente de si usás HTTP Basic o JWT?

<Reveal titulo="Respuesta">
Porque una API REST, por definición, no debe mantener estado del cliente en el servidor entre peticiones — cada petición debe ser autónoma. STATELESS garantiza que Spring Security no cree `HttpSession` ni use cookies, obligando a que cada petición incluya sus propias credenciales (sean Basic o JWT).
</Reveal>

## Problema

HTTP Basic funciona, pero tiene dos problemas graves para producción:
1. La CONTRASEÑA viaja en cada petición (en Base64, decodificable)
2. El CLIENTE debe guardar la contraseña para enviarla en cada request — si el cliente es una SPA (React/Vue) o una app móvil, guardar la contraseña del usuario en localStorage es un riesgo de seguridad enorme.

JWT resuelve ambos: el cliente envía credenciales UNA vez, recibe un token temporal firmado, y usa ESE token (no la contraseña) en las peticiones siguientes. Si el token es robado, expira en horas — la contraseña nunca estuvo expuesta en el tráfico repetido.

## Modelo mental

```
HTTP Basic:           cada peticion lleva usuario:contraseña
                      como un carnet que muestra tu PIN cada vez.

JWT:                  login UNA vez -> recibis un pase temporal
                      firmado por el servidor. Cada peticion
                      muestra el PASE, no la contraseña.
                      El pase expira solo.
```

## Explicación sencilla

JWT es un token (string largo) dividido en tres partes separadas por puntos: `header.payload.signature`. El servidor CREA el token durante el login, firmándolo con una clave secreta que solo él conoce. En cada petición posterior, el cliente envía el token en la cabecera `Authorization: Bearer <token>`. El servidor verifica la firma — si es válida y no expiró, extrae el `username` del payload y autentica al usuario, sin necesidad de consultar la base de datos.

## Explicación técnica

`JwtUtil` encapsula la lógica de creación y validación de tokens usando la biblioteca `jjwt` (io.jsonwebtoken). `generateToken(username)` construye un JWT con:
- `subject`: el username
- `issuedAt`: timestamp de creación
- `expiration`: timestamp de expiración (24 horas después)
- `signWith(key)`: firma HMAC-SHA512 con una clave secreta de 256+ bits

`extractUsername(token)` verifica la firma y extrae el `subject` del payload — si la firma es inválida o el token expiró, lanza excepción.

`JwtAuthenticationFilter` extiende `OncePerRequestFilter` (se ejecuta UNA vez por petición HTTP, garantizado). Extrae el token de la cabecera `Authorization: Bearer <token>`, lo valida, extrae el username, carga el `UserDetails`, y establece la autenticación en el `SecurityContextHolder` — a partir de ese momento, Spring Security considera al usuario autenticado para ESA petición.

`AuthController` expone `POST /api/auth/login` (público, sin autenticación) que recibe `{username, password}`, autentica contra el `AuthenticationManager`, y devuelve `{token: "eyJ..."}`.

## Cómo funciona

1. `POST /api/auth/login` con `{"username":"admin","password":"admin123"}` → el `AuthenticationManager` valida credenciales contra `UserDetailsService` (Etapa 18 L1) y `PasswordEncoder`.
2. Si las credenciales son correctas, `JwtUtil.generateToken("admin")` crea un JWT firmado con expiración de 24 horas, y el controlador lo devuelve.
3. En CADA petición posterior a endpoints protegidos, el cliente envía `Authorization: Bearer eyJ...` en la cabecera HTTP.
4. `JwtAuthenticationFilter` intercepta la petición ANTES de que llegue al controlador — extrae el token, verifica firma y expiración, extrae el username, y establece la autenticación.
5. `SecurityFilterChain` deja pasar la petición porque ahora el `SecurityContextHolder` contiene un usuario autenticado.
6. `ProductoController` recibe la petición normalmente — no sabe ni necesita saber si la autenticación fue por JWT, HTTP Basic, o cualquier otro mecanismo.

## Ejemplo mínimo

**Archivos nuevos creados:**

`proyecto-maven/src/main/java/com/curso/catalogo/JwtUtil.java`
```java
@Component
public class JwtUtil {
    private static final long EXPIRATION_MS = 86400000; // 24 horas
    private final SecretKey key;

    public JwtUtil() {
        byte[] secretBytes = Base64.getDecoder().decode(
                "c3VwZXItc2VjcmV0LWtleS1mb3Itand0...");
        this.key = Keys.hmacShaKeyFor(secretBytes);
    }

    public String generateToken(String username) {
        return Jwts.builder()
                .subject(username)
                .issuedAt(new Date())
                .expiration(new Date(System.currentTimeMillis() + EXPIRATION_MS))
                .signWith(key)
                .compact();
    }

    public String extractUsername(String token) {
        return Jwts.parser().verifyWith(key).build()
                .parseSignedClaims(token).getPayload().getSubject();
    }

    public boolean isTokenValid(String token) {
        try {
            Jwts.parser().verifyWith(key).build().parseSignedClaims(token);
            return true;
        } catch (Exception e) { return false; }
    }
}
```

`proyecto-maven/src/main/java/com/curso/catalogo/JwtAuthenticationFilter.java`
```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    private final JwtUtil jwtUtil;
    private final UserDetailsService userDetailsService;

    public JwtAuthenticationFilter(JwtUtil jwtUtil, UserDetailsService userDetailsService) {
        this.jwtUtil = jwtUtil;
        this.userDetailsService = userDetailsService;
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain filterChain)
            throws ServletException, IOException {
        String authHeader = request.getHeader("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        String token = authHeader.substring(7);
        if (!jwtUtil.isTokenValid(token)) {
            filterChain.doFilter(request, response);
            return;
        }

        String username = jwtUtil.extractUsername(token);
        if (username != null && SecurityContextHolder.getContext()
                .getAuthentication() == null) {
            UserDetails userDetails = userDetailsService.loadUserByUsername(username);
            UsernamePasswordAuthenticationToken authToken =
                new UsernamePasswordAuthenticationToken(
                    userDetails, null, userDetails.getAuthorities());
            authToken.setDetails(
                new WebAuthenticationDetailsSource().buildDetails(request));
            SecurityContextHolder.getContext().setAuthentication(authToken);
        }

        filterChain.doFilter(request, response);
    }
}
```

`proyecto-maven/src/main/java/com/curso/catalogo/AuthController.java`
```java
@RestController
@RequestMapping("/api/auth")
public class AuthController {
    private final AuthenticationManager authenticationManager;
    private final JwtUtil jwtUtil;

    public AuthController(AuthenticationManager authManager, JwtUtil jwtUtil) {
        this.authenticationManager = authManager;
        this.jwtUtil = jwtUtil;
    }

    @PostMapping("/login")
    public ResponseEntity<Map<String, String>> login(
            @RequestBody LoginRequest request) {
        Authentication auth = authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(
                request.getUsername(), request.getPassword()));
        String token = jwtUtil.generateToken(auth.getName());
        return ResponseEntity.ok(Map.of("token", token));
    }
}
```

**SecurityConfig actualizado:**
```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf.disable())
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/auth/**").permitAll()
            .requestMatchers("/api/productos/**").authenticated()
            .anyRequest().permitAll()
        )
        .sessionManagement(session ->
            session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .addFilterBefore(jwtAuthenticationFilter,
            UsernamePasswordAuthenticationFilter.class);
    return http.build();
}
```

## Predice

`/api/auth/login` está configurado como `.permitAll()`. Si un atacante intenta `POST /api/auth/login` con credenciales inválidas, ¿va a recibir 401 o va a recibir otra cosa?

<Reveal titulo="Resultado real (ejecutado con Maven)">
```
$ curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"wrongpass"}'
{"timestamp":"...","status":403,"error":"Forbidden","path":"/api/auth/login"}
```
403 Forbidden — no 401. El endpoint permite acceso SIN autenticación (permitAll), pero `AuthenticationManager.authenticate()` lanza `BadCredentialsException`, que Spring Security traduce a 403 (prohibido — acceso denegado después de intentar autenticar).
</Reveal>

## Explicación paso a paso

- `POST /api/auth/login` con credenciales correctas → `AuthenticationManager.authenticate()` compara la contraseña ingresada (hasheada con BCrypt) contra la almacenada en `UserDetails` → `JwtUtil.generateToken("admin")` crea token firmado → `{"token":"eyJ..."}`.
- `GET /api/productos` SIN token → `JwtAuthenticationFilter` encuentra `Authorization: null` → `filterChain.doFilter()` delega al siguiente filtro → `SecurityFilterChain` ve que no hay autenticación → 403.
- `GET /api/productos` CON `Authorization: Bearer eyJ...` → `JwtAuthenticationFilter` extrae token, verifica firma (HMAC-SHA512), extrae `username=admin` → `userDetailsService.loadUserByUsername("admin")` carga el usuario → `SecurityContextHolder` tiene autenticación → `SecurityFilterChain` deja pasar → `ProductoController.listar()` ejecuta normalmente.
- `/api/auth/**` = `permitAll()` porque el endpoint de login DEBE ser público — si requiriera autenticación, nadie podría loguearse por primera vez (paradoja del huevo y la gallina).

## Ejemplo aplicado al proyecto

Con JWT implementado, `ProductoController` no cambió NI UNA LÍNEA — sigue recibiendo peticiones normalmente. La diferencia es que ahora:
- Sin token (o con token inválido/expirado): 403
- Con token válido de login: 200, respuesta normal
- `/api/auth/login` es público (no requiere token)
- `SecurityContextHolder` contiene el usuario autenticado, accesible desde cualquier parte del código con `SecurityContextHolder.getContext().getAuthentication()`

## Transferencia

JWT es autocontenido — el token contiene toda la información necesaria para identificar al usuario, sin consultar la base de datos. Esto permite escalar horizontalmente: MÚLTIPLES instancias de tu aplicación pueden validar el mismo token (porque comparten la misma clave secreta) sin compartir estado. ¿Qué cambiarías si quisieras que un usuario pueda "cerrar sesión" y que su token deje de ser válido inmediatamente, incluso antes de expirar?

## Mini reto de debugging

```java
String authHeader = request.getHeader("Authorization");
if (authHeader == null || !authHeader.startsWith("Bearer ")) {
    filterChain.doFilter(request, response);
    return;
}
String token = authHeader.substring(7);  // quita "Bearer "
```

¿Qué pasa si el cliente envía `Authorization: Bearer` (sin token después de "Bearer ")?

<Reveal titulo="Diagnóstico">
`authHeader.substring(7)` sobre `"Bearer "` (7 caracteres exactos, sin token) devuelve `""` (string vacío). `jwtUtil.isTokenValid("")` lanza una excepción (el parser de JWT no reconoce un token vacío), que `JwtAuthenticationFilter` no captura dentro de `isTokenValid()` (que YA tiene try/catch) — `isTokenValid` devuelve `false`, y el filtro simplemente pasa la petición sin autenticar, resultando en 403. El filtro maneja correctamente este caso borde gracias al try/catch en isTokenValid().
</Reveal>

## Mini reto de diseño

`JwtUtil` contiene una clave secreta hardcodeada en Base64 — en producción, esto es inaceptable (la clave termina en el repositorio Git). ¿Dónde y cómo deberías almacenar esa clave?

<Reveal titulo="Respuesta">
En `application.properties` (o variables de entorno) como `jwt.secret=${JWT_SECRET}`, inyectada con `@Value`. En desarrollo local, un valor por defecto en `application.properties` es aceptable. En producción, la variable de entorno `JWT_SECRET` se configura en el sistema de despliegue (Kubernetes secrets, GitHub Actions secrets, etc.) — NUNCA en el código fuente.
</Reveal>

## Resumen

- JWT reemplaza HTTP Basic: el cliente envía credenciales UNA vez en `/api/auth/login` y recibe un token firmado.
- `JwtUtil` genera tokens con expiración y los valida usando firma HMAC-SHA512.
- `JwtAuthenticationFilter` (OncePerRequestFilter) extrae el token de cada petición, lo valida, y establece la autenticación en `SecurityContextHolder`.
- `SecurityFilterChain` reemplaza `.httpBasic()` por `.addFilterBefore(jwtAuthenticationFilter, ...)`.
- `/api/auth/**` debe ser `permitAll()` — si el login requiere autenticación, nadie puede obtener un token.

## Modelo mental final

HTTP Basic es como mostrar tu DNI con tu dirección y teléfono en CADA negocio al que entrás. JWT es como registrarte UNA vez en la entrada del edificio y recibir una tarjeta de visitante temporal — mostrás la tarjeta en cada piso, no tu DNI.

## Conexión

Así como `RepositorioProductos` desacopla `ServicioVentas` de CÓMO se almacenan los productos (Etapa 8 L1), `JwtAuthenticationFilter` desacopla `ProductoController` de CÓMO se autentica al usuario — el controlador solo ve `SecurityContextHolder.getContext().getAuthentication()`, sin importar si fue JWT, HTTP Basic, OAuth2, o cualquier otro mecanismo.

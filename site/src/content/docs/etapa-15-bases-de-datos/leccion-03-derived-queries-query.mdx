## Qué vas a aprender

Después de esta lección serás capaz de agregar búsquedas específicas a un `JpaRepository` escribiendo SOLO la firma del método (Spring Data genera el SQL a partir del NOMBRE), y de usar `@Query` con JPQL cuando la consulta es demasiado compleja para expresarse en un nombre de método.

## Por qué necesitas aprenderlo

`ClienteRepository` (Etapa 15 Lección 2) solo tenía los métodos heredados de `JpaRepository` (`save`, `findById`, `findAll`, `count`) — genéricos, sin conocer nada específico de `Cliente`. Necesitás buscar un cliente por email, o por fragmento de nombre, o por ciudad — consultas de NEGOCIO que `JpaRepository` no puede adivinar por sí solo.

## Qué debes saber antes

`JpaRepository`, `@Entity` (Etapa 15 Lección 2) — esta lección extiende `ClienteRepository` con métodos nuevos.

## Recupera lo aprendido

¿Por qué `Cliente` necesita un constructor `protected Cliente() { }` sin argumentos, ADEMÁS de su constructor de negocio?

<Reveal titulo="Respuesta">
Porque Hibernate lo necesita internamente (vía reflection) para reconstruir objetos Cliente a partir de filas de la base de datos — es un requisito del contrato de JPA, no algo que tu código de aplicación use directamente.
</Reveal>

## Problema

Necesitás: buscar un cliente por su email exacto, buscar clientes cuyo nombre CONTENGA cierto texto, y buscar clientes por ciudad (un campo que vive DENTRO de `direccionEnvio`, no como columna separada). Ninguna de estas consultas existe en `JpaRepository` — no hay forma de que Spring Data adivine QUÉ campo específico de `Cliente` te interesa sin que se lo digas.

## Modelo mental

```
Metodo derivado:   Spring Data LEE el nombre del método
                    (findByEmail, findByNombreContaining) y
                    genera el SQL correspondiente automáticamente.

@Query:            vos escribís la consulta explícitamente
                    (en JPQL, no SQL) cuando el nombre del
                    método no alcanza para expresar la lógica.
```

## Explicación sencilla

Spring Data JPA puede LEER el nombre de un método que declarás en la interfaz y, siguiendo una convención (`findBy` + nombre del campo + palabras clave como `Containing`), genera automáticamente la consulta SQL correspondiente — sin que escribas ni una línea de SQL. Cuando la consulta es demasiado específica para expresarse con ese patrón de nombres, `@Query` te permite escribir la consulta explícitamente en JPQL (una versión de SQL orientada a objetos, que usa nombres de clases y campos Java en vez de nombres de tablas y columnas).

## Explicación técnica

`Optional<Cliente> findByEmail(String email)` — Spring Data interpreta `findBy` + `Email` (el nombre del CAMPO en `Cliente`) y genera `SELECT * FROM clientes WHERE email = ?` automáticamente. El tipo de retorno `Optional<Cliente>` (Etapa 4 Lección 7) es apropiado porque puede no haber ningún cliente con ese email.

`List<Cliente> findByNombreContaining(String fragmento)` — la palabra clave `Containing` le agrega un `LIKE '%valor%'` a la consulta generada, en vez de una igualdad exacta. Spring Data soporta MUCHAS palabras clave similares: `StartingWith`, `EndingWith`, `GreaterThan`, `OrderBy`, y más — todas se combinan en el NOMBRE del método.

`@Query("SELECT c FROM Cliente c WHERE c.direccionEnvio LIKE %:ciudad%")` es JPQL (Jakarta Persistence Query Language) — se parece a SQL, pero opera sobre ENTIDADES Java (`Cliente`, `c.direccionEnvio`) en vez de tablas y columnas SQL directamente (`clientes`, `direccion_envio`). `@Param("ciudad")` conecta el parámetro del método con el placeholder `:ciudad` en la consulta. `@Query` es necesario acá porque "buscar por un FRAGMENTO dentro de un campo que no es exactamente 'ciudad'" no tiene una convención de nombre de método directa y clara.

## Cómo funciona

1. `findByEmail(String email)` genera `WHERE email = ?` — coincidencia EXACTA.
2. `findByNombreContaining(String fragmento)` genera `WHERE nombre LIKE '%fragmento%'` — coincidencia PARCIAL, en cualquier parte del campo.
3. `buscarPorCiudad(String ciudad)`, con `@Query` explícito, busca dentro de `direccionEnvio` (un campo que combina calle Y ciudad) — algo que ninguna convención de nombre de método podría expresar directamente, porque "ciudad" no es un campo real de `Cliente`.
4. Ninguno de los tres métodos tiene CUERPO — son solo firmas en una interfaz; Spring Data genera la implementación completa (el SQL, la ejecución, el mapeo del resultado) en tiempo de ejecución.

## Ejemplo mínimo

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/ClienteRepository.java`

```java
package com.curso.catalogo;

import java.util.List;
import java.util.Optional;

import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.query.Param;

public interface ClienteRepository extends JpaRepository<Cliente, Long> {

    Optional<Cliente> findByEmail(String email);

    List<Cliente> findByNombreContaining(String fragmento);

    @Query("SELECT c FROM Cliente c WHERE c.direccionEnvio LIKE %:ciudad%")
    List<Cliente> buscarPorCiudad(@Param("ciudad") String ciudad);
}
```

**Fragmento actualizado de `proyecto-maven/src/main/java/com/curso/catalogo/Application.java`:**

```java
clienteRepository.save(new Cliente("Juan Gomez", "juan.gomez@ejemplo.com", "Calle Falsa 456, Rosario"));

Optional<Cliente> porEmail = clienteRepository.findByEmail("juan@ejemplo.com");
porEmail.ifPresent(c -> System.out.println("Encontrado por email: " + c.getNombre()));

List<Cliente> porNombre = clienteRepository.findByNombreContaining("Juan");
System.out.println("Clientes con 'Juan' en el nombre: " + porNombre.size());

List<Cliente> porCiudad = clienteRepository.buscarPorCiudad("Cordoba");
System.out.println("Clientes en Cordoba: " + porCiudad.size());
```

## Explicación paso a paso

- "Encontrado por email: Juan Perez" confirma que `findByEmail("juan@ejemplo.com")` encontró EXACTAMENTE el cliente correcto, sin ambigüedad — coincidencia exacta de campo.
- "Clientes con 'Juan' en el nombre: 2" — AMBOS clientes ("Juan Perez" y "Juan Gomez") coinciden, porque `Containing` busca el fragmento "Juan" en CUALQUIER parte del campo `nombre`.
- "Clientes en Cordoba: 1" — solo "Juan Perez" (`direccionEnvio` = "Av. Siempre Viva 123, Cordoba") coincide; "Juan Gomez" vive en Rosario, así que el `LIKE '%Cordoba%'` generado por `@Query` lo excluye correctamente.
- Ninguno de estos tres métodos tiene una sola línea de implementación en `ClienteRepository` — Spring Data generó TODO el SQL a partir de la firma del método (los dos primeros) o de la anotación `@Query` (el tercero).

## Ejemplo aplicado al proyecto

Con estos métodos, `ClienteRepository` ahora expresa consultas de NEGOCIO reales (buscar por email para login, buscar por nombre para autocompletado, buscar por ciudad para logística de envíos) sin que ninguna de esas consultas requiera escribir SQL manualmente, como hubiera sido necesario con `JdbcTemplate` (Etapa 15 Lección 1).

## Mini reto de debugging

```java
@Query("SELECT * FROM clientes WHERE email = :email")
Optional<Cliente> buscarPorEmailMalEscrito(@Param("email") String email);
```

¿Qué problema tiene esta consulta, sabiendo que `@Query` en Spring Data JPA usa JPQL por defecto (no SQL nativo)?

<Reveal titulo="Diagnóstico">
`SELECT * FROM clientes` es SQL, no JPQL — JPQL opera sobre ENTIDADES y sus CAMPOS (`SELECT c FROM Cliente c WHERE c.email = :email`), no sobre tablas y columnas SQL directamente. Esta consulta fallaría al arrancar la aplicación con un error de sintaxis JPQL. Para usar SQL nativo real, `@Query` necesita el atributo adicional `nativeQuery = true`.
</Reveal>

## Resumen

- Spring Data JPA genera consultas automáticamente a partir del NOMBRE de un método (`findBy` + campo + palabras clave como `Containing`).
- `@Query` con JPQL cubre los casos donde la convención de nombres no alcanza.
- JPQL opera sobre entidades y campos Java (`Cliente`, `c.direccionEnvio`), no sobre tablas y columnas SQL directamente.
- Ningún método de consulta (derivado o `@Query`) necesita implementación escrita a mano — Spring Data genera todo en tiempo de ejecución.

## Modelo mental final

Pensá en un método derivado como pedirle algo a un asistente MUY literal: si le decís exactamente "buscame por email", entiende perfecto. Pero si la petición es más elaborada ("buscame por la ciudad que está escondida dentro de la dirección"), necesitás darle instrucciones explícitas (`@Query`) en vez de confiar en que adivine solo con el nombre.

## Qué vas a aprender

Después de esta lección serás capaz de encadenar `filter`, `map`, `count` y `forEach` sobre un `Stream` para procesar una colección completa en pocas líneas declarativas, en vez de escribir un bucle `for` manual con lógica intermedia.

## Por qué necesitas aprenderlo

Hasta ahora, cualquier procesamiento de `repositorio.listarTodos()` (filtrar, transformar, contar) requería un bucle `for` con una condición `if` adentro y una lista auxiliar para acumular resultados. Los Streams usan exactamente las interfaces funcionales de la lección anterior (`Predicate`, `Function`, `Consumer`) para expresar lo MISMO sin ese código repetitivo.

## Qué debes saber antes

`Predicate<T>`, `Function<T, R>` y `Consumer<T>` (Etapa 5 Lección 1) — un Stream se configura pasándole exactamente estas interfaces.

## Recupera lo aprendido

¿Qué diferencia hay entre `Function<T, R>` y `Consumer<T>`?

<Reveal titulo="Respuesta">
Function transforma un T en un R y DEVUELVE ese resultado (apply); Consumer recibe un T y actúa sobre él SIN devolver nada (accept) — típicamente para efectos como imprimir.
</Reveal>

## Problema

Necesitás: (1) quedarte solo con los productos con poco stock, (2) obtener solo sus nombres como lista de strings, (3) contar cuántos productos no tienen stock, y (4) imprimir los productos caros. Escribir esto con cuatro bucles `for` distintos, cada uno con su lista auxiliar, es mucho código repetitivo para una intención simple: "de esta colección, quiero esto otro".

## Modelo mental

```
coleccion.stream()
    .filter(condicion)      // se queda solo con los que cumplen
    .map(transformacion)    // convierte cada elemento en otra cosa
    .collect(...)           // vuelca el resultado a una lista nueva
```

## Explicación sencilla

Un `Stream` es una tubería (pipeline) por la que pasan los elementos de una colección, uno por uno, atravesando una secuencia de operaciones que vos definís. `filter` deja pasar solo los que cumplen una condición. `map` transforma cada elemento en otra cosa. `collect` junta el resultado final en una lista (u otra estructura). El Stream en sí NO modifica la colección original — produce un resultado nuevo.

## Explicación técnica

`coleccion.stream()` crea un `Stream<T>` a partir de cualquier `Collection` (como `List<T>`). Las operaciones se dividen en dos tipos:

- **Intermedias** (`filter`, `map`): devuelven OTRO Stream, así que se pueden encadenar. No hacen nada por sí solas — son perezosas (lazy).
- **Terminales** (`collect`, `count`, `forEach`): ejecutan todo el pipeline y producen un resultado final (una lista, un número, o un efecto). Sin una operación terminal, el Stream nunca se procesa.

`filter(Predicate<T>)` usa exactamente la interfaz funcional de la lección anterior. `map(Function<T, R>)` también. `Product::getNombre` es una **method reference** — una forma aún más corta de escribir `p -> p.getNombre()` cuando la lambda solo llama a un método existente. `collect(Collectors.toList())` es la forma estándar de volcar el resultado de un Stream a una `List` nueva.

## Cómo funciona

1. `repositorio.listarTodos().stream()` convierte la `List<Product>` en un `Stream<Product>`.
2. `.filter(p -> p.getStock() > 0 && p.getStock() < 20)` deja pasar solo los productos con stock entre 1 y 19.
3. `.collect(Collectors.toList())` junta el resultado filtrado en una `List<Product>` nueva — la lista original de `repositorio` no se toca.
4. `.map(Product::getNombre)` transforma cada `Product` en su `String` nombre — el Stream resultante es de tipo `Stream<String>`, no `Stream<Product>`.
5. `.count()` es una operación terminal que devuelve un `long` con la cantidad de elementos que llegaron hasta ahí.
6. `.forEach(accion)` ejecuta la acción sobre cada elemento que sobrevivió el `filter` — es la versión Stream del bucle `for` con `System.out.println` adentro.

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-05/StreamsDemo.java`

```java
import java.util.List;
import java.util.stream.Collectors;

public class StreamsDemo {
    public static void main(String[] args) {
        RepositorioEnMemoria<Product> repositorio = new RepositorioEnMemoria<>();
        repositorio.agregar(new Product("Teclado mecanico", 25000.0, 15));
        repositorio.agregar(new Product("Mouse inalambrico", 8000.0, 40));
        repositorio.agregar(new Product("Monitor 27 pulgadas", 180000.0, 3));
        repositorio.agregar(new Product("Auriculares", 15000.0, 0));

        List<Product> conStockBajo = repositorio.listarTodos().stream()
                .filter(p -> p.getStock() > 0 && p.getStock() < 20)
                .collect(Collectors.toList());
        System.out.println("Con stock bajo: " + conStockBajo.size());

        List<String> nombres = repositorio.listarTodos().stream()
                .map(Product::getNombre)
                .collect(Collectors.toList());
        System.out.println("Nombres: " + nombres);

        long sinStock = repositorio.listarTodos().stream()
                .filter(p -> p.getStock() == 0)
                .count();
        System.out.println("Sin stock: " + sinStock);

        repositorio.listarTodos().stream()
                .filter(p -> p.getPrecio() > 10000.0)
                .forEach(p -> System.out.println("Caro: " + p.getNombre()));
    }
}
```

## Predice

Hay 4 productos: Teclado (stock 15), Mouse (stock 40), Monitor (stock 3), Auriculares (stock 0). El filtro de "stock bajo" pide `stock > 0 && stock < 20`. ¿Cuántos productos pasan ese filtro?

<Reveal titulo="Resultado real (compilado y ejecutado)">
```
Con stock bajo: 2
Nombres: [Teclado mecanico, Mouse inalambrico, Monitor 27 pulgadas, Auriculares]
Sin stock: 1
Caro: Teclado mecanico
Caro: Monitor 27 pulgadas
Caro: Auriculares
```
</Reveal>

## Explicación paso a paso

- "Con stock bajo: 2" — solo Teclado (15) y Monitor (3) cumplen `stock > 0 && stock < 20`; Mouse tiene 40 (demasiado) y Auriculares tiene 0 (excluido explícitamente por `> 0`).
- "Nombres" lista los 4 productos completos porque ese `map` NO tiene `filter` antes — se aplica sobre TODO `listarTodos()`, sin descartar ninguno.
- "Sin stock: 1" — solo Auriculares tiene `stock == 0`.
- Los "Caro" son los que superan $10.000: Mouse ($8.000) queda afuera, los otros tres entran.
- Cada uno de los cuatro bloques usa `repositorio.listarTodos().stream()` de forma independiente — los Streams no se reutilizan una vez consumidos por una operación terminal.

## Ejemplo aplicado al proyecto

`RepositorioEnMemoria<T>.listarTodos()` (Etapa 4) ya devolvía una `List<T>` — con Streams, esa lista se vuelve la base de cualquier consulta declarativa (filtrar, transformar, contar) sin necesitar métodos nuevos en la propia clase `RepositorioEnMemoria`.

## Ejercicio guiado

Usá `map` para obtener una `List<Double>` con los precios de todos los productos, y después sumá esos precios con un bucle `for` normal (todavía no viste `reduce`, que resuelve esto de forma declarativa en la próxima lección).

<Reveal titulo="Pista 1">
`map(Product::getPrecio)` transforma cada `Product` en su `double` precio — pero cuidado, `Collectors.toList()` sobre un Stream de `double` primitivo requiere que el Stream sea de `Double` (objeto), no `double` (primitivo), porque `map` sobre un `Stream<T>` produce `Stream<R>` con R como tipo de referencia.
</Reveal>

<Reveal titulo="Solución">
```java
List<Double> precios = repositorio.listarTodos().stream()
        .map(Product::getPrecio)
        .collect(Collectors.toList());

double total = 0;
for (double precio : precios) {
    total += precio;
}
System.out.println("Total: " + total);
```
</Reveal>

## Resumen

- `stream()` convierte una `Collection` en un `Stream` para procesarla de forma declarativa.
- `filter` descarta elementos; `map` los transforma; ambas son intermedias y se encadenan.
- `collect(Collectors.toList())`, `count()` y `forEach()` son operaciones terminales que ejecutan el pipeline.
- Un Stream se consume una sola vez — si necesitás dos resultados, hay que crear el Stream de nuevo.

## Modelo mental final

Pensá en un Stream como una cinta transportadora: `filter` es una estación que saca lo que no sirve, `map` es una estación que transforma lo que pasa, y la operación terminal es la caja al final de la cinta donde cae el resultado — sin esa caja, la cinta nunca se pone en marcha.

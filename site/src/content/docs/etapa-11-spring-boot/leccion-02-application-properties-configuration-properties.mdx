## Qué vas a aprender

Después de esta lección serás capaz de externalizar valores de configuración en `application.properties` (en vez de números o strings sueltos en el código), leer un valor individual con `@Value`, y agrupar configuración relacionada de forma segura por tipos con `@ConfigurationProperties` sobre un `record`.

## Por qué necesitas aprenderlo

`UMBRAL_DESCUENTO` (Etapa 7, Clean Code) era una constante FIJA en el código — cambiar su valor exigía recompilar. En una aplicación real, valores como umbrales de negocio, URLs, credenciales o nombres de entorno necesitan poder cambiar SIN tocar el código fuente (por ejemplo, distintos valores en desarrollo vs. producción) — `application.properties` es la forma estándar de Spring Boot para eso.

## Qué debes saber antes

`@SpringBootApplication`, `CommandLineRunner` (Etapa 11 Lección 1) — los beans de esta lección se inyectan con el mismo mecanismo.

## Problema

`UMBRAL_DESCUENTO = 50000.0` (Etapa 7) está escrito directamente en el código Java — cambiarlo exige recompilar y volver a desplegar toda la aplicación. Y el nombre de la aplicación, si quisieras mostrarlo en un log o una respuesta, tampoco debería estar hardcodeado en ningún lugar del código fuente.

## Modelo mental

```
application.properties:  archivo de configuración FUERA del código
                          compilado — clave=valor.

@Value("${clave}"):        lee UN valor individual directamente.

@ConfigurationProperties:  agrupa VARIAS claves relacionadas (mismo
                            prefijo) en un objeto tipado, con
                            validación de tipos automática.
```

## Explicación sencilla

`application.properties` (o su equivalente en YAML) es un archivo de configuración que vive en `src/main/resources`, fuera del código Java compilado — Spring Boot lo lee automáticamente al arrancar. `@Value("${clave}")` inyecta el valor de UNA clave específica directamente en un campo o parámetro. `@ConfigurationProperties` es la alternativa "en grupo": en vez de una anotación `@Value` por cada propiedad, agrupás varias relacionadas (con el mismo prefijo) en un solo objeto, con los TIPOS ya validados por Spring al arrancar.

## Explicación técnica

`catalogo.nombre-app=Catalogo Demo` en `application.properties` define una clave `catalogo.nombre-app` — Spring Boot convierte automáticamente el `kebab-case` (con guiones) del archivo a `camelCase` en Java (`nombreApp`), siguiendo una convención llamada **relaxed binding**.

`@Value("${catalogo.nombre-app}")` como parámetro de un método `@Bean` (o campo `@Autowired`) le pide a Spring que resuelva ESE valor específico del entorno de configuración y lo inyecte directamente — útil para valores sueltos, pero se vuelve repetitivo si necesitás VARIOS relacionados.

`@ConfigurationProperties(prefix = "catalogo")` sobre un `record` (`CatalogoProperties(String nombreApp, double umbralDescuento)`) le dice a Spring: "tomá TODAS las claves que empiecen con `catalogo.` y construí este record con ellas" — `nombreApp` se llena con `catalogo.nombre-app`, `umbralDescuento` con `catalogo.umbral-descuento`. Usar un `record` (Etapa 4 Lección 5) aquí es la forma MODERNA y recomendada en Spring Boot 3.x: inmutable, con validación de tipos automática (si `catalogo.umbral-descuento` no fuera un número válido, la aplicación fallaría al ARRANCAR, no en tiempo de ejecución más adelante).

`@ConfigurationPropertiesScan` sobre `Application` le dice a Spring que busque automáticamente TODAS las clases anotadas con `@ConfigurationProperties` en el proyecto y las registre como beans, sin necesitar declarar cada una explícitamente.

## Cómo funciona

1. `application.properties` define `catalogo.nombre-app` y `catalogo.umbral-descuento`.
2. `CatalogoProperties` (un `record` con `@ConfigurationProperties(prefix = "catalogo")`) se registra automáticamente como bean gracias a `@ConfigurationPropertiesScan` en `Application`.
3. `configDemo(CatalogoProperties propiedades, @Value("${catalogo.nombre-app}") String nombreAppPorValue)` recibe AMBAS formas de acceder a la configuración: el objeto agrupado, y un valor individual leído directamente.
4. `propiedades.nombreApp()` y `propiedades.umbralDescuento()` acceden a los campos del record con la sintaxis de accessor automática (Etapa 4 Lección 5).

## Ejemplo mínimo

**Archivo:** `proyecto-maven/src/main/resources/application.properties`

```properties
catalogo.nombre-app=Catalogo Demo
catalogo.umbral-descuento=100000.0
```

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/CatalogoProperties.java`

```java
package com.curso.catalogo;

import org.springframework.boot.context.properties.ConfigurationProperties;

@ConfigurationProperties(prefix = "catalogo")
public record CatalogoProperties(String nombreApp, double umbralDescuento) {
}
```

**Fragmento agregado a `proyecto-maven/src/main/java/com/curso/catalogo/Application.java`:**

```java
@SpringBootApplication
@ConfigurationPropertiesScan
public class Application {

    // ... main() y demo() sin cambios ...

    @Bean
    public CommandLineRunner configDemo(CatalogoProperties propiedades,
                                         @Value("${catalogo.nombre-app}") String nombreAppPorValue) {
        return args -> {
            System.out.println("Nombre de la app (ConfigurationProperties): " + propiedades.nombreApp());
            System.out.println("Umbral de descuento (ConfigurationProperties): " + propiedades.umbralDescuento());
            System.out.println("Nombre de la app (Value): " + nombreAppPorValue);
        };
    }
}
```

## Predice

`catalogo.nombre-app=Catalogo Demo` y `catalogo.umbral-descuento=100000.0` en `application.properties`. ¿Qué valores esperás leer de `propiedades.nombreApp()` y `nombreAppPorValue`?

<Reveal titulo="Resultado real (ejecutado con Spring Boot)">
```
$ ./mvnw -q spring-boot:run
...
Nombre de la app (ConfigurationProperties): Catalogo Demo
Umbral de descuento (ConfigurationProperties): 100000.0
Nombre de la app (Value): Catalogo Demo
Vendido: 5 x Teclado mecanico
```
</Reveal>

## Explicación paso a paso

- Ambas formas de leer `catalogo.nombre-app` ("ConfigurationProperties" y "Value") dan EXACTAMENTE el mismo resultado ("Catalogo Demo") — son dos caminos distintos hacia la MISMA fuente de configuración.
- `catalogo.umbral-descuento=100000.0` (texto en el archivo) se convirtió automáticamente en `double 100000.0` en `propiedades.umbralDescuento()` — Spring hizo la conversión de tipos sin código adicional tuyo.
- Si `application.properties` tuviera `catalogo.umbral-descuento=noEsUnNumero`, la aplicación fallaría al ARRANCAR (no cuando se usa el valor más adelante) — esa es la ventaja concreta de `@ConfigurationProperties` con tipos fuertes sobre leer un `String` suelto y convertirlo manualmente.
- El `CommandLineRunner` de `demo` (con "Vendido: 5 x Teclado mecanico") también se ejecutó, confirmando que ambos beans `CommandLineRunner` coexisten sin conflicto.

## Ejemplo aplicado al proyecto

Con `application.properties`, valores de negocio como el umbral de descuento (que en la Etapa 7 vivía FIJO en el código como `UMBRAL_DESCUENTO`) podrían externalizarse — cambiar el umbral en producción ya no requeriría recompilar la aplicación, solo cambiar el archivo de configuración (o una variable de entorno, algo que Spring Boot también soporta transparentemente).

<Reveal titulo="Diagnóstico">
Falla con un error de bean no encontrado (similar a `NoSuchBeanDefinitionException`). `@ConfigurationProperties` por sí sola NO registra la clase como bean — necesita `@ConfigurationPropertiesScan` (o `@EnableConfigurationProperties(CatalogoProperties.class)` explícito) en algún lugar del contexto para que Spring la descubra y la construya. ## Resumen

- `application.properties` externaliza configuración fuera del código Java compilado.
- `@Value("${clave}")` lee un valor individual; útil para casos puntuales.
- `@ConfigurationProperties` sobre un `record` agrupa configuración relacionada con validación de tipos automática al arrancar.
- `@ConfigurationPropertiesScan` registra automáticamente las clases `@ConfigurationProperties` del proyecto como beans.

## Modelo mental final

Pensá en `application.properties` como el panel de control externo de la aplicación — cambiar un dial ahí no requiere abrir la máquina (recompilar) para ajustar su comportamiento. `@ConfigurationProperties` es como tener ese panel completo ya traducido a un formulario legible, en vez de leer cada dial suelto con `@Value` uno por uno.

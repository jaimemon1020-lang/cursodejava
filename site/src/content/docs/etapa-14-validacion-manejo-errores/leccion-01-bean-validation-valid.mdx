## Qué vas a aprender

Después de esta lección serás capaz de declarar reglas de validación directamente sobre los campos de un DTO con anotaciones de Bean Validation (`@NotBlank`, `@Positive`, `@PositiveOrZero`), y activarlas en un controlador con `@Valid` — rechazando peticiones inválidas con `400 Bad Request` ANTES de que la lógica de negocio las procese.

## Por qué necesitas aprenderlo

`POST /api/productos` (Etapa 12) acepta CUALQUIER `NuevoProductoRequest`, incluyendo un nombre vacío o un precio negativo — `Product` (Etapa 2) sí valida el precio en su constructor, pero eso lanza una excepción DESPUÉS de que la petición ya llegó hasta la lógica de negocio. Bean Validation mueve esa verificación al BORDE de la aplicación, antes de construir nada.

## Qué debes saber antes

`NuevoProductoRequest`, `@RequestBody` (Etapa 12 Lección 2) — Bean Validation se aplica directamente sobre ese mismo DTO.

## Recupera lo aprendido

¿Por qué `ProductoMapper` (bean) puede recibir dependencias inyectadas, mientras que `ProductoResponse.desde` (método estático) no podía?

<Reveal titulo="Respuesta">
Porque un bean de Spring participa del contenedor de inyección de dependencias — puede recibir cualquier otro bean por constructor. Un método estático no está gestionado por Spring, así que nunca puede recibir @Autowired.
</Reveal>

## Problema

`Product` valida el precio en su CONSTRUCTOR (Etapa 2): `if (precio < 0) throw new IllegalArgumentException(...)`. Pero para cuando esa excepción se lanza, la petición HTTP YA atravesó el controlador, el mapper, y llegó hasta la construcción del objeto de dominio — sin haber validado siquiera que el `nombre` no viniera vacío. Necesitás rechazar datos inválidos LO ANTES POSIBLE, con un mensaje claro sobre QUÉ campo falló.

## Modelo mental

```
Bean Validation: reglas declaradas DIRECTAMENTE sobre los
                  campos del DTO (@NotBlank, @Positive, ...).

@Valid:           activa esa validación en el punto de entrada
                   (el parámetro @RequestBody de un controlador).

Si falla:          Spring responde 400 Bad Request AUTOMÁTICAMENTE,
                    sin que tu código de negocio se ejecute.
```

## Explicación sencilla

Bean Validation es un estándar de Java (parte de Jakarta EE) para declarar reglas de validación como ANOTACIONES sobre los campos de una clase — `@NotBlank` significa "no puede estar vacío ni ser solo espacios", `@Positive` significa "debe ser mayor a cero". Cuando un parámetro de controlador tiene `@Valid`, Spring ejecuta TODAS esas validaciones automáticamente antes de que el método del controlador se ejecute — si alguna falla, la petición se rechaza con `400 Bad Request` sin que tu código vea siquiera el dato inválido.

## Explicación técnica

`spring-boot-starter-validation` trae Hibernate Validator (la implementación de referencia de Bean Validation) como dependencia. Las anotaciones (`jakarta.validation.constraints.*`) se colocan directamente sobre los COMPONENTES de un `record` — Java permite anotar parámetros del constructor canónico de un record, y esas anotaciones se propagan automáticamente al campo y al accessor correspondiente.

`@NotBlank` rechaza `null`, cadena vacía (`""`), o solo espacios en blanco. `@Positive` exige un número estrictamente mayor a cero. `@PositiveOrZero` exige un número mayor o igual a cero (distinto de `@Positive`: permite el cero). El parámetro `message` de cada anotación define el texto del error si la validación falla — aunque, como vas a ver en el resultado real, Spring Boot NO expone ese mensaje detallado por defecto (por razones de seguridad: no revelar detalles internos de validación a un cliente no autenticado).

`@Valid` sobre `@RequestBody NuevoProductoRequest request` activa la validación DESPUÉS de deserializar el JSON, pero ANTES de ejecutar el cuerpo del método `crear()`. Si alguna constraint falla, Spring lanza internamente `MethodArgumentNotValidException`, que el manejador de errores POR DEFECTO de Spring Boot convierte en una respuesta `400 Bad Request` — sin que tu código escriba ningún `if` de validación manual.

## Cómo funciona

1. `spring-boot-starter-validation` en `pom.xml` habilita el motor de Bean Validation.
2. `NuevoProductoRequest` declara `@NotBlank` sobre `nombre`, `@Positive` sobre `precio`, `@PositiveOrZero` sobre `stock`.
3. `crear(@Valid @RequestBody NuevoProductoRequest request)` activa esas validaciones en el punto de entrada.
4. Si la petición es válida, `crear()` se ejecuta normalmente, exactamente como en la Etapa 13.
5. Si la petición es inválida (por ejemplo, `nombre` vacío), Spring responde `400 Bad Request` AUTOMÁTICAMENTE — `crear()` nunca llega a ejecutarse, y `Product` nunca llega a construirse con datos inválidos.

## Ejemplo mínimo

**Fragmento agregado a `proyecto-maven/pom.xml`:**

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
```

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/NuevoProductoRequest.java`

```java
package com.curso.catalogo;

import jakarta.validation.constraints.NotBlank;
import jakarta.validation.constraints.Positive;
import jakarta.validation.constraints.PositiveOrZero;

public record NuevoProductoRequest(
        @NotBlank(message = "El nombre no puede estar vacio") String nombre,
        @Positive(message = "El precio debe ser mayor a cero") double precio,
        @PositiveOrZero(message = "El stock no puede ser negativo") int stock) {
}
```

**Fragmento actualizado de `proyecto-maven/src/main/java/com/curso/catalogo/ProductoController.java`:**

```java
import jakarta.validation.Valid;

@PostMapping
public ProductoResponse crear(@Valid @RequestBody NuevoProductoRequest request) {
    Product producto = mapper.toEntity(request);
    repositorio.agregar(producto);
    return mapper.toResponse(producto);
}
```

## Predice

Un `POST /api/productos` con `nombre` vacío (`""`) y `precio` negativo (`-100.0`) viola DOS constraints a la vez. ¿Qué código de estado HTTP esperás recibir?

<Reveal titulo="Resultado real (ejecutado con Spring Boot + curl)">
```
$ curl -s -o /tmp/valid.json -w "HTTP %{http_code}\n" -X POST http://localhost:8080/api/productos \
    -H "Content-Type: application/json" \
    -d '{"nombre":"Monitor 27 pulgadas","precio":180000.0,"stock":3}'
HTTP 200
{"nombre":"Monitor 27 pulgadas","precio":180000.0,"stock":3}

$ curl -s -o /tmp/invalid.json -w "HTTP %{http_code}\n" -X POST http://localhost:8080/api/productos \
    -H "Content-Type: application/json" \
    -d '{"nombre":"","precio":-100.0,"stock":5}'
HTTP 400
{"timestamp":"2026-07-19T05:30:31.265+00:00","status":400,"error":"Bad Request","path":"/api/productos"}
```
</Reveal>

## Explicación paso a paso

- El producto válido ("Monitor 27 pulgadas", precio 180000, stock 3) responde `HTTP 200` con el JSON esperado — ninguna constraint se violó.
- El producto inválido responde `HTTP 400 Bad Request` AUTOMÁTICAMENTE — `crear()` NUNCA se ejecutó, `Product` nunca se construyó con esos datos inválidos.
- El cuerpo del error `400` es GENÉRICO (`"error":"Bad Request"`), SIN mencionar cuáles campos fallaron ni los mensajes personalizados que escribiste (`"El nombre no puede estar vacio"`) — este es el comportamiento POR DEFECTO de Spring Boot, que oculta detalles de validación por seguridad.
- Esta limitación (error genérico, sin detalle) es EXACTAMENTE el problema que la próxima lección resuelve con un manejador de excepciones personalizado (`@ExceptionHandler`), devolviendo un mensaje útil para quien consume la API.

## Ejemplo aplicado al proyecto

Con Bean Validation, `NuevoProductoRequest` ahora declara sus propias reglas de negocio BÁSICAS (nombre no vacío, precio positivo, stock no negativo) de forma DECLARATIVA — sin ningún `if` manual en el controlador, y verificadas automáticamente en CADA petición, sin que puedas olvidarte de llamarlas.

## Ejercicio guiado

Agregá `@Size(max = 100)` al campo `nombre` de `NuevoProductoRequest`, limitando su longitud máxima a 100 caracteres.

<Reveal titulo="Pista 1">
`@Size` es otra anotación de `jakarta.validation.constraints`, con parámetros `min`/`max` — se combina con `@NotBlank` sobre el MISMO campo sin conflicto.
</Reveal>

<Reveal titulo="Solución">
```java
import jakarta.validation.constraints.Size;

public record NuevoProductoRequest(
        @NotBlank(message = "El nombre no puede estar vacio")
        @Size(max = 100, message = "El nombre no puede superar los 100 caracteres")
        String nombre,
        @Positive(message = "El precio debe ser mayor a cero") double precio,
        @PositiveOrZero(message = "El stock no puede ser negativo") int stock) {
}
```
</Reveal>

## Mini reto de debugging

```java
public record NuevoProductoRequest(String nombre, double precio, int stock) {
    // sin anotaciones de validacion
}

@PostMapping
public ProductoResponse crear(@Valid @RequestBody NuevoProductoRequest request) { ... }
```

`@Valid` está presente, pero `NuevoProductoRequest` no tiene ninguna anotación de constraint. ¿Qué pasa con un `nombre` vacío en este caso?

<Reveal titulo="Diagnóstico">
Se acepta sin error — `@Valid` solo EJECUTA las validaciones que el DTO DECLARA; si el DTO no tiene ninguna anotación de Bean Validation, `@Valid` no tiene nada que verificar, y cualquier dato (incluyendo un nombre vacío) pasa sin problema. `@Valid` y las anotaciones de constraint son piezas COMPLEMENTARIAS: ambas son necesarias, ninguna alcanza sola.
</Reveal>

## Resumen

- Bean Validation declara reglas (`@NotBlank`, `@Positive`, `@PositiveOrZero`, `@Size`) directamente sobre los campos de un DTO.
- `@Valid` en un parámetro `@RequestBody` activa esa validación antes de ejecutar el método del controlador.
- Por defecto, Spring Boot responde `400 Bad Request` genérico sin detalle de qué falló, por razones de seguridad.
- La validación en el DTO (borde de la API) y la validación en el constructor del dominio (`Product`) cumplen propósitos DISTINTOS y complementarios, no redundantes.

## Modelo mental final

Pensá en Bean Validation como un control de seguridad en la entrada de un edificio: revisa que cada visitante cumpla los requisitos básicos (identificación válida) ANTES de dejarlo entrar — lo que pase DENTRO del edificio (la lógica de negocio de `Product`) sigue teniendo sus propias reglas, pero ya no tiene que lidiar con visitantes que ni siquiera deberían haber cruzado la puerta.

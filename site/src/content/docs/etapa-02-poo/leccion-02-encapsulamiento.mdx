## QuГ© vas a aprender

DespuГ©s de esta lecciГіn serГЎs capaz de declarar atributos `private`, exponerlos de forma controlada con getters y setters, y validar datos dentro del setter (o del constructor) para que un objeto nunca quede en un estado invГЎlido.

## Por quГ© necesitas aprenderlo

En la lecciГіn anterior, cualquier cГіdigo podГӯa hacer `teclado.stock = -500;` directamente вҖ” un stock negativo no tiene sentido, pero Java no lo impedГӯa. Encapsulamiento es la herramienta para que eso deje de ser posible.

## ExplicaciГіn sencilla

Encapsular significa esconder los atributos (`private`) y ofrecer Гәnicamente mГ©todos pГәblicos controlados para leerlos (`getX`) o modificarlos (`setX`). El setter es el lugar exacto donde podГ©s rechazar un valor invГЎlido, antes de que llegue a guardarse.

## AnalogГӯa вҖ” El cajero automГЎtico

PensГЎ en un cajero automГЎtico:

- Vos interactuГЎs con la PANTALLA y los BOTONES (la interfaz pГәblica: los getters, setters, y mГ©todos como `venderUnidades`)
- El mecanismo INTERNO вҖ” la caja fuerte, los billetes, el contador вҖ” estГЎ SELLADO (los atributos `private`)
- No podГ©s meter la mano y sacar plata directamente (no podГ©s hacer `cajero.billetes = -500`)
- El cajero valida CADA operaciГіn: "ВҝtenГ©s saldo suficiente?" antes de entregar plata, igual que `venderUnidades` verifica `stock >= cantidad`
- Si el cajero falla, no perdГ©s plata вҖ” simplemente la operaciГіn se rechaza

Eso es encapsulamiento: exponer QUГү se puede hacer (mГ©todos pГәblicos), pero esconder CГ“MO se hace por dentro (atributos privados + validaciones).

## ExplicaciГіn tГ©cnica

`private` restringe el acceso a un atributo o mГ©todo: solo el cГіdigo dentro de la misma clase puede tocarlo directamente. Desde afuera, hace falta pasar por un mГ©todo `public`.

La convenciГіn de Java para exponer un atributo `private` es:
- **Getter**: `public Tipo getNombreAtributo()` вҖ” devuelve el valor.
- **Setter**: `public void setNombreAtributo(Tipo valor)` вҖ” lo modifica, idealmente validando antes de aceptar.

Si una clase no ofrece setter para un atributo (y ese atributo es `private` y se asigna una sola vez en el constructor), el objeto es efectivamente **inmutable**: una vez creado, su estado nunca cambia. Vas a ver inmutabilidad formalizada con `record` mГЎs adelante en el curso.

El constructor tambiГ©n es un buen lugar para validar: si el precio inicial es invГЎlido, ni siquiera deberГӯa poder crearse el objeto.

## CГіmo funciona

1. `nombre`, `precio` y `stock` pasan de no tener modificador a `private` вҖ” ya no son accesibles como `producto.precio` desde afuera de la clase.
2. El constructor valida `precio < 0` y lanza una excepciГіn si es invГЎlido, ANTES de crear el objeto.
3. `setPrecio` repite esa misma validaciГіn cada vez que alguien intenta cambiar el precio despuГ©s de creado el objeto.
4. `venderUnidades` valida que no se venda mГЎs stock del disponible, protegiendo la regla de negocio (no solo el tipo de dato).

## Ejemplo mГӯnimo

**Archivo:** `proyecto/etapa-02/Product.java`

```java
public class Product {

    private String nombre;
    private double precio;
    private int stock;

    Product(String nombre, double precio, int stock) {
        if (precio < 0) {
            throw new IllegalArgumentException("El precio no puede ser negativo");
        }
        this.nombre = nombre;
        this.precio = precio;
        this.stock = stock;
    }

    public String getNombre() {
        return this.nombre;
    }

    public double getPrecio() {
        return this.precio;
    }

    public void setPrecio(double nuevoPrecio) {
        if (nuevoPrecio < 0) {
            throw new IllegalArgumentException("El precio no puede ser negativo");
        }
        this.precio = nuevoPrecio;
    }

    public int getStock() {
        return this.stock;
    }

    public void venderUnidades(int cantidad) {
        if (cantidad > this.stock) {
            throw new IllegalStateException("No hay stock suficiente");
        }
        this.stock = this.stock - cantidad;
    }

    double calcularValorInventario() {
        return this.precio * this.stock;
    }

    void describir() {
        System.out.println(this.nombre + " | precio: " + this.precio + " | stock: " + this.stock);
    }
}
```

## Resumen

- `private` restringe el acceso a un atributo desde fuera de la clase.
- Getters exponen lectura; setters exponen escritura, idealmente validando.
- Validar en el constructor y en el setter protege la regla de negocio en un solo lugar.
- Un objeto sin setters para sus atributos (asignados una sola vez) es efectivamente inmutable.

## Invariantes de clase вҖ” el contrato invisible

Un **invariante** es una regla que DEBE cumplirse SIEMPRE para que un objeto estГ© en un estado vГЎlido. En `Product`, el invariante es: **"el precio nunca puede ser negativo"**.

```
         new Product(..., -100, ...)  вқҢ  constructor lo rechaza
         setPrecio(-50)               вқҢ  setter lo rechaza
         setPrecio(200)               вң…  pasa la validaciГіn
```

El invariante se protege en DOS lugares:
1. **Constructor**: para que el objeto NAZCA vГЎlido
2. **Setters**: para que SIGA siendo vГЎlido despuГ©s de cada modificaciГіn

Si protegГ©s ambos, tu objeto NUNCA va a estar en un estado invГЎlido, sin importar desde dГіnde lo llamen. Esto es lo que separa un diseГұo robusto de uno frГЎgil.

## Ley de Demeter вҖ” no hables con extraГұos

TambiГ©n conocido como "principio de menor conocimiento": un mГ©todo solo deberГӯa hablar con:
- Sus propios atributos (`this`)
- Sus parГЎmetros
- Objetos que Г©l mismo creГі

**No hagas esto:**
```java
// mal: encadenamiento de llamadas вҖ” "train wreck"

```

**HacГ© esto:**
```java
// bien: pedile al objeto que haga su trabajo, no le saquees sus tripas

```

Cada clase deberГӯa conocer lo mГӯnimo indispensable sobre las demГЎs. Vas a profundizar esto con la D de SOLID (Dependency Inversion) en la Etapa 7.

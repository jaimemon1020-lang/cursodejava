## Qué vas a aprender

Después de esta lección serás capaz de definir un contrato con `interface`, una base parcialmente implementada con clase abstracta, y recorrer distintas implementaciones de forma uniforme gracias al polimorfismo — sin un `if`/`switch` gigante preguntando "¿qué tipo sos?".

## Por qué necesitas aprenderlo

Vas a tener varios tipos de descuento y de envío en el sistema real. Sin polimorfismo, cada vez que agregues uno nuevo tendrías que buscar y modificar cada `if` que distingue tipos en todo el código — con polimorfismo, agregás una clase nueva y el resto del código ni se entera.

## Qué debes saber antes

Herencia (`extends`, `super`, `@Override`) y composición (lección anterior de esta etapa).

## Recupera lo aprendido

¿Cuál es la diferencia entre herencia y composición?

<Reveal titulo="Respuesta">
Herencia modela "es un" (Cliente extends Persona). Composición modela "tiene un" (Order tiene un Cliente), sin jerarquía de clases.
</Reveal>

## Problema

Necesitás aplicar distintos tipos de descuento (10% de descuento, o $5000 fijos) a un mismo total, y distintos métodos de envío (estándar, express) con costos distintos. Sin una forma común de tratarlos, terminarías con código como "si es descuento porcentual, hacé esto; si es descuento fijo, hacé esto otro" — y esa lista crece cada vez que agregás un tipo nuevo.

## Modelo mental

```
interface Descuento
      │
      │ implements
      ├──── DescuentoPorcentual
      └──── DescuentoFijo

Descuento[] descuentos = { new DescuentoPorcentual(...), new DescuentoFijo(...) };
for (Descuento d : descuentos) {
    d.aplicar(total);   ← cada uno ejecuta SU PROPIA versión, sin preguntar de qué tipo es
}
```

## Explicación sencilla

Una `interface` es un contrato: dice QUÉ métodos debe tener una clase, sin decir CÓMO los implementa. Una clase abstracta es parecida, pero puede tener código ya implementado (compartido) y dejar solo una parte como "pendiente" para las clases hijas. Polimorfismo es poder tratar objetos de distintas clases de forma uniforme, porque todos cumplen el mismo contrato.

## Analogía — El control remoto universal

Compraste un control remoto universal. El control define BOTONES (la interfaz `Descuento`):

```
CONTROL REMOTO (interface Descuento)
┌─────────────────────────────┐
│  [Aplicar descuento]        │  ← método aplicar(double total)
└─────────────────────────────┘
```

Programás el control para que funcione con:
- **TV Samsung** (`DescuentoPorcentual`): el botón "aplicar" calcula `total - 10%`
- **TV LG** (`DescuentoFijo`): el mismo botón "aplicar" calcula `total - $5000`
- **TV Sony** (`SinDescuento`): el botón "aplicar" devuelve el mismo total

Vos apretás EL MISMO BOTÓN (`descuento.aplicar(total)`) y cada TV responde a SU manera. No necesitás saber qué marca de TV tenés — el control (la interfaz) garantiza que TODAS entienden el botón "aplicar".

Eso es polimorfismo: MISMO mensaje (`aplicar`), DISTINTO comportamiento según quién lo recibe.

## Explicación técnica

**`interface`:** declara métodos sin cuerpo (solo la firma). Una clase los implementa con `implements` y debe proveer el cuerpo de cada método. Una interfaz no tiene estado (atributos de instancia) ni constructor — es puro contrato. Una clase puede implementar **varias** interfaces a la vez.

**Clase abstracta (`abstract class`):** puede mezclar métodos ya implementados (como `informar()`) con métodos abstractos (`calcularCosto()`, sin cuerpo, cada hija debe darle uno). A diferencia de una interfaz, SÍ puede tener atributos, constructor y estado. Una clase solo puede extender **una** clase abstracta (Java no permite herencia múltiple de clases).

**Polimorfismo:** cuando declarás `Descuento descuento = new DescuentoPorcentual(...)`, la variable tiene el tipo del contrato (`Descuento`), pero en tiempo de ejecución Java llama a la implementación REAL del objeto (`DescuentoPorcentual.aplicar`), no una versión genérica. Eso permite que el mismo bucle `for` funcione con cualquier implementación futura de `Descuento`, sin tocarlo.

## Cómo funciona

1. `Descuento` declara `aplicar(double total)` sin implementación.
2. `DescuentoPorcentual` y `DescuentoFijo` implementan `aplicar` cada una a su manera.
3. El array `descuentosDisponibles` guarda ambos tipos como `Descuento[]` — el `for-each` no necesita saber cuál es cuál.
4. `MetodoEnvio` (clase abstracta) ya implementa `informar()` una sola vez; cada hija (`EnvioEstandar`, `EnvioExpress`) solo agrega su propio `calcularCosto()`.

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-02/Descuento.java`

```java
public interface Descuento {
    double aplicar(double total);
}
```

**Archivo:** `proyecto/etapa-02/DescuentoPorcentual.java`

```java
public class DescuentoPorcentual implements Descuento {
    private double porcentaje;

    DescuentoPorcentual(double porcentaje) {
        this.porcentaje = porcentaje;
    }

    @Override
    public double aplicar(double total) {
        return total - (total * this.porcentaje);
    }
}
```

**Archivo:** `proyecto/etapa-02/MetodoEnvio.java`

```java
public abstract class MetodoEnvio {
    protected String nombre;

    MetodoEnvio(String nombre) {
        this.nombre = nombre;
    }

    public abstract double calcularCosto(double totalCompra);

    public void informar(double totalCompra) {
        System.out.println(this.nombre + ": $" + this.calcularCosto(totalCompra));
    }
}
```

**Archivo:** `proyecto/etapa-02/PolimorfismoDemo.java`

```java
public class PolimorfismoDemo {
    public static void main(String[] args) {
        double totalOriginal = 40000.0;

        Descuento[] descuentosDisponibles = {
            new DescuentoPorcentual(0.10),
            new DescuentoFijo(5000.0)
        };

        for (Descuento descuento : descuentosDisponibles) {
            double totalConDescuento = descuento.aplicar(totalOriginal);
            System.out.println("Total con descuento: " + totalConDescuento);
        }

        System.out.println("---");

        MetodoEnvio[] enviosDisponibles = {
            new EnvioEstandar(),
            new EnvioExpress()
        };

        for (MetodoEnvio envio : enviosDisponibles) {
            envio.informar(totalOriginal);
        }
    }
}
```

## Predice

El `for-each` recorre `Descuento[]` sin ningún `if` que distinga tipos. ¿Cómo sabe Java cuál `aplicar()` ejecutar en cada vuelta?

<Reveal titulo="Resultado real (compilado y ejecutado)">
```
Total con descuento: 36000.0
Total con descuento: 35000.0
---
Envio estandar: $1500.0
Envio express: $3500.0
```
</Reveal>

## Explicación paso a paso

- `40000.0 * 0.10 = 4000.0` restado → `36000.0` (descuento porcentual).
- `40000.0 - 5000.0 = 35000.0` (descuento fijo).
- Java decide en tiempo de ejecución cuál `aplicar()` correr, mirando el tipo REAL del objeto guardado en cada posición del array — no el tipo declarado (`Descuento`).
- `informar()` (heredado, escrito una sola vez en `MetodoEnvio`) llama a `this.calcularCosto(...)`, que ejecuta la versión de la subclase real (`EnvioEstandar` o `EnvioExpress`).

## Mini reto de debugging


¿Por qué no compila?

<Reveal titulo="Diagnóstico">

Aunque el objeto real sea un `DescuentoPorcentual`, la variable está declarada como `Descuento`, y solo podés acceder a lo que ESE tipo declara — el atributo privado de `DescuentoPorcentual` ni siquiera sería accesible desde afuera aunque lo declararas ahí.

## Resumen

- `interface` define un contrato sin estado; una clase puede implementar varias.
- `abstract class` mezcla comportamiento compartido con métodos pendientes; solo se extiende una.
- El polimorfismo permite tratar distintas implementaciones de forma uniforme, sin preguntar el tipo real.
- Agregar una implementación nueva no requiere tocar el código que ya recorre el contrato.

## Modelo mental final

Cuando pienses en polimorfismo, imaginá un enchufe universal: no te importa qué aparato conectás (descuento porcentual, fijo, sin descuento), mientras respete la forma del enchufe (el contrato `Descuento`).

## ¿Cómo sabe Java qué método ejecutar? — Dynamic Dispatch

Cuando escribís:

```java
Descuento d = new DescuentoPorcentual(0.10);
d.aplicar(40000);  // ¿cómo sabe Java cuál versión correr?
```

Java NO decide en tiempo de compilación. En tiempo de EJECUCIÓN, cada objeto guarda una referencia a lo que se llama su **vtable** (tabla de métodos virtuales):

```
Variable d (tipo Descuento)       OBJETO REAL en memoria
───────────────────────────       ────────────────────────
                                  ┌──────────────────────┐
                                  │ DescuentoPorcentual  │
                                  │                      │
d ──────────────────────────────→ │ vtable → aplicar()   │──→ total - (total * 0.10)
                                  │                      │
                                  └──────────────────────┘
```

1. El compilador verifica que `Descuento` TIENE el método `aplicar` (type check)
2. En runtime, la JVM mira el objeto REAL (`DescuentoPorcentual`)
3. Busca en su vtable la implementación CONCRETA de `aplicar`
4. Ejecuta ESA versión, no una genérica

Esto se llama **dynamic dispatch** (despacho dinámico) o **late binding** (enlace tardío). Es lo que hace posible que el mismo `for` funcione con cualquier implementación futura.

## Principio Open/Closed — abierto a extensión, cerrado a modificación

El polimorfismo habilita uno de los principios SOLID más importantes (Etapa 7): el código debe estar **abierto a extensión** pero **cerrado a modificación**.

```
SIN polimorfismo (cerrado):
  if (tipo == "PORCENTUAL") { ... }
  else if (tipo == "FIJO") { ... }
  else if (tipo == "NUEVO") { ... }  ← hay que MODIFICAR este código cada vez

CON polimorfismo (abierto):
  Descuento[] descuentos = { ... };
  for (Descuento d : descuentos) {
      d.aplicar(total);  ← este código NUNCA se modifica
  }
  // Agregar nuevo descuento = crear nueva clase. CERO cambios acá.
```

Con polimorfismo, agregar un `DescuentoEnEscala` nuevo es crear UNA clase nueva. El bucle `for` que recorre el array sigue funcionando sin tocar ni una coma. Eso es estar "cerrado a modificación" (el bucle no cambia) pero "abierto a extensión" (nuevas clases son bienvenidas).

## Qué vas a aprender

Después de esta lección serás capaz de capturar errores con `try`/`catch`/`finally` en vez de dejar que el programa se detenga, crear tu propia excepción con información específica del problema, y usar `try-with-resources` para garantizar que un recurso se cierre siempre.

## Por qué necesitas aprenderlo

Desde la Etapa 2, `Product.venderUnidades()` lanza una excepción cuando falta stock — pero cada demo hasta ahora dejaba que el programa se cortara ahí. Un sistema real necesita reaccionar al error (avisar al cliente, registrar el intento) sin detenerse por completo.

## Problema

Hasta ahora, cuando `venderUnidades(100)` fallaba por falta de stock, el programa entero se detenía con un stack trace. Un sistema real necesita capturar ese error, informarlo de forma controlada, y seguir funcionando — sin perder información específica sobre CUÁNTO stock había y CUÁNTO se pidió.

## Modelo mental

```
try {
    operación que puede fallar
} catch (TipoDeExcepcion e) {
    qué hacer si falla
} finally {
    esto se ejecuta SIEMPRE, haya fallado o no
}
```

## Explicación sencilla

`try` marca el código que puede fallar. `catch` atrapa el error si ocurre, y te da la oportunidad de reaccionar en vez de cortar el programa. `finally` se ejecuta pase lo que pase — falle o no falle — típicamente para liberar recursos.

## Explicación técnica

Java distingue dos familias de excepciones:

- **Unchecked** (heredan de `RuntimeException`): el compilador NO te obliga a capturarlas ni a declararlas con `throws`. `IllegalArgumentException`, `IllegalStateException`, `NullPointerException`, `ArrayIndexOutOfBoundsException` — todas las que ya viste desde la Etapa 2 — son unchecked. Representan errores de programación o violaciones de reglas de negocio.
- **Checked** (heredan de `Exception`, pero NO de `RuntimeException`): el compilador te OBLIGA a capturarlas o declarar `throws NombreExcepcion` en la firma del método. Representan condiciones externas esperables (un archivo que no existe, una conexión que falla) — las vas a ver en profundidad cuando trabajemos con I/O y bases de datos.

Crear tu propia excepción es extender `RuntimeException` (unchecked, la elección más común hoy en día para reglas de negocio) o `Exception` (checked). Podés agregar campos propios (como `stockDisponible`, `cantidadSolicitada`) para que quien capture la excepción tenga información específica del problema, no solo un mensaje de texto.

`try-with-resources` es una forma especial de `try` que cierra automáticamente un recurso al finalizar el bloque (haya éxito o excepción), siempre que ese recurso implemente `AutoCloseable`. Evita tener que escribir manualmente el cierre en un `finally`.

## Cómo funciona

1. `StockInsuficienteException extends RuntimeException` — es unchecked, no exige `throws` en la firma de `venderUnidades`.
2. El constructor guarda `stockDisponible` y `cantidadSolicitada` como campos propios, además del mensaje heredado.
3. `try { teclado.venderUnidades(100); } catch (StockInsuficienteException e) { ... }` captura el error específico y accede a `e.getStockDisponible()`.
4. `finally` se ejecuta tanto si la venta tuvo éxito como si falló.
5. `try (RegistroDeOperaciones registro = new RegistroDeOperaciones()) { ... }` cierra automáticamente el registro al salir del bloque, llamando a `close()` sin que lo pidas explícitamente.

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-04/StockInsuficienteException.java`

```java
public class StockInsuficienteException extends RuntimeException {

    private int stockDisponible;
    private int cantidadSolicitada;

    public StockInsuficienteException(int stockDisponible, int cantidadSolicitada) {
        super("Stock insuficiente: disponible " + stockDisponible + ", solicitado " + cantidadSolicitada);
        this.stockDisponible = stockDisponible;
        this.cantidadSolicitada = cantidadSolicitada;
    }

    public int getStockDisponible() {
        return this.stockDisponible;
    }

    public int getCantidadSolicitada() {
        return this.cantidadSolicitada;
    }
}
```

**Fragmento actualizado de `proyecto/etapa-02/Product.java`:**

```java
public void venderUnidades(int cantidad) {
    if (cantidad > this.stock) {
        throw new StockInsuficienteException(this.stock, cantidad);
    }
    this.stock = this.stock - cantidad;
}
```

**Archivo:** `proyecto/etapa-04/RegistroDeOperaciones.java`

```java
public class RegistroDeOperaciones implements AutoCloseable {

    RegistroDeOperaciones() {
        System.out.println("Registro de operaciones abierto");
    }

    public void registrar(String mensaje) {
        System.out.println("LOG: " + mensaje);
    }

    @Override
    public void close() {
        System.out.println("Registro de operaciones cerrado");
    }
}
```

**Archivo:** `proyecto/etapa-04/ExceptionsDemo.java`

```java
public class ExceptionsDemo {
    public static void main(String[] args) {
        Product teclado = new Product("Teclado mecanico", 25000.0, 15);

        try {
            teclado.venderUnidades(100);
            System.out.println("Venta realizada");
        } catch (StockInsuficienteException e) {
            System.out.println("No se pudo vender: " + e.getMessage());
            System.out.println("Stock disponible era: " + e.getStockDisponible());
        } finally {
            System.out.println("Intento de venta finalizado (esto se ejecuta siempre)");
        }

        System.out.println("Stock actual: " + teclado.getStock());

        try (RegistroDeOperaciones registro = new RegistroDeOperaciones()) {
            registro.registrar("Venta procesada para " + teclado.getNombre());
        }

        System.out.println("Programa continua normalmente");
    }
}
```

## Explicación paso a paso

- El `catch` atrapa `StockInsuficienteException`, imprime el mensaje Y accede a `e.getStockDisponible()` — información específica que un `catch` genérico de `Exception` no te daría con la misma claridad.
- `finally` se ejecuta SIEMPRE, incluso porque hubo excepción — por eso ves "Intento de venta finalizado" después del catch.
- `teclado.getStock()` sigue en `15`: la venta nunca se completó, el stock quedó intacto.
- `try (RegistroDeOperaciones registro = ...)` abre, usa, y CIERRA el registro automáticamente al final del bloque — sin un `finally` explícito escrito por vos.

## Mini reto de debugging

```java
try {
    teclado.venderUnidades(100);
} finally {
    System.out.println("Termine");
}
```

Este código no tiene ningún `catch`. ¿Qué pasa cuando `venderUnidades` lanza la excepción?

<Reveal titulo="Diagnóstico">
`finally` se ejecuta igual (imprime "Termine"), pero la excepción SIGUE propagándose hacia arriba después — un `try`/`finally` sin `catch` no atrapa el error, solo garantiza que ese bloque se ejecute antes de que el error siga su curso normal (y probablemente corte el programa si nadie más lo captura).
</Reveal>

## Resumen

- `try`/`catch` captura errores; `finally` se ejecuta siempre, con o sin error.
- Unchecked (RuntimeException) no exige manejo del compilador; checked (Exception) sí.
- Una excepción propia puede llevar campos con información específica del problema.
- `try-with-resources` cierra automáticamente cualquier `AutoCloseable` al finalizar el bloque.

## Modelo mental final

Cuando pienses en try/catch/finally, imaginá un intento con red de seguridad: `try` es el intento, `catch` es la red que te atrapa si te caés, `finally` es lo que pasa después sin importar si te caíste o no.

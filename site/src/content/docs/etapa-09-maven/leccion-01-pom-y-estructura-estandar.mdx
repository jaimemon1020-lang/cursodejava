## Qué vas a aprender

Después de esta lección serás capaz de leer y escribir un `pom.xml` básico, entender la estructura estándar de carpetas que Maven espera (`src/main/java`), y compilar/empaquetar un proyecto con `mvn` (o su versión portable, `mvnw`) en vez de armar manualmente el `classpath` con `javac` como hiciste desde la Etapa 3.

## Por qué necesitas aprenderlo

Desde la Etapa 3, cada demo se compiló combinando archivos de distintas carpetas `etapa-NN/` a mano en la línea de comandos (`javac etapa-02/Product.java etapa-04/RepositorioEnMemoria.java ...`). Eso funciona para un curso, pero es completamente inviable en un proyecto real con cientos de archivos y dependencias externas (como las que vas a necesitar para Spring, JPA, testing). Maven resuelve exactamente ese problema.

## Qué debes saber antes

Todo el código Java visto hasta la Etapa 8 — esta lección no agrega sintaxis Java nueva, cambia la HERRAMIENTA que compila y organiza ese código.

## Recupera lo aprendido

¿Qué patrón resuelve el problema de "tengo una interfaz que mi código espera, pero una librería externa tiene una API incompatible"?

<Reveal titulo="Respuesta">
Adapter: una clase que implementa la interfaz esperada, y por dentro traduce hacia/desde la API incompatible de la librería externa, sin modificar ninguna de las dos partes.
</Reveal>

## Problema

Compilar manualmente listando cada archivo (`javac etapa-02/Product.java etapa-04/StockInsuficienteException.java ...`) no escala: a medida que el proyecto crece, esa lista de archivos crece, y agregar una DEPENDENCIA EXTERNA (una librería de un tercero, como las que vas a necesitar para JSON, testing, o bases de datos) es directamente imposible sin descargarla y gestionar su classpath a mano.

## Modelo mental

```
Sin Maven:  vos armás manualmente la lista de archivos y el classpath
            en cada comando javac/java.

Con Maven:  un archivo (pom.xml) describe QUÉ es el proyecto y QUÉ
            necesita — Maven se encarga de compilar, empaquetar,
            y descargar dependencias siguiendo una estructura
            de carpetas ESTÁNDAR que ya no hay que explicarle.
```

## Explicación sencilla

Maven es una herramienta de build (construcción) y gestión de dependencias. En vez de que vos le digas a `javac` exactamente qué archivos compilar y en qué classpath, Maven asume una estructura de carpetas ESTÁNDAR (`src/main/java` para tu código, `src/test/java` para tests) y un archivo de configuración (`pom.xml`) que describe el proyecto: su nombre, su versión, y qué librerías externas necesita. Con eso, `mvn compile` o `mvn package` hacen todo el trabajo que antes hacías a mano.

## Explicación técnica

`pom.xml` (Project Object Model) es el archivo central de un proyecto Maven. Los datos mínimos:

- **`groupId`**: identifica quién/qué organización posee el proyecto (convención: dominio invertido, como `com.curso`).
- **`artifactId`**: el nombre del proyecto en sí (`catalogo-app`).
- **`version`**: la versión actual — `1.0-SNAPSHOT` indica que es una versión EN DESARROLLO, no una release estable.
- **`packaging`**: qué tipo de artefacto produce el build — `jar` (el más común para una app standalone o librería).

La **estructura estándar de directorios** (Maven la llama "convention over configuration" — convención sobre configuración) espera el código de producción en `src/main/java`, organizado en subcarpetas que reflejan el PAQUETE Java (`com.curso.catalogo` vive en `src/main/java/com/curso/catalogo/`). Maven NUNCA necesita que le digas dónde está cada archivo — simplemente los encuentra siguiendo esa convención.

Antes de Maven, todo el código del curso vivía en el **paquete por defecto** (sin declaración `package`), lo cual funciona para un curso pero es mala práctica en cualquier proyecto real: sin paquetes, es imposible tener dos clases con el mismo nombre en módulos distintos, y el código no se puede organizar jerárquicamente. Migrar a Maven obligó a agregarle `package com.curso.catalogo;` a cada clase — y con paquetes con NOMBRE, los modificadores de acceso (como el constructor `package-private` que tenía `Product` desde la Etapa 2) empiezan a importar de verdad: por eso el constructor de `Product` pasó a ser `public` en esta migración.

**Maven Wrapper** (`mvnw`/`mvnw.cmd`) es un script que descarga automáticamente la versión EXACTA de Maven que el proyecto necesita, sin que quien clona el repositorio tenga que instalar Maven manualmente ni preocuparse por versiones incompatibles — es el equivalente, para Maven, a lo que sería tener Java empaquetado junto al proyecto.

## Cómo funciona

1. `pom.xml` en la raíz de `proyecto-maven/` declara `groupId`, `artifactId`, `version`, y la versión de Java a usar (`maven.compiler.release`).
2. `src/main/java/com/curso/catalogo/Product.java` (y las demás clases) viven en la ruta que refleja EXACTAMENTE su paquete `com.curso.catalogo`.
3. `./mvnw compile` compila todo el código de `src/main/java`, generando las clases en `target/classes` — sin que hiciera falta listar archivos manualmente.
4. `./mvnw package` compila Y empaqueta todo en un `.jar` dentro de `target/`, listo para distribuir.
5. `java -cp target/classes com.curso.catalogo.App` ejecuta la clase con el paquete completo — a diferencia de antes, el nombre de clase ahora incluye su paquete.

## Ejemplo mínimo

**Archivo:** `proyecto-maven/pom.xml`

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.curso</groupId>
    <artifactId>catalogo-app</artifactId>
    <version>1.0-SNAPSHOT</version>
    <packaging>jar</packaging>

    <properties>
        <maven.compiler.release>17</maven.compiler.release>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    </properties>
</project>
```

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/App.java`

```java
package com.curso.catalogo;

public class App {
    public static void main(String[] args) {
        Product teclado = new Product("Teclado mecanico", 25000.0, 15);
        teclado.describir();

        teclado.venderUnidades(5);
        System.out.println("Stock restante: " + teclado.getStock());
        System.out.println("Valor de inventario: " + teclado.calcularValorInventario());
    }
}
```

## Predice

Después de `./mvnw compile` y ejecutar `App`, sabiendo que el teclado arranca con 15 unidades y se venden 5 (`venderUnidades(5)`), ¿qué stock restante y qué valor de inventario esperás?

<Reveal titulo="Resultado real (compilado y ejecutado con Maven)">
```
$ ./mvnw -q clean package
$ java -cp target/classes com.curso.catalogo.App
Teclado mecanico | precio: 25000.0 | stock: 15
Stock restante: 10
Valor de inventario: 250000.0
```
</Reveal>

## Explicación paso a paso

- `./mvnw -q clean package` limpia el `target/` anterior, compila TODO `src/main/java`, y empaqueta el resultado en un `.jar` — sin que se haya escrito ningún nombre de archivo en el comando.
- `describir()` imprime el estado inicial: stock 15.
- `venderUnidades(5)` reduce el stock a 10, exactamente igual que en la Etapa 2 — el comportamiento de `Product` no cambió, solo CÓMO se compila.
- `calcularValorInventario()` da `10 * 25000 = 250000.0` — la lógica de negocio es idéntica a la de `proyecto/etapa-02/Product.java`, migrada sin cambios de comportamiento.
- La clase se ejecuta como `com.curso.catalogo.App`, no solo `App` — el paquete es ahora parte de su identidad.

## Ejercicio guiado

Migrá `StockInsuficienteException` (ya migrada como referencia) y agregale una clase nueva `Cliente` a `com.curso.catalogo`, con los mismos campos que la versión de la Etapa 2 (`nombre`, `email`, `direccion`), recordando declarar el `package` correcto.

<Reveal titulo="Pista 1">
El archivo debe vivir en `src/main/java/com/curso/catalogo/Cliente.java`, empezando con `package com.curso.catalogo;` — la ruta de carpetas DEBE coincidir exactamente con el paquete declarado.
</Reveal>

<Reveal titulo="Solución">
```java
package com.curso.catalogo;

public class Cliente {
    private String nombre;
    private String email;
    private String direccion;

    public Cliente(String nombre, String email, String direccion) {
        this.nombre = nombre;
        this.email = email;
        this.direccion = direccion;
    }

    public String getNombre() {
        return this.nombre;
    }

    public String getEmail() {
        return this.email;
    }

    public String getDireccion() {
        return this.direccion;
    }
}
```
</Reveal>

## Resumen

- `pom.xml` describe el proyecto (`groupId`, `artifactId`, `version`) y sus propiedades de build.
- La estructura estándar `src/main/java/<paquete>` reemplaza la necesidad de listar archivos manualmente.
- `./mvnw compile`/`package` compilan y empaquetan sin que definas el classpath a mano.
- Migrar a paquetes con nombre real hace que los modificadores de acceso (`public`, `package-private`) empiecen a importar de verdad.

## Modelo mental final

Pensá en el proyecto sin Maven como una casa sin planos, donde vos tenés que recordar dónde está cada cosa; y en un proyecto Maven como una casa con planos estándar (`src/main/java`) que cualquier otra persona (o herramienta) puede leer sin que se los expliques — el `pom.xml` es la ficha técnica de la casa completa.

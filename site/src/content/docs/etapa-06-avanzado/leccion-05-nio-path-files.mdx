## Qué vas a aprender

Después de esta lección serás capaz de escribir el contenido de una colección a un archivo de texto, leerlo de vuelta, verificar si existe, y borrarlo — usando `Path` y `Files` de `java.nio.file`, la API moderna de Java para trabajar con el sistema de archivos.

## Por qué necesitas aprenderlo

Hasta ahora, todo el "catálogo" de productos vive solo en memoria (`RepositorioEnMemoria`) — se pierde cada vez que el programa termina. Antes de llegar a bases de datos (Etapa 15), necesitás la herramienta más básica de persistencia: escribir y leer archivos del sistema operativo.

## Qué debes saber antes

`Stream`, `collect(Collectors.joining(...))` (Etapa 5) — se usa para armar el contenido del archivo a partir de una colección.

## Problema

Necesitás guardar el catálogo de productos en un archivo de texto (una línea por producto), confirmar que el archivo se creó, leer su contenido de vuelta línea por línea, y eventualmente borrarlo. Todo esto requiere interactuar con el sistema de archivos del sistema operativo, algo que ningún código visto hasta ahora hizo.

## Modelo mental

```
Path:   una RUTA — la dirección de un archivo o carpeta, sin abrirlo todavía.
Files:  la clase con las OPERACIONES sobre esa ruta: escribir, leer,
        verificar existencia, borrar, listar contenido de una carpeta.
```

## Explicación sencilla

`Path` representa una ubicación en el sistema de archivos — como una dirección postal, no la casa en sí. `Files` es la clase que hace el trabajo real usando esa dirección: `Files.writeString(ruta, texto)` escribe, `Files.readAllLines(ruta)` lee, `Files.exists(ruta)` pregunta si existe algo ahí, `Files.delete(ruta)` lo borra.

## Explicación técnica

`java.nio.file` (NIO, "New I/O", desde Java 7) reemplaza a la vieja `java.io.File`, que mezclaba la ruta y las operaciones en una sola clase con una API confusa (por ejemplo, `File.delete()` devolvía `false` silenciosamente si fallaba, en vez de lanzar una excepción clara). NIO separa la responsabilidad: `Path` es solo la ruta (inmutable, se puede componer con `.resolve()`), `Files` son operaciones ESTÁTICAS que reciben un `Path`.

`Path.of(carpeta, nombreArchivo)` construye una ruta combinando partes, de forma segura para cualquier sistema operativo (Windows usa `\`, Linux/Mac usan `/` — `Path` lo maneja automáticamente). `System.getProperty("java.io.tmpdir")` devuelve la carpeta temporal del sistema operativo — útil para archivos de prueba que no necesitás conservar.

`Files.writeString(ruta, contenido)` crea el archivo (o lo sobrescribe si ya existe) con ese texto. `Files.readAllLines(ruta)` devuelve una `List<String>`, una entrada por línea. Muchas operaciones de `Files` pueden lanzar `IOException` (checked, Etapa 4 Lección 6) porque el sistema de archivos es un recurso EXTERNO que puede fallar por razones fuera del control del programa (permisos, disco lleno, archivo bloqueado) — por eso `main` en el ejemplo declara `throws IOException`.

## Cómo funciona

1. `Path archivo = Path.of(System.getProperty("java.io.tmpdir"), "catalogo.txt")` arma la ruta completa al archivo, en la carpeta temporal del sistema.
2. `repositorio.listarTodos().stream().map(...).collect(Collectors.joining(System.lineSeparator()))` arma el contenido completo como un solo `String`, con un producto por línea (reutilizando Streams de la Etapa 5).
3. `Files.writeString(archivo, contenido)` crea el archivo físicamente en el sistema de archivos.
4. `Files.exists(archivo)` confirma que el archivo realmente se creó; `Files.size(archivo)` da su tamaño en bytes.
5. `Files.readAllLines(archivo)` lee el archivo de vuelta como una lista de líneas — cada elemento es una línea del `String` que se escribió.
6. `Files.delete(archivo)` borra el archivo; `Files.exists(archivo)` después confirma que ya no existe.

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-06/NioDemo.java`

```java
import java.io.IOException;
import java.nio.file.Files;
import java.nio.file.Path;
import java.util.List;
import java.util.stream.Collectors;

public class NioDemo {
    public static void main(String[] args) throws IOException {
        RepositorioEnMemoria<Product> repositorio = new RepositorioEnMemoria<>();
        repositorio.agregar(new Product("Teclado mecanico", 25000.0, 15));
        repositorio.agregar(new Product("Mouse inalambrico", 8000.0, 40));

        Path archivo = Path.of(System.getProperty("java.io.tmpdir"), "catalogo.txt");

        String contenido = repositorio.listarTodos().stream()
                .map(p -> p.getNombre() + ";" + p.getPrecio() + ";" + p.getStock())
                .collect(Collectors.joining(System.lineSeparator()));

        Files.writeString(archivo, contenido);
        System.out.println("Archivo existe: " + Files.exists(archivo));
        System.out.println("Tamano en bytes: " + Files.size(archivo));

        List<String> lineas = Files.readAllLines(archivo);
        System.out.println("Lineas leidas: " + lineas.size());
        for (String linea : lineas) {
            System.out.println("  " + linea);
        }

        Files.delete(archivo);
        System.out.println("Archivo existe despues de borrar: " + Files.exists(archivo));
    }
}
```

## Predice

Se escriben 2 productos, uno por línea, separados por `System.lineSeparator()`. ¿Cuántas líneas va a leer `Files.readAllLines`?

<Reveal titulo="Resultado real (compilado y ejecutado)">
```
Archivo existe: true
Tamano en bytes: 55
Lineas leidas: 2
  Teclado mecanico;25000.0;15
  Mouse inalambrico;8000.0;40
Archivo existe despues de borrar: false
```
</Reveal>

## Explicación paso a paso

- "Archivo existe: true" confirma que `writeString` creó el archivo físicamente en el disco, no solo en memoria.
- "Tamano en bytes: 55" es el peso real del texto escrito — confirma que hay contenido, no un archivo vacío.
- "Lineas leidas: 2" coincide exactamente con los 2 productos agregados al repositorio — cada `map` generó una línea, unidas por `Collectors.joining(System.lineSeparator())`.
- Cada línea impresa muestra el formato `nombre;precio;stock` tal como se armó con el `Stream`.
- "Archivo existe despues de borrar: false" confirma que `Files.delete` realmente eliminó el archivo del sistema, no solo lo "olvidó" en el programa.

## Ejemplo aplicado al proyecto

Con `Files.writeString`/`readAllLines`, el catálogo de productos puede sobrevivir más allá de una sola ejecución del programa — un primer paso hacia persistencia real, antes de la Etapa 15 (bases de datos), donde vas a reemplazar archivos de texto por una base de datos de verdad.

## Ejercicio guiado

Escribí un método que reciba un `Path` y devuelva `true` si el archivo existe Y no está vacío (`Files.size(ruta) > 0`), sin lanzar excepción si el archivo no existe.

<Reveal titulo="Pista 1">
`Files.exists(ruta)` primero, como cortocircuito — si no existe, ni siquiera intentes llamar a `Files.size`, que fallaría con `NoSuchFileException` sobre un archivo inexistente.
</Reveal>

<Reveal titulo="Solución">
```java
static boolean tieneContenido(Path ruta) throws IOException {
    return Files.exists(ruta) && Files.size(ruta) > 0;
}
```
</Reveal>

## Mini reto de debugging

```java
Path archivo = Path.of(System.getProperty("java.io.tmpdir"), "catalogo.txt");
Files.delete(archivo);
Files.delete(archivo);
```

¿Qué pasa en la segunda llamada a `Files.delete`?

<Reveal titulo="Diagnóstico">
Lanza `NoSuchFileException`. A diferencia de la vieja `File.delete()` (que devolvía silenciosamente `false` si el archivo no existía), `Files.delete()` de NIO lanza una excepción explícita si el archivo ya no está ahí — justamente la mejora de diseño que NIO trajo: fallar de forma clara en vez de fallar en silencio.
</Reveal>

## Resumen

- `Path` representa una ruta; `Files` contiene las operaciones estáticas que actúan sobre esa ruta.
- `Files.writeString`/`readAllLines` cubren el caso más común: texto plano, todo de una vez.
- `Files.exists` es la única operación "segura" que nunca lanza excepción por ausencia de archivo — las demás sí.
- NIO reemplaza a la vieja `java.io.File` con una API que separa responsabilidades y falla explícitamente en vez de en silencio.

## Modelo mental final

Pensá en `Path` como escribir una dirección en un papel — no implica que la casa exista ni que puedas entrar — y en `Files` como el cartero que realmente va hasta esa dirección a entregar, retirar, o confirmar si hay alguien ahí.

## Conexión

`Collectors.joining` (Etapa 5) arma el contenido a persistir; `IOException` (Etapa 4 Lección 6, checked) es la excepción que casi todas las operaciones de `Files` pueden lanzar, porque el sistema de archivos es un recurso externo que puede fallar de formas que el programa no controla.

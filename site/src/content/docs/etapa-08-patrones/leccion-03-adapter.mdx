## Qué vas a aprender

Después de esta lección serás capaz de usar el patrón Adapter para conectar una clase con una API incompatible (por ejemplo, un método que recibe `String` en vez de `double`) con la interfaz que tu código ya espera, sin modificar NI la clase incompatible NI el código que la necesita usar.

## Por qué necesitas aprenderlo

Cuando integres librerías de terceros (más adelante, clientes HTTP, drivers de bases de datos, SDKs de pago), casi nunca su API va a coincidir EXACTAMENTE con la interfaz que tu código de negocio espera. Reescribir la librería externa no es una opción; reescribir tu código de negocio para acomodarse a cada librería tampoco escala. Adapter resuelve exactamente esa fricción.

## Qué debes saber antes

`Descuento`/`Strategy` (Etapa 8 Lección 2) — Adapter también usa una interfaz común, pero con un propósito distinto: no elegir entre estrategias, sino ENVOLVER algo incompatible.

## Recupera lo aprendido

¿Por qué `DescuentoFactory.crearParaMontoTotal` debe devolver `Descuento` (la interfaz) y no `DescuentoPorcentual` (la implementación concreta)?

<Reveal titulo="Respuesta">
Porque si devolviera el tipo concreto, cualquier código que llame a la Factory quedaría atado a esa implementación específica — rompiendo el propósito del patrón, que es que el código cliente programe contra la abstracción, no contra un tipo concreto.
</Reveal>

## Problema

Tu sistema espera trabajar con una interfaz `ProcesadorPago` con un método `procesar(double monto) -> boolean`. Pero el proveedor de pagos EXTERNO (que no podés modificar, porque es una librería de terceros) tiene un método completamente distinto: `cobrar(String montoEnTexto) -> String`, que recibe el monto como texto y devuelve `"OK"` o `"ERROR"` como texto. ¿Cómo hacés que tu sistema use ese proveedor externo SIN reescribir tu interfaz ni la librería externa?

## Modelo mental

```
Tu sistema espera:     ProcesadorPago.procesar(double) -> boolean

Lo que tenés:          ProveedorPagoExterno.cobrar(String) -> String
                        (incompatible, no lo podés modificar)

Adapter:                una clase que IMPLEMENTA lo que tu sistema
                         espera, y por DENTRO traduce hacia/desde
                         lo que la clase externa realmente ofrece.
```

## Explicación sencilla

Un Adapter es un "traductor": implementa la interfaz que tu código espera, pero por dentro delega el trabajo real a una clase incompatible, convirtiendo los tipos de entrada y salida en el camino. Ni la interfaz que esperás ni la clase externa se modifican — el Adapter es una pieza NUEVA que se pone en el medio.

## Explicación técnica

`AdaptadorPagoExterno implements ProcesadorPago` — cumple el CONTRATO que tu sistema espera (`procesar(double) -> boolean`). Por dentro, recibe (por Dependency Injection, Etapa 8 Lección 1) una instancia de `ProveedorPagoExterno`, y en su implementación de `procesar` hace la TRADUCCIÓN: convierte el `double` recibido a `String` (`String.valueOf(monto)`), llama al método real de la librería externa (`cobrar`), y convierte el `String` de respuesta ("OK"/"ERROR") de vuelta a `boolean`.

Esto es distinto de Strategy: Strategy elige ENTRE VARIAS implementaciones que YA cumplen la misma interfaz desde el diseño. Adapter existe porque hay UNA implementación que NO cumple la interfaz que necesitás, y la envolvés para que sí la cumpla — sin tocar ni la interfaz ni la implementación original.

## Cómo funciona

1. `ProcesadorPago` es la interfaz que tu código de negocio espera — nunca sabe que existe `ProveedorPagoExterno`.
2. `ProveedorPagoExterno` simula una librería externa con una API incompatible (`String` en vez de `double`, `String` en vez de `boolean`) — no se puede modificar (imaginá que viene de un `.jar` de terceros).
3. `AdaptadorPagoExterno` implementa `ProcesadorPago`, recibe un `ProveedorPagoExterno` por constructor (DI), y en `procesar` hace la conversión de tipos en ambas direcciones.
4. El código cliente (`AdapterDemo`) solo interactúa con `ProcesadorPago` — nunca menciona `ProveedorPagoExterno` ni sus tipos incompatibles directamente.

## Ejemplo mínimo

**Archivo:** `proyecto/etapa-08/ProcesadorPago.java` (lo que tu sistema espera)

```java
public interface ProcesadorPago {
    boolean procesar(double monto);
}
```

**Archivo:** `proyecto/etapa-08/ProveedorPagoExterno.java` (simula una librería externa incompatible)

```java
public class ProveedorPagoExterno {

    public String cobrar(String montoEnTexto) {
        double monto = Double.parseDouble(montoEnTexto);
        return monto > 0 ? "OK" : "ERROR";
    }
}
```

**Archivo:** `proyecto/etapa-08/AdaptadorPagoExterno.java` (el traductor)

```java
public class AdaptadorPagoExterno implements ProcesadorPago {

    private final ProveedorPagoExterno proveedor;

    public AdaptadorPagoExterno(ProveedorPagoExterno proveedor) {
        this.proveedor = proveedor;
    }

    @Override
    public boolean procesar(double monto) {
        String resultado = proveedor.cobrar(String.valueOf(monto));
        return "OK".equals(resultado);
    }
}
```

**Archivo:** `proyecto/etapa-08/AdapterDemo.java`

```java
public class AdapterDemo {
    public static void main(String[] args) {
        ProcesadorPago procesador = new AdaptadorPagoExterno(new ProveedorPagoExterno());

        System.out.println("Pago de 25000.0: " + procesador.procesar(25000.0));
        System.out.println("Pago de -100.0: " + procesador.procesar(-100.0));
    }
}
```

## Predice

`ProveedorPagoExterno.cobrar` devuelve `"OK"` si el monto (como número, después de convertir el texto) es mayor a 0, `"ERROR"` en caso contrario. ¿Qué `boolean` esperás de `procesar(25000.0)` y de `procesar(-100.0)`?

<Reveal titulo="Resultado real (compilado y ejecutado)">
```
Pago de 25000.0: true
Pago de -100.0: false
```
</Reveal>

## Explicación paso a paso

- `procesar(25000.0)` convierte `25000.0` a `"25000.0"`, se lo pasa a `cobrar`, que lo vuelve a convertir a `double` (25000.0), confirma que es mayor a 0, y devuelve `"OK"` — el Adapter traduce eso de vuelta a `true`.
- `procesar(-100.0)` sigue el mismo camino, pero `cobrar` devuelve `"ERROR"` porque -100.0 no es mayor a 0 — el Adapter traduce eso a `false`.
- El código de `AdapterDemo` NUNCA vio un `String` de ida o vuelta — solo trabajó con `double` y `boolean`, exactamente el contrato de `ProcesadorPago`. Toda la conversión de tipos quedó encapsulada dentro del Adapter.

## Ejemplo aplicado al proyecto

Este es el patrón que vas a usar constantemente al integrar librerías reales: un cliente HTTP de terceros, un SDK de pago, un driver de base de datos — casi ninguno va a coincidir exactamente con la interfaz que tu dominio de negocio necesita, y Adapter es la forma estándar de conectarlos sin ensuciar tu código de negocio con los detalles de la librería externa.

## Ejercicio guiado

Escribí un SEGUNDO proveedor externo simulado, `ProveedorPagoLegado`, con un método `ejecutarCobro(int montoEnCentavos) -> int` (devuelve `1` para éxito, `0` para fallo, y recibe el monto en centavos como `int`), y su Adapter correspondiente que implemente `ProcesadorPago`.

<Reveal titulo="Pista 1">
El monto en `double` necesita convertirse a centavos (`(int)(monto * 100)`) antes de llamar a `ejecutarCobro`, y el resultado `int` (1 o 0) necesita convertirse a `boolean`.
</Reveal>

<Reveal titulo="Solución">
```java
public class ProveedorPagoLegado {
    public int ejecutarCobro(int montoEnCentavos) {
        return montoEnCentavos > 0 ? 1 : 0;
    }
}

public class AdaptadorPagoLegado implements ProcesadorPago {
    private final ProveedorPagoLegado proveedor;

    public AdaptadorPagoLegado(ProveedorPagoLegado proveedor) {
        this.proveedor = proveedor;
    }

    @Override
    public boolean procesar(double monto) {
        int centavos = (int) (monto * 100);
        return proveedor.ejecutarCobro(centavos) == 1;
    }
}
```
</Reveal>

## Mini reto de debugging

```java
public class AdaptadorPagoExterno implements ProcesadorPago {
    private ProveedorPagoExterno proveedor = new ProveedorPagoExterno();

    @Override
    public boolean procesar(double monto) {
        return proveedor.cobrar(String.valueOf(monto)).equals("OK");
    }
}

AdaptadorPagoExterno adaptador1 = new AdaptadorPagoExterno();
AdaptadorPagoExterno adaptador2 = new AdaptadorPagoExterno();
```

¿Qué principio (visto en Etapa 8 Lección 1) está siendo violado acá, y por qué sería un problema si quisieras testear `AdaptadorPagoExterno` con un `ProveedorPagoExterno` de prueba?

<Reveal titulo="Diagnóstico">
Viola Dependency Injection: `ProveedorPagoExterno` se crea INTERNAMENTE con `new` en vez de recibirse por constructor. Esto significa que NUNCA vas a poder testear `AdaptadorPagoExterno` con un proveedor de prueba/simulado — siempre va a usar la instancia real creada internamente, exactamente el mismo problema que la Etapa 8 Lección 1 explicó con `ServicioVentas`.
</Reveal>

## Resumen

- Adapter conecta una interfaz incompatible con la que tu código espera, sin modificar ninguna de las dos.
- El Adapter implementa la interfaz esperada, y por dentro traduce tipos y llamadas hacia la clase incompatible.
- A diferencia de Strategy (elegir entre opciones ya compatibles), Adapter hace compatible algo que no lo era por diseño.
- El código de negocio nunca debería depender directamente de la clase externa — siempre a través del Adapter.

## Modelo mental final

Pensá en un Adapter como un enchufe universal de viaje: tu dispositivo (el código de negocio) espera un tipo de enchufe específico, el tomacorriente del país (la librería externa) es distinto, y el adaptador físico traduce entre ambos sin que vos tengas que modificar ni tu dispositivo ni la instalación eléctrica del hotel.

## Conexión

Esta lección cierra la Etapa 8: DI y Repository (Lección 1) dieron la base de "recibir dependencias, no crearlas"; Factory y Strategy (Lección 2) dieron "elegir entre implementaciones compatibles"; Adapter (esta lección) completa el cuadro con "hacer compatible algo que no lo era". Los tres patrones combinados son la base conceptual de cómo Spring va a conectar tus clases automáticamente a partir de la Etapa 10.

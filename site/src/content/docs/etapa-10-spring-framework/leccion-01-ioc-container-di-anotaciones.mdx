## Qué vas a aprender

Después de esta lección serás capaz de marcar tus clases con `@Component`/`@Service`/`@Repository` para que Spring las gestione, usar `@Autowired` para que el framework inyecte dependencias automáticamente por vos, y arrancar un `ApplicationContext` que crea y conecta todos esos objetos sin que escribas un solo `new` manual.

## Por qué necesitas aprenderlo

En la Etapa 8, `new ServicioVentas(repositorio)` requería que VOS crearas manualmente el `RepositorioEnMemoria` y se lo pasaras al constructor. Eso funciona con dos o tres clases — en una aplicación real con decenas de servicios interconectados, cablear todo eso a mano se vuelve inviable. Spring automatiza EXACTAMENTE ese cableado.

## Qué debes saber antes

Dependency Injection manual por constructor (Etapa 8 Lección 1), dependencias en `pom.xml` (Etapa 9 Lección 2).

## Recupera lo aprendido

</Reveal>

## Problema

`ServicioVentas` (Etapa 8) necesita un `RepositorioEnMemoria`, y en algún lugar del código TENÍAS que escribir `new RepositorioEnMemoria<>()` y `new ServicioVentas(repositorio)` a mano, en el orden correcto. A medida que agregás más clases con más dependencias entre sí, ese cableado manual (quién crea qué, en qué orden, pasándole qué a quién) se vuelve una fuente constante de errores y código repetitivo.

## Modelo mental

```
Sin Spring (Etapa 8):  VOS creás cada objeto con new,
                        en el orden correcto, y se lo pasás
                        manualmente a quien lo necesita.

Con Spring:             Marcás las clases con anotaciones.
                        El CONTENEDOR (ApplicationContext) las
                        descubre, las crea, y las conecta
                        automáticamente entre sí.
```

## Explicación sencilla

**Inversión de Control (IoC)** significa que, en vez de que TU código controle cuándo y cómo se crean los objetos (como hacías con `new` en la Etapa 8), le CEDÉS ese control a un contenedor externo (el `ApplicationContext` de Spring). Vos solo declarás QUÉ clases existen (con anotaciones) y QUÉ necesita cada una (con `@Autowired`), y Spring se encarga de crearlas, conectarlas, y dártelas cuando las pedís.

## Explicación técnica

`@Component` (y sus especializaciones semánticas `@Service` para lógica de negocio, `@Repository` para acceso a datos) marcan una clase como un **bean** — un objeto que Spring va a gestionar. `@ComponentScan(basePackages = "com.curso.catalogo")` le dice al contenedor DÓNDE buscar esas clases anotadas.

`@Autowired` sobre un constructor le dice a Spring: "cuando crees este bean, resolvé automáticamente sus parámetros buscando OTROS beans del tipo correcto, y pasámelos". Spring resuelve `RepositorioProductos` como parámetro de `ServicioVentas` porque `RepositorioProductos` también está anotada como bean (`@Repository`) — sin que vos escribas ningún `new RepositorioProductos()` en ningún lado.

`ApplicationContext` es el contenedor IoC en sí — `new AnnotationConfigApplicationContext(AppConfig.class)` lo arranca, escanea el paquete configurado, crea TODOS los beans encontrados (resolviendo sus dependencias en el orden correcto automáticamente), y queda listo para entregártelos con `contexto.getBean(TipoDeClase.class)`.

Por defecto, cada bean de Spring es un **singleton**: existe UNA sola instancia compartida en todo el contenedor — pedirle `ServicioVentas` al contexto dos veces te devuelve la MISMA instancia, no dos objetos distintos.

## Cómo funciona

1. `@Repository` sobre `RepositorioProductos` la marca como bean gestionado por Spring.
2. `@Service` sobre `ServicioVentas`, junto con `@Autowired` en su constructor, le dice a Spring que este bean necesita un `RepositorioProductos` para construirse.
3. `@Configuration` + `@ComponentScan(basePackages = "com.curso.catalogo")` en `AppConfig` le dice al contenedor dónde buscar clases anotadas.
4. `new AnnotationConfigApplicationContext(AppConfig.class)` arranca el contenedor: escanea el paquete, encuentra `RepositorioProductos` y `ServicioVentas`, y los crea EN EL ORDEN CORRECTO (primero el repositorio, porque el servicio lo necesita).
5. `contexto.getBean(ServicioVentas.class)` devuelve la instancia YA CONSTRUIDA y CONECTADA — el `RepositorioProductos` ya está inyectado adentro, sin que vos hicieras nada manualmente.

## Ejemplo mínimo

**Fragmento agregado a `proyecto-maven/pom.xml`:**

```xml
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-context</artifactId>
    <version>6.1.21</version>
</dependency>
```

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/RepositorioProductos.java`

```java
package com.curso.catalogo;

import java.util.ArrayList;
import java.util.List;
import java.util.Optional;
import java.util.function.Predicate;

import org.springframework.stereotype.Repository;

@Repository
public class RepositorioProductos {

    private final List<Product> productos = new ArrayList<>();

    public void agregar(Product producto) {
        this.productos.add(producto);
    }

    public List<Product> listarTodos() {
        return this.productos;
    }

    public Optional<Product> buscar(Predicate<Product> criterio) {
        return this.productos.stream().filter(criterio).findFirst();
    }
}
```

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/ServicioVentas.java`

```java
package com.curso.catalogo;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

@Service
public class ServicioVentas {

    private final RepositorioProductos repositorio;

    @Autowired
    public ServicioVentas(RepositorioProductos repositorio) {
        this.repositorio = repositorio;
    }

    public void venderProducto(String nombre, int cantidad) {
        Product producto = repositorio.buscar(p -> p.getNombre().equals(nombre))
                .orElseThrow(() -> new IllegalArgumentException("Producto no encontrado: " + nombre));
        producto.venderUnidades(cantidad);
        System.out.println("Vendido: " + cantidad + " x " + producto.getNombre());
    }
}
```

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/AppConfig.java`

```java
package com.curso.catalogo;

import org.springframework.context.annotation.ComponentScan;
import org.springframework.context.annotation.Configuration;

@Configuration
@ComponentScan(basePackages = "com.curso.catalogo")
public class AppConfig {
}
```

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/SpringDemo.java`

```java
package com.curso.catalogo;

import org.springframework.context.ApplicationContext;
import org.springframework.context.annotation.AnnotationConfigApplicationContext;

public class SpringDemo {
    public static void main(String[] args) {
        ApplicationContext contexto = new AnnotationConfigApplicationContext(AppConfig.class);

        RepositorioProductos repositorio = contexto.getBean(RepositorioProductos.class);
        repositorio.agregar(new Product("Teclado mecanico", 25000.0, 15));

        ServicioVentas servicio = contexto.getBean(ServicioVentas.class);
        servicio.venderProducto("Teclado mecanico", 5);

        ServicioVentas otraReferencia = contexto.getBean(ServicioVentas.class);
        System.out.println("Es el mismo bean (singleton): " + (servicio == otraReferencia));
    }
}
```

## Predice

`ServicioVentas` nunca se construyó con `new` en ningún lugar del código — Spring lo creó automáticamente. ¿`contexto.getBean(ServicioVentas.class)` llamado dos veces va a devolver el MISMO objeto o dos objetos distintos?

<Reveal titulo="Resultado real (ejecutado con Maven)">
```
$ ./mvnw -q compile exec:java -Dexec.mainClass=com.curso.catalogo.SpringDemo
Vendido: 5 x Teclado mecanico
Es el mismo bean (singleton): true
```
</Reveal>

## Explicación paso a paso

- "Vendido: 5 x Teclado mecanico" confirma que `ServicioVentas.venderProducto` funcionó, usando un `RepositorioProductos` que NUNCA se le pasó manualmente en `SpringDemo` — Spring lo inyectó automáticamente en el constructor gracias a `@Autowired`.
- "Es el mismo bean (singleton): true" confirma el comportamiento por defecto de Spring: pedir el mismo tipo de bean dos veces devuelve la MISMA instancia, no una copia — por eso ambas variables (`servicio` y `otraReferencia`) apuntan al mismo objeto (`==` da `true`, como viste en Etapa 6 Lección 1).
- En NINGÚN lugar del código escribiste `new RepositorioProductos()` ni `new ServicioVentas(repositorio)` — el `ComponentScan` encontró ambas clases anotadas, y el contenedor resolvió automáticamente que `ServicioVentas` necesitaba un `RepositorioProductos`, creándolo primero.

## Ejemplo aplicado al proyecto

Este es literalmente el MISMO patrón de `ServicioVentas` recibiendo un repositorio por constructor que viste en la Etapa 8 — la diferencia es que ahí VOS escribías `new ServicioVentas(repositorio)` en `DiDemo`, y acá Spring lo hace automáticamente al escanear las anotaciones. DI manual y DI por Spring son el MISMO principio, con distinto nivel de automatización.

Creá una clase `ServicioReportes` anotada con `@Service`, que reciba `RepositorioProductos` por constructor con `@Autowired`, con un método `contarProductos()` que devuelva `repositorio.listarTodos().size()`.

<Reveal titulo="Solución">
```java
package com.curso.catalogo;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

@Service
public class ServicioReportes {

    private final RepositorioProductos repositorio;

    @Autowired
    public ServicioReportes(RepositorioProductos repositorio) {
        this.repositorio = repositorio;
    }

    public int contarProductos() {
        return repositorio.listarTodos().size();
    }
}
```
</Reveal>

## Mini reto de debugging

```java
@Service
public class ServicioVentas {
    private RepositorioProductos repositorio;

    public ServicioVentas(RepositorioProductos repositorio) {
        this.repositorio = repositorio;
    }
}
```

Esta versión NO tiene `@Autowired` en el constructor. Si `ServicioVentas` tiene un ÚNICO constructor, ¿Spring igual lo va a poder inyectar?

<Reveal titulo="Diagnóstico">
Sí. Desde Spring 4.3, si una clase tiene UN SOLO constructor, Spring lo usa automáticamente para inyección, incluso sin `@Autowired` explícito — la anotación se vuelve obligatoria solo cuando hay MÚLTIPLES constructores y necesitás indicarle a Spring cuál usar. Aun así, se considera buena práctica escribir `@Autowired` explícitamente para que la intención sea clara con solo leer el código, sin depender de esa regla implícita.
</Reveal>

## Resumen

- IoC significa que un contenedor externo (Spring) controla la creación y conexión de objetos, en vez de tu código.
- `@Component`/`@Service`/`@Repository` marcan clases como beans gestionados por Spring.
- `@Autowired` le dice a Spring que resuelva automáticamente las dependencias de un constructor.
- Por defecto, cada bean es singleton: una sola instancia compartida en todo el contenedor.

## Modelo mental final

Pensá en el `ApplicationContext` como un director de orquesta que sabe qué instrumentos (beans) existen y en qué orden entran — vos solo le decís qué instrumento toca cada músico (anotaciones), y el director se encarga de que todos entren en el momento correcto, sin que cada músico tenga que coordinarse manualmente con los demás.

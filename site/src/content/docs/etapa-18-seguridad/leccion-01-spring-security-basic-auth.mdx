## Qué vas a aprender

Después de esta lección serás capaz de proteger tu API REST con Spring Security: configurar QUÉ endpoints requieren autenticación, definir usuarios en memoria, y entender cómo el contexto de seguridad bloquea peticiones no autenticadas automáticamente.

## Por qué necesitas aprenderlo

Hasta ahora, `ProductoController` es completamente público — cualquiera que conozca la URL puede crear, vender y consultar productos. En una aplicación real, eso es inaceptable: necesitás que solo usuarios IDENTIFICADOS puedan modificar datos, y que el sistema sepa QUIÉN hizo cada operación.

## Qué debes saber antes

`@RestController`, `@PostMapping`, `@GetMapping` (Etapa 12) — Spring Security se integra EXACTAMENTE en el mismo flujo HTTP que ya manejás.

## Problema

`ProductoController` expone endpoints CRUD y de venta sin ninguna protección — `curl -X POST .../api/productos` funciona sin credenciales, sin token, sin nada. Necesitás que solo usuarios AUTENTICADOS puedan acceder a esos endpoints, rechazando peticiones anónimas con HTTP 401 Unauthorized.

## Modelo mental

```
Sin Spring Security:   /api/productos  -->  ProductoController
                        (cualquiera)        (sin preguntar quién)

Con Spring Security:    /api/productos  -->  [FILTRO DE SEGURIDAD]
                                             |
                                        ¿Autenticado?
                                       /            \
                                     SI              NO
                                    /                  \
                        ProductoController          401 Unauthorized
```

## Explicación sencilla

Spring Security es un FILTRO que se ejecuta ANTES de que tu `ProductoController` reciba la petición. Si la petición no incluye credenciales válidas, el filtro la RECHAZA con HTTP 401 — tu controlador ni siquiera se ejecuta. Si las credenciales son válidas, el filtro deja pasar la petición y tu controlador funciona exactamente igual que antes.

## Explicación técnica

`spring-boot-starter-security` en el classpath activa la auto-configuración de Spring Boot: TODOS los endpoints pasan a requerir autenticación por defecto, y se genera una contraseña aleatoria que aparece en el log al arrancar. Un `SecurityFilterChain` bean personalizado REEMPLAZA esa configuración por defecto con reglas explícitas:

- `.csrf(csrf -> csrf.disable())` desactiva CSRF (Cross-Site Request Forgery) — necesario para APIs REST sin estado (stateless) que no usan sesiones ni cookies de formulario.
- `.authorizeHttpRequests(auth -> auth.requestMatchers("/api/productos/**").authenticated().anyRequest().permitAll())` define: los endpoints bajo `/api/productos/` requieren autenticación; todo lo demás (H2 console, etc.) es público.
- `.httpBasic(basic -> {})` activa HTTP Basic Authentication — el cliente envía `Authorization: Basic base64(usuario:contraseña)` en cada petición.
- `.sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))` configura la API como SIN ESTADO — cada petición debe incluir credenciales, no se usa sesión del servidor.

`UserDetailsService` define DE DÓNDE vienen los usuarios — en esta lección, un `InMemoryUserDetailsManager` con un solo usuario fijo (`admin/admin123`), ideal para aprender. `PasswordEncoder` (BCrypt) asegura que las contraseñas NUNCA se guarden en texto plano.

## Cómo funciona

1. Agregás `spring-boot-starter-security` al `pom.xml` — solo con eso, TODOS los endpoints ya requieren autenticación (configuración por defecto de Spring Boot).
2. Creás `SecurityConfig` con un `@Bean SecurityFilterChain` que SOBREESCRIBE ese comportamiento por defecto, definiendo reglas explícitas.
3. `curl http://localhost:8080/api/productos` sin credenciales → HTTP 401 Unauthorized.
4. `curl -u admin:admin123 http://localhost:8080/api/productos` con credenciales → HTTP 200, lista de productos normal.

## Ejemplo mínimo

**Archivo:** `proyecto-maven/pom.xml` (dependencia agregada)

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>
```

**Archivo:** `proyecto-maven/src/main/java/com/curso/catalogo/SecurityConfig.java`

```java
package com.curso.catalogo;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.core.userdetails.User;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;
import org.springframework.security.provisioning.InMemoryUserDetailsManager;
import org.springframework.security.web.SecurityFilterChain;

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/productos/**").authenticated()
                .anyRequest().permitAll()
            )
            .httpBasic(basic -> {})
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS));

        return http.build();
    }

    @Bean
    public UserDetailsService userDetailsService() {
        UserDetails admin = User.builder()
                .username("admin")
                .password(passwordEncoder().encode("admin123"))
                .roles("ADMIN")
                .build();

        return new InMemoryUserDetailsManager(admin);
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

## Predice

Antes de Spring Security, `curl http://localhost:8080/api/productos` devolvía la lista de productos. Ahora, SIN la cabecera `Authorization`, ¿qué va a devolver?

<Reveal titulo="Resultado real (ejecutado con Maven)">
```
$ curl -i http://localhost:8080/api/productos
HTTP/1.1 401 Unauthorized
...
```
401 — Spring Security bloqueó la petición ANTES de que llegara a `ProductoController.listar()`.
</Reveal>

## Explicación paso a paso

- `spring-boot-starter-security` en el classpath activa `SpringBootWebSecurityConfiguration` — un `SecurityFilterChain` por defecto que protege TODOS los endpoints con HTTP Basic y un usuario generado aleatoriamente.
- Nuestro `SecurityFilterChain` bean REEMPLAZA ese default — ahora las reglas son las que NOSOTROS definimos, no las automáticas.
- `.requestMatchers("/api/productos/**").authenticated()` aplica a `/api/productos`, `/api/productos/Teclado`, `/api/productos/Teclado/vender` — el `**` cubre cualquier subpath.
- `.anyRequest().permitAll()` es el "catch-all" — todo lo que no coincida con los matchers anteriores (H2 console, endpoints futuros) es público.
- `SessionCreationPolicy.STATELESS` le dice a Spring Security que no cree `HttpSession` ni use cookies — cada petición es independiente, ideal para APIs REST.

## Ejemplo aplicado al proyecto

Con Spring Security activo, `ProductoController` sigue funcionando EXACTAMENTE igual — la única diferencia es que ahora cada petición DEBE incluir credenciales. El controlador no cambió ni una línea: la seguridad es un aspecto TRANSVERSAL que Spring maneja en el filtro, sin tocar la lógica de negocio.

## Ejercicio trabajado

Verificar que `curl -u admin:admin123 http://localhost:8080/api/productos` sigue devolviendo la lista de productos normalmente — con credenciales correctas, el comportamiento del endpoint NO cambia:

```
$ curl -u admin:admin123 http://localhost:8080/api/productos
[{"nombre":"Teclado mecanico",...}]
```

Y que `curl -u admin:wrongpass http://localhost:8080/api/productos` devuelve 401 — credenciales inválidas son rechazadas.

## Ejercicio guiado

Proteger SOLO las operaciones de escritura (`POST /api/productos`, `POST /api/productos/{nombre}/vender`) y dejar las de lectura (`GET /api/productos`, `GET /api/productos/{nombre}`) como públicas — usando `requestMatchers` con `HttpMethod`.

<Reveal titulo="Solución">
```java
import org.springframework.http.HttpMethod;

http
    .authorizeHttpRequests(auth -> auth
        .requestMatchers(HttpMethod.POST, "/api/productos/**").authenticated()
        .requestMatchers(HttpMethod.GET, "/api/productos/**").permitAll()
        .anyRequest().permitAll()
    );
```
`HttpMethod.POST` restringe el matcher a peticiones POST — los GET pasan sin autenticación.
</Reveal>

## Transferencia

HTTP Basic envía `usuario:contraseña` codificado en Base64 en CADA petición — Base64 NO es encriptación, es codificación (cualquiera puede decodificarlo). En producción, HTTP Basic SOLO es aceptable sobre HTTPS (TLS). ¿Qué mecanismo de autenticación resuelve el problema de enviar credenciales en cada petición, reemplazándolas por un token de duración limitada?

## Mini reto de debugging

```java
@Bean
public UserDetailsService userDetailsService() {
    UserDetails admin = User.builder()
            .username("admin")
            .password("admin123")  // sin .encode()
            .roles("ADMIN")
            .build();
    return new InMemoryUserDetailsManager(admin);
}
```

El test de autenticación falla — `curl -u admin:admin123` devuelve 401. ¿Por qué?

<Reveal titulo="Diagnóstico">
`.password("admin123")` sin `.encode()` guarda la contraseña en TEXTO PLANO en el `UserDetails`. Pero `PasswordEncoder` (BCrypt) espera un hash BCrypt — cuando Spring Security compara la contraseña ingresada (`admin123`) con lo que hay en `UserDetails`, usa el `PasswordEncoder` para verificar, y BCrypt no reconoce "admin123" como un hash válido. Siempre usar `.password(passwordEncoder().encode("admin123"))` — el encoder debe aplicarse al CREAR el usuario, no solo al verificarlo.
</Reveal>

## Mini reto de diseño

¿Por qué creamos un `PasswordEncoder` como bean separado en vez de instanciarlo directamente en `userDetailsService()`?

<Reveal titulo="Respuesta">
Porque Spring Security busca un bean `PasswordEncoder` en el contexto para usarlo GLOBALMENTE — en la verificación de contraseñas durante el login, en el `DaoAuthenticationProvider`, y potencialmente en otros lugares. Si lo instanciás solo dentro de `userDetailsService()`, el resto de Spring Security no lo encuentra y usa un encoder por defecto (que probablemente no sea BCrypt), causando inconsistencia entre cómo se GUARDÓ la contraseña y cómo se VERIFICA.
</Reveal>

## Resumen

- `spring-boot-starter-security` protege TODOS los endpoints por defecto — necesitás un `SecurityFilterChain` bean para personalizar las reglas.
- `.requestMatchers("/api/productos/**").authenticated()` protege endpoints específicos; `.anyRequest().permitAll()` deja el resto público.
- HTTP Basic + `SessionCreationPolicy.STATELESS` es un primer paso funcional para APIs REST, pero envía credenciales en cada petición.
- `UserDetailsService` + `PasswordEncoder` definen el origen de usuarios y cómo se codifican las contraseñas.

## Modelo mental final

Spring Security es un GUARDIÁN en la puerta de tu aplicación: revisa CADA petición ANTES de que llegue a tu controlador. Si la petición no trae las credenciales correctas, ni siquiera entra. Si las trae, tu controlador trabaja exactamente igual que siempre — no sabe ni necesita saber que hay un guardián.

## Conexión

Así como `ManejadorDeErrores` con `@RestControllerAdvice` (Etapa 14 L2) intercepta excepciones de tus controladores para transformarlas en respuestas HTTP consistentes, `SecurityFilterChain` intercepta PETICIONES HTTP para decidir si pueden llegar a tus controladores — ambos son aspectos transversales que Spring maneja fuera de tu lógica de negocio.
